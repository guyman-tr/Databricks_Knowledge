# Skill Authoring Guide

## Before Writing ג€” Capture Intent

Answer these questions before drafting (extract from conversation history if already discussed):

1. **What should this skill enable?** ג€” Which questions/tasks should it help answer?
2. **When should it trigger?** ג€” What user phrases, synonyms, or contexts should activate it?
3. **What table(s) does it cover?** ג€” Fully qualified Unity Catalog names.
4. **What mistakes does it prevent?** ג€” Common misuses of the data that produce wrong results.

If creating from an existing conversation (e.g., "turn this into a skill"), extract: tables used, join keys, filters applied, corrections made, and the business logic that emerged.

---

## Core Principles

**Conciseness.** Only add information the model doesn't already have. Challenge each line: does it prevent a wrong answer or enable a correct one? If not, remove it.

**Degrees of freedom.** Match specificity to fragility:
- **High freedom** (general guidance): when multiple approaches are valid. Example: "Join to the relevant dimension table."
- **Low freedom** (exact SQL): when a specific filter is critical and alternatives silently produce wrong numbers. Example: `WHERE IncludedInTotalRevenue = 1`.

Query patterns are typically low-freedom ג€” wrong filters fail silently.

**Explain the why.** Instead of rigid "ALWAYS do X", explain the reasoning. The model understands importance through context better than through commands.

**Generalize.** Skills are used across many prompts. Write for the pattern, not one specific instance.

**Consistent terminology.** Pick one term and stick with it. Don't alternate between "funded customers" / "funded users" / "funded accounts" unless they mean different things.

## Required Sections

The markdown body must include these sections to pass CI checks:

### 1. Title
Short, descriptive. No org prefixes.

### 2. `## When to Use`
Trigger patterns ג€” what phrases/contexts should activate this skill. Maps to CI check DOMAIN-004.

### 3. `## Scope`
Must contain all three lines (CI check DOMAIN-001):
```
In scope: [explicit table names, metrics, concepts]
Out of scope: [what belongs to sibling skills]
Last verified: 2026-05-07
```
`Last verified` must be a valid ISO date, not older than 90 days (CI checks DOMAIN-002, DOMAIN-003).

### 4. `## Critical Warnings`
Numbered list, ordered by severity tier (CI checks DOMAIN-005, DOMAIN-006, DOMAIN-007):
- **Tier 1** ג€” silent wrong numbers (e.g., missing filter produces inflated results)
- **Tier 2** ג€” aggregate inflation (e.g., summing snapshot data across dates)
- **Tier 3** ג€” dependencies / edge cases (e.g., table refresh timing)

### 5. Body Content (flexible)
After the required sections, include:
- **Core Concepts** ג€” `| Concept | What It Is | Aliases |` table (5-15 concepts per skill)
- **Query Patterns** ג€” 4 patterns with SQL + `Use when:` trigger phrases
- **Additional context** ג€” hierarchies, flag combinations, domain structures as needed

## Description Best Practices

The `description` field is the **primary embedding signal** ג€” it determines whether the skill gets retrieved. Write it slightly "pushy" to combat under-triggering:

- **Third person always** ג€” "Covers...", "Defines...", "Provides..." ג€” never "I help you..." or "You can use this..."
- Include both **what the skill covers** AND **specific triggers** for when to retrieve it
- List keywords, synonyms, and adjacent phrasings a user might say
- Cover edge cases where the user doesn't name the domain explicitly

**Weak:** `Revenue streams and metrics.`

**Strong:** `Revenue streams and metrics (Total Net Revenue, FullCommission, Rollover, Ticket Fees, Conversion Fee, Dormant Fee, Staking, ShareLending, Dividends, Options PFOF). Load when users ask about revenue definitions, revenue breakdown, or how to calculate total revenue.`

```markdown
# Revenue Business Logic

## When to Use
Load when users ask about: revenue, fee definitions, total net revenue,
fee breakdown, "how much did we make", commission, rollover fees.

## Scope
In scope: DDR revenue view, trading revenue materialized view, all 19 fee types
Out of scope: Cost/expense data, P&L calculations (see portfolio-value skill)
Last verified: 2026-05-07

## Critical Warnings
1. SUM(Amount) without `IncludedInTotalRevenue = 1` includes Commission
   (double-counts) and Dividends (negative) ג€” produces silently wrong totals.
2. Aggregating across DateID without GROUP BY inflates numbers when multiple
   fee types exist per day per customer.
3. The DDR view refreshes at 06:00 UTC ג€” intraday queries show stale data.

## Core Concepts
| Concept | What It Is | Aliases |
| --- | --- | --- |
| **Total Net Revenue** | SUM(Amount) where IncludedInTotalRevenue = 1 | TNR, net revenue |
| **Rollover** | Overnight financing fee for holding CFD/FX past close | swap fee, overnight fee |

## Query Patterns

### Pattern 1 ג€” Total Net Revenue
SELECT SUM(Amount) FROM main.etoro_kpi.vg_ddr_revenue
WHERE IncludedInTotalRevenue = 1 AND etr_ymd BETWEEN '...' AND '...';

Use when: "what's our total revenue?", "how much did we make?"
```

## Rules

**Date Filtering ג€” Partition Columns:** For non-SCD fact tables, check if the table has `etr_ymd` (or `etr_ym`, `etr_y`) partition columns. If available, use `etr_ymd` for date filtering instead of `DateID` ג€” this enables partition pruning and dramatically improves query performance. Format is `'YYYY-MM-DD'` (string). Note the table's partitions in the Scope section. SCD tables use `FromDateID`/`ToDateID` logic instead.

**Source Integrity (Critical):** All business content ג€” column names, metric names, flag values, filter values, enum labels ג€” MUST be verified against actual source tables before inclusion. Use `readTable` to inspect schemas, run queries (e.g., `SELECT DISTINCT column`) to confirm values. Never invent or assume names from business jargon alone.

Business content must originate from one of:
1. Source table metadata (column names, comments, distinct values)
2. Explicit user input (stated definitions, confirmed conventions)
3. **Logical consequences** of the above ג€” mathematical facts, definitional truths, or structural observations

What is NOT acceptable:
- Made-up numbers unless from source
- Formulas not stated in any source
- Editorial opinions
- Guessed dimension table names or join targets

**Sizing:** One skill per domain (5-15 concepts). Target 3.5-6 KB, max 500 lines. If a skill grows beyond 6 KB, split into a `references/` subdirectory with additional files referenced from SKILL.md with clear "read this when..." guidance.

## Iterative Development

1. **Complete a task without a skill.** Notice what context you repeatedly provide and what goes wrong.
2. **Identify the reusable pattern.** Tables, filters, definitions, cautions that would prevent mistakes.
3. **Write a minimal skill.** Address only observed gaps ג€” not imagined ones.
4. **Test on similar prompts.** Does it trigger? Produce correct results? Interfere with siblings?
5. **Iterate.** If it misses something, add it. If it triggers incorrectly, narrow the description or triggers.

Start specific, consolidate later. Only merge skills when the consolidated version performs equivalently.

---

## Anti-Patterns

| Anti-pattern | Why it's harmful |
| --- | --- |
| Multiple options without a default | Model picks arbitrarily. Provide one approach, mention alternatives only for edge cases. |
| Time-sensitive instructions | Skills persist ג€” state the current correct approach only. |
| Vague scope ("all revenue data") | Fails CI check QUAL-005. List explicit tables and metrics. |
| Duplicating Unity Catalog metadata | Column types already exist in `readTable` output. Add only the business layer. |
| Overlapping descriptions between skills | Causes mis-triggering. Keep each skill's description distinct from siblings. |

## Coexistence & Overlap Prevention

- **Descriptions compete.** Each skill's description + triggers are embedded for retrieval. Overlapping descriptions cause the wrong skill to activate.
- **Test coexistence.** After adding a skill, verify siblings still trigger correctly.
- **One skill per domain.** Don't fragment a domain into multiple skills.
- **Narrow "Out of scope"** to genuine confusion points with siblings only.

### Mandatory Pre-Creation Overlap Check

**Before creating or modifying any skill, scan all existing skills in the general workspace (`/.assistant/skills/`).** Compare the proposed skill's domain, triggers, tables, and description against every workspace-level skill.

If the new skill's domain overlaps with an existing workspace-level skill:
1. **STOP** ג€” do not create the skill.
2. **Alert the user** with a clear message: identify which existing workspace skill(s) overlap, what specific concepts/tables/triggers conflict, and why creating a duplicate would cause mis-triggering.
3. **Suggest alternatives**:
   - **Merge** the new content into the existing workspace skill (preferred if same domain).
   - **Narrow** the new skill's scope to cover only the non-overlapping portion, with explicit "Out of scope" boundaries pointing to the existing workspace skill.
   - **Restructure** ג€” if the existing skill is too broad, propose splitting it into two non-overlapping skills with clear boundaries.

**Overlap signals to check:**
- Same `required_tables` entries
- Overlapping `triggers` keywords
- Similar `description` phrasing or concepts
- Same business domain (e.g., two skills both covering "revenue" or "customer segments")

This check ensures every skill has a unique, well-bounded domain and prevents retrieval confusion.

After writing or editing a skill, test against 2-3 realistic prompts:

- One obvious trigger ("what's our total revenue last month?")
- One edge case where the user doesn't name the domain ("how much did we make from overnight fees?")
- One near-miss that should NOT trigger this skill but a sibling instead

Also verify: frontmatter fields complete, `## Scope` has all three lines, `## Critical Warnings` numbered and severity-ordered, `Last verified` date is today, file under 500 lines.
