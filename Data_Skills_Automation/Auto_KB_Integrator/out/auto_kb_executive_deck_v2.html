Executive Brief

Auto KB Existing Automations

Automated knowledge operations that detect meaningful platform deltas and update skills only when durable value exists.

Live architecture in place
Business Problem

Why we built this

  • Knowledge changed faster than manual updates to skills and domains.
  • Important updates could be missed between human curation cycles.
  • Signals were fragmented across Genie, Unity Catalog, DB wikis, and Confluence.

Objective: move to a repeatable daily process that detects, evaluates, and applies only high-value knowledge changes.

Operating Model

One consistent flow across all sources

  • Detect: snapshot diff finds new/changed items.
  • Evaluate: durability + overlap checks filter noise.
  • Act: conditional ingest/push for approved items only.
  • Log: every decision is persisted and reportable.
Diff -> Evaluate -> Conditional Push
Coverage

Six trackers currently running

Genie Watcher

Detects new/changed Genie spaces and scores curation quality before action.

UC Object + Pipeline Watcher

Tracks new/changed UC objects, resolves build lineage, updates docs/comments.

DBSchema Lake Wiki Watcher

Finds DB wiki changes that map to lake objects via generic pipeline links.

Confluence Delta Watcher

Processes only version deltas from MCP snapshots; avoids spray-and-pray ingest.

Skill Submission Watcher

Monitors the Databricks skill-submission app intake and processes proposals through ingest and conditional push.

Questions Watcher

Mines MCP and Genie question logs, clusters recurring intents, scores skill coverage, and proposes gap dossiers for underserved topics.

Databricks App Channels

  • Dedicated watcher exists: submission rows are processed by the skill-suggestions automation flow (ingest + conditional push).
  • Feedback App: present but currently underutilized in operations.

Submission tracked now, feedback tracker deferred
Quality Safeguards

How we prevent skill bloat

  • Existing overlap gate blocks duplicate semantic coverage.
  • New adversarial durability gate filters temporary/trivial artifacts.
  • Heuristic + adversarial judge returns approve, reject, or review.
  • Rejected/reviewed items are logged as skipped with explicit reasons.
Noise filtered before ingest
Daily Operations

Single orchestrated run

  • Wrapper executes: Confluence bridge → 6 watchers → implications → integrator.
  • Staging mode enables safe proving before production push behavior.
  • Snapshots advance only on fully successful non-dry runs.

Outcome: predictable, auditable, and resumable knowledge operations.

Executive Outputs

What leadership receives

  • Integrated daily summary from the 6th agent (cross-source rollup).
  • Implication summaries showing accepted, skipped, and blocked items.
  • Action-focused appendix for follow-up decisions and ownership.

Everything is produced under Data_Skills_Automation/Auto_KB_Integrator/out/.

CI/CD Decision Point

PR review model for automation changes

  • Current mode: automation opens skill-change PRs, humans approve/merge.
  • Scale risk: as volume grows, manual review can become a bottleneck.
  • Proposed option: dedicated Auto-KB CI/CD lane with policy-based auto-merge for low-risk changes.
  • Safety controls: strict checks (tests/lint/format), durability score threshold, duplicate gate pass, protected-path allowlist, rollback path.
  • Governance: keep high-risk domains or low-confidence items on mandatory human review.
Hybrid model: autonomous for safe changes, human review for sensitive ones
