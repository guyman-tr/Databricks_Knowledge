## What the trace bought us that the CSV alone couldn't

For every case we now have:
- **What skill the agent loaded** (file_read path, NOT MCP). 14/15 cases loaded the right domain hub on the first read.
- **What SQL it actually ran** (5 cases ran 3+ queries; the agent self-corrects column names and table FQNs in real-time)
- **Whether it ran SQL via MCP or via `python tools/dbx_query.py`** — answer: 100% of SQL went via `dbx_query.py`. Our workspace rule `mcp-latency-signal.mdc` tells the agent to fall back to `dbx_query.py` after one MCP miss; in practice the agent doesn't even try MCP first. **This is a finding for the MCP-vs-Genie comparison plan.** `databricks_mcp` SUT and `cursor_agent` SUT measure two completely different things.
- **The agent's reasoning chain** — visible in `trace_thinking_excerpt`. Case #7's thinking shows the model identifying the correct query, computing $105M, then second-guessing itself and adding a parallel "$142M total" path. Question wording could trim that.
- **Tokens per case** — avg ~330k cache-read + 2k output. Very consistent across cases.

## Failure of MCP fallback rule (incidental finding)

**Every single SQL exec in this run went through `python tools/dbx_query.py`, not the MCP gateway.**
The agent's reasoning shows it:
1. Reads the workspace rule (`mcp-latency-signal.mdc`)
2. Emits the user-facing latency signal text
3. Goes straight to `python tools/dbx_query.py` — without trying the MCP first

This is consistent with the rule as written ("After ONE failed or missing `user-databricks_sql-*` tool call, run `python tools/dbx_query.py`"), but the agent is interpreting this as "always use dbx_query.py because MCP might fail" rather than "try MCP, fall back if it fails." If we want to actually evaluate the MCP path (which is the user's stated end goal: "test the custom Databricks MCP"), we need either:
- A separate SUT that disables the workspace rule and forces MCP
- Or rewrite the rule to "try MCP first, fall back ONLY on actual failure"

This is **not** a blocker for shipping the eval suite — but it's a critical thing the user needs to know before declaring the agent's results representative of "the custom MCP".

## Recommended next moves (in priority order)

1. **Fix the scalar parser (item A)** — biggest leverage, zero token cost. Three concrete changes:
   - Don't strip negative signs (case #3 was sign-dropped)
   - Detect "X in N customer actions / N rows / N customers" patterns and exclude N from the candidate set (cases #1, #2)
   - Prefer numbers that appear AFTER currency symbols and BEFORE units (`$X` or `X USD`) over bare numerics (cases #1, #7)
   Expected impact: cases #1, #2, #3, #7 flip to PASS. **+4 cases → 10/15.**

2. **Refine NL questions for category B** — three of four cases need explicit scope (`ddr_revenue_totals`, `ddr_aum_equity_global`, `ddr_aum_realized_equity_global`). Current wording: "What was eToro's Total Revenue on 2026-06-08?". Better: "What was eToro's Total Revenue from the trading platform DDR (`BI_DB_DDR_Fact_Revenue_Generating_Actions` with `IncludedInTotalRevenue=1`) on 2026-06-08? Exclude Spaceship and other acquired platforms." This sounds heavy-handed but it disambiguates the rollup contract and is exactly what the user asked for ("split into excluding spaceship and spaceship only"). Expected impact: 3 more cases flip → **13/15**.

3. **Build the SQL-replay scorer (item C)** — replay the agent's SQL ourselves, compare against canonical SQL. This is the missing third leg. With the trace I now have all the inputs (`sql_used` is captured for every case). For passing cases this is a no-op; for failing cases it tells us if the issue was the SQL itself (skill gap) or the post-SQL summary (parser/question). I can ship this in ~45 min: parse `sql_used`, run via `dbx_query.py`, diff against `uc_equivalent.sql`.

4. **Refine the drift_verdict taxonomy (item D)** — extend `SKILL_GAP` into a richer set:
   - `SKILL_NOT_LOADED` — skill list doesn't include the relevant domain hub
   - `WRONG_SKILL` — agent loaded a different domain than the canonical SQL touches
   - `RIGHT_SKILL_WRONG_TABLE` — agent loaded right skill, picked wrong table FQN
   - `RIGHT_SKILL_RIGHT_TABLE_WRONG_FILTER` — agent's SQL hits right table but with different WHERE/JOIN (e.g. missing valid-users)
   - `RIGHT_SQL_WRONG_REPORT` — SQL would return the canonical answer if replayed; agent reported wrong number
   - `PARSER_DROP` — number IS in the agent's text, but our scalar parser missed it
   These map deterministically off the trace data + replayed SQL output.

5. **Re-run the same 15** with all of the above. Expect 13–14/15.
