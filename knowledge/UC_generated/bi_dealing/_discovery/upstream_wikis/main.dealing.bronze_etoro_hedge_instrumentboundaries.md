## 1. Business Meaning

Hedge.InstrumentBoundaries configures the **CBH "Boundaries" hedging strategy** for each (HedgeServer, Instrument) pair. In the Boundaries strategy (as described in HedgeServer Overview), the hedge server monitors net customer exposure against two thresholds and hedges a partial amount up to the HRL:

- **OpenThresholdUSD**: The USD exposure amount at which hedging begins. When |NetUSDExposure| > OpenThresholdUSD, the hedge server places an order to hedge.
- **CloseThresholdPercentage**: The percentage of OpenThresholdUSD below which the existing hedge is reduced/closed. E.g., 50% means: if exposure falls below (OpenThresholdUSD * 50%), reduce the hedge position.
- **HedgeRiskLimitUSD (HRL)**: The maximum USD amount to hedge. Hedge until hedged amount = HRL. HRL=0 means fully hedge the entire exposure above the threshold (no upper cap).

**Examples from live data**:
- HedgeServerID=1, instruments 1-5: Open=$50,000, Close=50%, HRL=$0 (fully hedge when exposure > $50K)
- HedgeServerID=2, instruments 1-5: Open=$2,000,000, Close=50%, HRL=$500,000 (hedge up to $500K when exposure > $2M)
- HedgeServerID=0 (all zeros): Default/no-boundary configuration; rows with 0,0,0 effectively disable boundary hedging for those instruments

The table has 111,312 rows covering 32 distinct servers and 10,498 distinct instruments. The large row count reflects that boundaries are configured at the granular instrument+server level - every tradeable instrument on every hedge server gets its own boundary configuration.

**Audit**: All DML changes are tracked via three ASM-generated triggers (INSERT/UPDATE/DELETE) writing column-level audit records to `History.AuditHistory`. The column-level audit captures old and new values for each of the three threshold columns separately.

**View**: `Hedge.GetInstrumentBoundaries` is a thin NOLOCK wrapper over this table, used by the hedge server application for efficient reads.

## 2. Business Logic

### 2.1 Boundary Hedging Thresholds

**What**: The three threshold columns together define the full hedging lifecycle for the Boundaries CBH strategy.

**Columns/Parameters Involved**: `OpenThresholdUSD`, `CloseThresholdPercentage`, `HedgeRiskLimitUSD`

**Rules**:
- **OpenThresholdUSD**: When |NetUSDExposure| exceeds this value, the hedge server triggers a hedge order. Units: USD. Int precision (no decimals) - thresholds are set in whole USD amounts. Range in data: 0 to 400,000,000 (max $400M threshold for very large instruments).
- **HedgeRiskLimitUSD (HRL)**: The target hedged amount. After triggering, hedge until `HedgedUnits * Price >= HRL`. HRL=0 = no upper cap - hedge the full exposure above the threshold.
- **CloseThresholdPercentage**: When exposure falls BELOW (OpenThresholdUSD * CloseThresholdPercentage / 100), close/reduce the existing hedge position. Prevents whipsawing - the close threshold is lower than the open threshold. E.g., Open=$50K, Close=50%: open hedge when >$50K, close hedge when <$25K. Typical value: 50 (50%).
- All zeros (OpenThresholdUSD=0, CloseThresholdPercentage=0, HedgeRiskLimitUSD=0): Boundary hedging is disabled for this instrument/server; the Fully strategy or no hedging applies instead.

### 2.2 Per-Server Configuration

**What**: Boundaries are configured independently per (HedgeServerID, InstrumentID) pair, allowing different thresholds for the same instrument on different servers.

**Columns/Parameters Involved**: `HedgeServerID`, `InstrumentID`

**Rules**:
- HedgeServerID=0 appears with all-zero thresholds - likely a global default placeholder or historical rows.
- Different servers hedge the same instrument with different risk tolerances (server 1: $50K open, server 2: $2M open).
- 1,646 all-zero rows (OpenThresholdUSD=0, CloseThresholdPercentage=0, HedgeRiskLimitUSD=0) - these instruments effectively have no boundary configuration on those servers.

### 2.3 ASM Audit Triggers

**What**: All INSERT/UPDATE/DELETE operations are automatically audited at the column level.

**Columns/Parameters Involved**: `OpenThresholdUSD`, `CloseThresholdPercentage`, `HedgeRiskLimitUSD`

**Rules**:
- Three triggers auto-generated by ASM (Audit System Manager): AuditInsert, AuditUpdate, AuditDelete.
- Each trigger writes three rows to `History.AuditHistory` - one row per audited column (OpenThresholdUSD, CloseThresholdPercentage, HedgeRiskLimitUSD).
- Audit row contains: AuditDate, UserName, AppName, HostName, SchemaName='Hedge', TableName='InstrumentBoundaries', ColumnName, OldValue, NewValue, Operation ('I'/'U'/'D'), PK_Value ('HedgeServerID,InstrumentID').
- Do NOT alter or drop these triggers - ASM manages them.

## 8. Sample Queries

### 8.1 Boundary configuration for a specific server
```sql
SELECT InstrumentID, OpenThresholdUSD, CloseThresholdPercentage, HedgeRiskLimitUSD
FROM Hedge.InstrumentBoundaries WITH (NOLOCK)
WHERE HedgeServerID = 1
  AND OpenThresholdUSD > 0
ORDER BY InstrumentID;
```

### 8.2 Instruments with active boundaries (non-zero thresholds)
```sql
SELECT HedgeServerID, COUNT(1) AS ActiveBoundaryCount,
       AVG(OpenThresholdUSD) AS AvgOpenThreshold,
       AVG(HedgeRiskLimitUSD) AS AvgHRL
FROM Hedge.InstrumentBoundaries WITH (NOLOCK)
WHERE OpenThresholdUSD > 0
GROUP BY HedgeServerID
ORDER BY ActiveBoundaryCount DESC;
```

### 8.3 Use the view (same as direct table access but preferred by hedge server)
```sql
SELECT * FROM Hedge.GetInstrumentBoundaries WITH (NOLOCK)
WHERE HedgeServerID = 1
ORDER BY InstrumentID;
```
