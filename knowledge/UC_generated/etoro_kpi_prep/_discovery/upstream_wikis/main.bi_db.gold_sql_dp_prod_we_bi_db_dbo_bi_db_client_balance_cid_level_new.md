## 1. Business Meaning

`BI_DB_Client_Balance_CID_Level_New` is the **primary client balance reporting table** for the eToro platform. It provides a daily, CID-level view of every financial dimension of a customer's account: opening/closing balances, deposits, cashouts, compensations, realized and unrealized PnL, commissions, liabilities, net open positions (NOP), and position amounts — broken down by asset class (CFD, real stocks, real crypto, TRS crypto, futures, stock margin).

The table was created in October 2020 by Guy Manova to replace the legacy cube-based Client Balance reports. It is the **last table to be computed daily** (Priority 99 in OpsDB orchestration) because it depends on virtually every other DWH table being loaded first — Fact_SnapshotCustomer, Fact_SnapshotEquity, V_Liabilities, Fact_CustomerAction, and 8 dimension tables.

### Data Structure — Transfer Direction

The data structure is at the **CID + movement level**, not just CID + Date. For customers who transferred regulation or credit-valid status between yesterday and today, there are **two rows per CID per date**:

- `TransferDirection = 1` → the row under the customer's **current** (destination) regulation
- `TransferDirection = -1` → the row under the customer's **previous** (source) regulation

This is critical for regulatory completeness: the balance cycle (Opening + Flows = Closing) must hold within each regulation. Non-transfer customers have one row with `TransferDirection = 1`.

**Query implication**: Always use `SUM() ... GROUP BY CID` when aggregating, never assume one row per CID per date.

### Business Usage

- **Finance Reconciliation**: Completeness check — OpeningBalance + all flows = ClosingBalance (see RegTech Confluence page)
- **Regulatory Reporting**: IFRS-15 balance (SP_IFRS_15_Balance), CySEC ICF report (SP_CySEC_Submission_ICF), RBSF (SP_RBSF, SP_Y_RBSF)
- **DDR (Daily Data Report)**: Customer daily status and AUM aggregation (SP_DDR_Customer_Daily_Status, SP_DDR_Fact_AUM)
- **CMR Automation**: Automated client money reports for USA, Seychelles, EU, and cross-regulation negative balance checking
- **EY Audit**: External audit data feeds (SP_EY_Audit_Client_Balance, SP_EY_Audit_RollOverFee)
- **Real Crypto Loans**: SP_Real_Crypto_Loans reads from this table for loan tracking
- **Data Quality**: SP_Client_Balance_Check_Opening_Balance validates ClosingBalance(D-1) = OpeningBalance(D); SP_Daily_CB_Gaps_All monitors cycle gaps
- **Tableau Dashboards**: Client Balance Report (workbook 4123), Revenue dashboards

## 2. Business Logic

### 2.1 ETL Pattern — DELETE + INSERT per DateID

**What**: The SP deletes all rows for `@dateID` then inserts fresh rows. This makes re-runs idempotent — reprocessing a date replaces all data cleanly.

**Key Steps**:

1. Extract 2-day snapshot from `Fact_SnapshotCustomer` (today + yesterday) via `Dim_Range`
2. Detect regulation transfers (today's RegulationID ≠ yesterday's) and credit-valid transfers
3. Build dimension lookups (#allUsers × Dim_Regulation × Dim_Country × Dim_Label × Dim_PlayerStatus × Dim_PlayerLevel × Dim_MifidCategorization × Dim_AccountType)
4. Unpivot all financial metrics from V_Liabilities / Fact_SnapshotEquity / Fact_CustomerAction into #fullMetricsUnion (Metric name → TotalMetric value)
5. Aggregate in #CIDAgg: SUM by (CID, TransferDirection, Regulation) with CASE WHEN filters on metric names
6. INSERT into target table with ISNULL(x, 0) wrapping

### 2.2 Balance Cycle Equation

The fundamental accounting equation this table enforces:

```
ClosingBalance = OpeningBalance + Deposits + CompensationDeposit + Bonus + Compensation
    + CompensationPI + CompensationToAffiliate + NWAAdjustment + NegativeRefill
    - Cashouts + TransferCoins + CompensationCashouts + CashoutFee + TransferCoinFees
    + Chargeback + Refund + OvernightFee + ChargebackLoss + OtherNegatives
    + CompensationDormantFee + ClientBalanceRealizedPnL + UnrealizedPnLChange
    + LostDebt + Foreclosure + CompensationPnLAdjustments
    + NetTransfersNWA + NetTransfersLiability + NetTransfersUnrealizedPnL
```

Any deviation from this equation is a "cycle gap" — monitored by SP_Daily_CB_Gaps_All and SP_Client_Balance_Check_Opening_Balance.

### 2.3 IsValidCustomer Flag

Inherited from `Fact_SnapshotCustomer`. Excludes demo accounts (PlayerLevelID=4), internal labels (LabelID IN 26,30), and blocked countries (CountryID=250). Use `WHERE IsValidCustomer = 1` for most analytics.

### 2.4 IsGermanBaFin Flag

Derived from `V_GermanBaFin` — indicates whether a customer falls under German BaFin regulatory supervision. Used for BaFin-specific reporting requirements.

### 2.5 IsDLTUser and DidDLTTransfer

- `IsDLTUser`: 1 if the customer's `DltStatusID = 4` in Fact_SnapshotCustomer (active DLT/Tangany wallet user)
- `DidDLTTransfer`: 1 if DLT status changed between yesterday and today
- `TanganyStatus`: Human-readable Tangany wallet status from `External_UserApiDB_Dictionary_TanganyStatus`

### 2.6 Commission Breakdown

Commissions are split by asset class and type:

- `ClientBalanceFullCommission` = total commission including eToro's spread markup
- `ClientBalanceCommission` = net commission (client's portion only)
- Breakdowns: CFD, RealCrypto, RealStocks, TRSCrypto, RealFutures
- `Unrealized*CommissionChange` variants = day-over-day delta in unrealized commission

### 2.7 DepositConversionFee / WithdrawConversionFee

Both columns are **always NULL** — the SP inserts `NULL` for these. They were added to the DDL as placeholders for future implementation.

## 3. Query Advisory

### 3.1 Distribution & Indexing

- **HASH(CID)**: Queries filtering on a single CID are highly efficient (single node). Date-range queries across all CIDs require data movement.
- **Clustered Index on DateID ASC**: Optimized for date-range scans. Use `WHERE DateID BETWEEN x AND y` to leverage the index.
- **No columnstore**: Unlike DWH Fact tables, this uses a traditional B-tree clustered index because the primary access pattern is date-based inserts and date-range reads.

### 3.2 Multi-Row Per CID Warning

For CIDs with regulation or credit-valid transfers, there are 2 rows per DateID. **Always aggregate**:

```sql
SELECT CID, SUM(Deposits) AS TotalDeposits, SUM(Cashouts) AS TotalCashouts
FROM BI_DB_dbo.BI_DB_Client_Balance_CID_Level_New
WHERE DateID = 20260310 AND IsValidCustomer = 1
GROUP BY CID
```

### 3.3 Data Freshness

- Runs as the **last table** in the daily ETL (Priority 99)
- After it completes, SP_Client_Balance_Check_Opening_Balance validates the data
- Depends on: Fact_SnapshotCustomer, Fact_SnapshotEquity, V_Liabilities, Fact_CustomerAction, and all 8 dimension tables being loaded first

### 3.4 View: V_BI_DB_Client_Balance_CID_Level_New

A thin wrapper: `SELECT * FROM BI_DB_Client_Balance_CID_Level_New WHERE DateID >= 20210901`. Use for queries that don't need pre-2021 data.

### 3.5 Related Table: BI_DB_Client_Balance_CID_Level_New_Blocked

Subset of this table for **blocked customers only** (PlayerStatus NOT IN 1,3,5,7). Populated by SP_Client_Balance_CID_Level_New_Blocked.

## 7. Sample Queries

```sql
-- Daily balance for a specific CID
SELECT CID, DateID, SUM(OpeningBalance) AS OpenBal, SUM(ClosingBalance) AS CloseBal,
       SUM(Deposits) AS Deposits, SUM(Cashouts) AS Cashouts,
       SUM(ClientBalanceRealizedPnL) AS RealizedPnL
FROM BI_DB_dbo.BI_DB_Client_Balance_CID_Level_New
WHERE CID = 12345 AND DateID BETWEEN 20260301 AND 20260310
GROUP BY CID, DateID
ORDER BY DateID;

-- Balance cycle gap detection
SELECT CID, DateID,
       SUM(ClosingBalance) - (SUM(OpeningBalance) + SUM(Deposits) + SUM(CompensationDeposit)
       + SUM(Bonus) + SUM(Compensation) + SUM(NWAAdjustment) + SUM(NegativeRefill)
       - SUM(Cashouts) + SUM(ClientBalanceRealizedPnL) + SUM(UnrealizedPnLChange)
       + SUM(OvernightFee) + SUM(Chargeback) + SUM(Refund) + SUM(ChargebackLoss)
       + SUM(OtherNegatives) + SUM(LostDebt) + SUM(Foreclosure)
       + SUM(NetTransfersNWA) + SUM(NetTransfersLiability) + SUM(NetTransfersUnrealizedPnL)
       ) AS CycleGap
FROM BI_DB_dbo.BI_DB_Client_Balance_CID_Level_New
WHERE DateID = 20260310 AND IsValidCustomer = 1
GROUP BY CID, DateID
HAVING ABS(SUM(ClosingBalance) - (SUM(OpeningBalance) + SUM(Deposits) + SUM(CompensationDeposit)
       + SUM(Bonus) + SUM(Compensation) + SUM(NWAAdjustment) + SUM(NegativeRefill)
       - SUM(Cashouts) + SUM(ClientBalanceRealizedPnL) + SUM(UnrealizedPnLChange)
       + SUM(OvernightFee) + SUM(Chargeback) + SUM(Refund) + SUM(ChargebackLoss)
       + SUM(OtherNegatives) + SUM(LostDebt) + SUM(Foreclosure)
       + SUM(NetTransfersNWA) + SUM(NetTransfersLiability) + SUM(NetTransfersUnrealizedPnL)
       )) > 0.01;

-- Total platform deposits and cashouts by regulation
SELECT Regulation, DateID,
       SUM(Deposits) AS TotalDeposits, SUM(Cashouts) AS TotalCashouts,
       COUNT(DISTINCT CID) AS CustomerCount
FROM BI_DB_dbo.BI_DB_Client_Balance_CID_Level_New
WHERE DateID = 20260310 AND IsValidCustomer = 1 AND TransferDirection = 1
GROUP BY Regulation, DateID;
```
