## 1. Business Meaning

EXW_dbo.EXW_WalletLogins tracks login events for eToro wallet users — customers who have an active crypto wallet (HasWallet=1 in DWH_dbo.Dim_Customer). Each row represents one login session by one wallet user on the eToro platform.

**What makes it wallet-specific**: The table is not a general login log. It filters the broader Fact_CustomerAction login stream to only wallet users via the `Dim_Customer.HasWallet=1` join. This makes it suitable for wallet product engagement analysis — how often wallet customers log in, session patterns, and cross-referencing with wallet transaction activity.

**Application context**: `ApplicationIdentifier` is hardcoded as `'retoro'` — identifying the primary eToro wallet/crypto application. `EnvironmentDetails` is always NULL (the field exists for potential platform/browser context but is not populated in this pipeline).

**Refresh pattern**: SP_WalletLogins is parameterized by `@date`. For each execution, it deletes all rows where `CAST(LoggedInOn AS DATE) = @date` then inserts fresh data from Fact_CustomerAction. This is an idempotent replace-by-day pattern — safe to re-run for the same date without creating duplicates.

**Relationship to EXW_FCA_UserLogin**: A similarly named table `EXW_dbo.EXW_FCA_UserLogin` also stores wallet user logins but focuses on FCA-regulated users. EXW_WalletLogins covers all wallet users regardless of regulatory jurisdiction.

## 2. Business Logic

### 2.1 Wallet User Filter

**What**: Only customers with an active wallet are included.

**Columns Involved**: `GCID`, `RealCID`

**Rules**:
- Filtered via `JOIN DWH_dbo.Dim_Customer dc ON fca.RealCID = dc.RealCID AND dc.HasWallet = 1`
- Excludes eToro customers who have not activated wallet/crypto features
- `GCID` and `RealCID` identify the same customer — GCID is the group-wide ID (used for HASH distribution), RealCID is the real-account customer ID

### 2.2 Login Event Sourcing

**What**: Rows come exclusively from Fact_CustomerAction login events.

**Columns Involved**: `LoggedInOn`, `SessionIdentifier`

**Rules**:
- `ActionTypeID = 14` = LoggedIn (from STS_Audit_UserOperationsData via Fact_CustomerAction ETL)
- `LoggedInOn` = `Fact_CustomerAction.Occurred` — the UTC timestamp of the login event
- `SessionIdentifier` = `Fact_CustomerAction.SessionID` — the session key generated by the Session Tracking Service
- `DateID = @dateID` limits the source read to the requested date's partition of Fact_CustomerAction

### 2.3 Delete-Insert Refresh Pattern

**What**: Each SP execution replaces all rows for the target date.

**Rules**:
- `DELETE WHERE CAST(LoggedInOn AS DATE) = @date` — removes prior write for this date
- Then `INSERT SELECT` from Fact_CustomerAction for the same date
- Safe to re-run (idempotent): duplicate execution produces the same result
- `UpdateDate` = `GETDATE()` records the ETL execution timestamp

## 3. Query Advisory

### 3.1 Synapse Distribution & Index

HASH(GCID) — optimal for per-user GROUP BY and JOINs to other GCID-distributed EXW tables (EXW_DimUser, Staking tables). HEAP is appropriate for a table queried with date-range filters (DateID or CAST(LoggedInOn AS DATE)).

### 3.2 Common Query Patterns

| Analyst Question | Recommended Approach |
|-----------------|---------------------|
| Daily wallet login count | `SELECT CAST(LoggedInOn AS DATE), COUNT(*) FROM EXW_WalletLogins GROUP BY CAST(LoggedInOn AS DATE)` |
| Unique wallet users who logged in (date range) | `SELECT COUNT(DISTINCT GCID) WHERE LoggedInOn BETWEEN @start AND @end` |
| Login sessions per user | `SELECT GCID, COUNT(SessionIdentifier) AS Sessions ... GROUP BY GCID` |
| Cross-reference with staking activity | JOIN to Staking_BI_Version_WalletUserRewards on GCID |

### 3.3 Common JOINs

| Join To | Join Condition | Purpose |
|---------|---------------|---------|
| EXW_dbo.EXW_DimUser | `d.GCID = l.GCID` | User demographics (country, registration date, club tier) |
| EXW_dbo.Staking_BI_Version_WalletUserRewards | `s.GCID = l.GCID` | Check if staking users were actively logging in |
| DWH_dbo.Dim_Customer | `dc.RealCID = l.RealCID` | Full customer profile |

### 3.4 Gotchas

- **EnvironmentDetails is always NULL**: Do not use this column for platform or browser analysis — it is not populated by SP_WalletLogins.
- **ApplicationIdentifier is always 'retoro'**: Only one application in this table. Include in queries for documentation but filtering by it adds no value.
- **LoggedInOn datetime precision**: Stored as datetime (not datetime2) — truncates to millisecond precision.
- **Partition filter**: For large date-range queries, filter on `CAST(LoggedInOn AS DATE)` or derive the integer DateID; direct datetime predicates may scan more data without partition pruning.
- **GCID nullable**: GCID is defined as nullable (int NULL) — filter `WHERE GCID IS NOT NULL` if joining to ensure distribution quality.
- **Backup table**: `EXW_WalletLogins_Backup_20241216` exists in the SSDT — indicates a one-time backup taken Dec 2024. Do not query this for production analysis.

## 7. Sample Queries

### 7.1 Daily Unique Wallet Users Who Logged In
```sql
SELECT
    CAST(LoggedInOn AS DATE) AS LoginDate,
    COUNT(DISTINCT GCID) AS UniqueWalletUsers,
    COUNT(*) AS TotalSessions
FROM [EXW_dbo].[EXW_WalletLogins]
WHERE LoggedInOn >= '2023-01-01'
GROUP BY CAST(LoggedInOn AS DATE)
ORDER BY LoginDate DESC;
```
