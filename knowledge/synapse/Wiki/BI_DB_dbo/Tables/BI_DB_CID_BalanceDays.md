# BI_DB_dbo.BI_DB_CID_BalanceDays

> Customer cohort balance table — 25 columns storing cumulative revenue, cumulative deposits, and point-in-time equity at 8 fixed milestones after each customer's First Time Deposit (D+1, D+7, D+14, D+30, D+60, D+90, D+180, D+365). One row per depositor. Columns are progressively filled daily as customers reach each milestone. Primary use case: LTV modeling and early customer engagement analysis.

## 1. Business Meaning

`BI_DB_dbo.BI_DB_CID_BalanceDays` is a **cohort milestone table** for early customer behavior analysis. It answers: "How much revenue, deposits, and equity did each customer generate/have at specific intervals after their first deposit?"

The table contains **one row per depositor**. Each row is created on the customer's First Time Deposit day (D+1) and the metric columns are progressively populated as the customer reaches each day-milestone:

| Milestone | Days after FTD | DateDiff value |
|-----------|---------------|----------------|
| D+1  | 0 days elapsed | 0 |
| D+7  | 6 days elapsed | 6 |
| D+14 | 13 days elapsed | 13 |
| D+30 | 29 days elapsed | 29 |
| D+60 | 59 days elapsed | 59 |
| D+90 | 89 days elapsed | 89 |
| D+180 | 179 days elapsed | 179 |
| D+365 | 364 days elapsed | 364 |

**Three metric families**:
1. **Revenue{N}days** — cumulative revenue generated by the customer from FTD through N days. Source: BI_DB_DailyCommisionReport (FullCommissions + RollOverFee for direct non-mirror trades in qualifying instrument types + all CopyTrading trades).
2. **Deposit{N}days** — cumulative deposit amount from FTD through N days. Source: Fact_CustomerAction ActionTypeID=7.
3. **Equity{N}days** — point-in-time equity snapshot ON the Nth day after FTD. Source: V_Liabilities.Liabilities + V_Liabilities.ActualNWA.

**Note**: Revenue and Deposit are CUMULATIVE (sum from FTD through N days). Equity is a SNAPSHOT (the value on exactly the Nth day, not cumulative).

**NULL semantics**: A NULL in any metric column means either: (a) the customer has not yet reached that milestone (e.g., a customer who deposited 10 days ago will have Deposit14days=NULL because they haven't hit D+14 yet), or (b) no activity occurred during that window (Revenue1day=NULL if no qualifying trades on FTD day).

**Revenue source change**: Since 2023-09-07, revenue source changed from BI_DB_CID_DailyPanel_FullData to BI_DB_DailyCommisionReport (per SP change history by Eti).

## 2. Business Logic

### 2.1 Population Selection

**What**: Only customers who have reached an exact milestone anniversary are processed each day.

**Rule**:
```sql
#pop = Dim_Customer
  WHERE IsDepositor = 1
    AND IsValidCustomer = 1
    AND DATEDIFF(DAY, FirstDepositDate, @yesterday) IN (0, 6, 13, 29, 59, 89, 179, 364)
```
On any given day, the SP processes exactly those customers whose FTD was 0, 6, 13, 29, 59, 89, 179, or 364 days ago. A customer with FTD yesterday is processed for Revenue1day. A customer with FTD 6 days ago is processed for Revenue7days.

### 2.2 Revenue Computation

**What**: Cumulative commissions and rollover fees from qualifying trades up to milestone day N.

**Formula** (from BI_DB_DailyCommisionReport):
```
Revenue_Total per day = 
  SUM(FullCommissions + RollOverFee WHERE IsMirror=0 AND InstrumentTypeID IN (1,2,4,5,6,10))  -- Direct trades
+ SUM(FullCommissions + RollOverFee WHERE IsMirror=1)                                          -- CopyTrading trades
```

**Cumulative rule**: For Revenue{N}days, the WHERE clause is `ActiveDate <= DATEADD(DAY, N-1, FTDdate)` — all days from FTD through day N-1 of customer lifetime (0-indexed, so "7 days" = days 0-6).

**InstrumentTypeID scope**: Direct trades are only counted for InstrumentTypeIDs {1,2,4,5,6,10}. Other instrument types (e.g., CFD) are excluded from Revenue for non-mirror trades.

### 2.3 Deposit Computation

**What**: Cumulative deposit amount from FTD through milestone day N.

**Source**: `Fact_CustomerAction WHERE ActionTypeID=7 (deposits)`.

**Cumulative rule**: `WHERE datediff <= N-1` where `datediff = DATEDIFF(DAY, FirstDepositDate, CAST(Occurred AS DATE))`.

### 2.4 Equity Computation

**What**: Point-in-time equity snapshot on the Nth day after FTD.

**Formula**: `V_Liabilities.Liabilities + V_Liabilities.ActualNWA` on the exact milestone date.

**Not cumulative**: Unlike Revenue and Deposit, Equity{N}days is a snapshot, not a sum. It reflects the customer's total holdings on that specific day.

**Equity365days special case**: ISNULL(Equity365days, 0) — this column is set to 0 when the equity join produces NULL at the 365-day milestone (other milestones leave NULL as-is).

### 2.5 Row Lifecycle

**What**: The SP creates a row on a customer's FTD day and updates it incrementally.

**Day 0** (DateDiff=0):
1. DELETE any existing row for this CID (handles re-runs)
2. INSERT row with only CID + UpdateDate (all metric columns NULL)
3. UPDATE Revenue1day, Deposit1day, Equity1day for this customer

**Subsequent milestones** (DateDiff = 6, 13, 29, ...):
- UPDATE the existing row with the new column value
- UpdateDate is refreshed at each UPDATE pass

**Result**: A 2-year-old customer's row has all 24 metric columns populated. A 5-day-old customer's row has only Revenue1day, Deposit1day, Equity1day populated; others are NULL.

## 3. Query Advisory

### 3.1 Distribution and Index

ROUND_ROBIN distribution. Clustered index on CID for fast per-customer lookups.

### 3.2 NULL Interpretation

**Critical distinction**: NULL can mean two very different things:
1. **Milestone not yet reached**: Customer is fewer than N days from FTD — column will be filled in the future
2. **Zero activity**: Milestone was reached but no qualifying activity occurred — column may remain NULL (Revenue columns) or 0 (Equity365days only)

Do not treat NULL as equivalent to 0 in cohort aggregations. Filter by FTD cohorts to get meaningful averages.

### 3.3 Revenue vs Deposit vs Equity

- **Revenue{N}days** = eToro's earnings FROM this customer (commissions + rollovers) — not the customer's P&L
- **Deposit{N}days** = money the customer moved INTO eToro — not revenue
- **Equity{N}days** = customer's total holdings at eToro on day N — a snapshot of wealth on platform

### 3.4 Common Query Patterns

| Analyst Question | Recommended Approach |
|-----------------|---------------------|
| Avg revenue by FTD cohort month | GROUP BY MONTH(Dim_Customer.FirstDepositDate), AVG(Revenue30days) WHERE Revenue30days IS NOT NULL |
| Revenue-to-deposit ratio at D+30 | SUM(Revenue30days)/SUM(Deposit30days) WHERE both NOT NULL |
| D+7 vs D+30 equity retention | AVG(Equity7days), AVG(Equity30days) for FTD cohort |
| Customers with no activity at D+30 | WHERE Revenue30days IS NULL AND Deposit30days IS NOT NULL |
| Fully mature cohorts (all columns filled) | WHERE Revenue365days IS NOT NULL |

## 7. Sample Queries

### 7.1 Average LTV by FTD month cohort

```sql
SELECT
    CONVERT(VARCHAR(7), dc.FirstDepositDate, 120) AS FTD_Month,
    COUNT(*) AS CohortSize,
    AVG(b.Revenue30days) AS Avg_Rev_30d,
    AVG(b.Revenue90days) AS Avg_Rev_90d,
    AVG(b.Revenue365days) AS Avg_Rev_365d_Mature
FROM [BI_DB_dbo].[BI_DB_CID_BalanceDays] b
JOIN [DWH_dbo].[Dim_Customer] dc ON b.CID = dc.RealCID
WHERE b.Revenue365days IS NOT NULL  -- Only mature cohorts (365-day milestone reached)
GROUP BY CONVERT(VARCHAR(7), dc.FirstDepositDate, 120)
ORDER BY FTD_Month DESC
```

### 7.2 Revenue/Deposit ratio at 30 days

```sql
SELECT
    SUM(Revenue30days) AS Total_Rev_30d,
    SUM(Deposit30days) AS Total_Dep_30d,
    CAST(SUM(Revenue30days) AS float) / NULLIF(SUM(Deposit30days), 0) AS Rev_Dep_Ratio_30d
FROM [BI_DB_dbo].[BI_DB_CID_BalanceDays]
WHERE Revenue30days IS NOT NULL
  AND Deposit30days IS NOT NULL
  AND Deposit30days > 0
```

### 7.3 Equity retention curve for 2025 FTD cohort

```sql
SELECT
    AVG(Equity1day) AS Day1,
    AVG(Equity7days) AS Day7,
    AVG(Equity30days) AS Day30,
    AVG(Equity90days) AS Day90,
    AVG(Equity365days) AS Day365
FROM [BI_DB_dbo].[BI_DB_CID_BalanceDays] b
JOIN [DWH_dbo].[Dim_Customer] dc ON b.CID = dc.RealCID
WHERE dc.FirstDepositDate >= '2025-01-01'
  AND dc.FirstDepositDate < '2026-01-01'
  AND b.Equity365days IS NOT NULL
```
