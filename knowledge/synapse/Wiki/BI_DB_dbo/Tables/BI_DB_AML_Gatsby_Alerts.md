## 1. Overview

AML transaction monitoring alert table for eToro's USA and UK **APEX Options** trading accounts. The table accumulates daily alerts generated by the Gatsby AML rules engine — a set of 6 pattern-detection rules that flag potentially suspicious deposit/withdrawal/trading activity on APEX SOD (Statement of Daily) files.

**Data scope**: APEX Options accounts only (OfficeCode='4GS'), covering USA (RegisteredRepCode='GAT') and UK (RegisteredRepCode='UK1') jurisdictions. APEX SOD data is available from November 2022 (when the eToro Options App launched in the USA).

**Important — current state (2026-04-22)**: All 808 rows in the table are sentinel "Dummy Line" rows — no real alerts have fired since the SP began running on 2024-01-17. Each daily run inserts exactly one dummy row to confirm the SP executed successfully for that date. Real alert rows would have a specific `Rule_Checked` value (DC10US-1A, DC10US-1B, DC5US, DC25US, DC2US, DC26US) and a real `AccountNumber`.

The 6 alert rules detect:
1. **DC10US-1A** — structuring via many deposits without trading (20+ deposits in 14 days > $50K total, no trades)
2. **DC10US-1B** — round-trip structuring (same as 1A but with 5+ withdrawals instead of no trades)
3. **DC5US** — rapid deposit structuring (10+ deposits in 48 hours > $25K, no trades)
4. **DC25US** — CTR smurfing (3+ deposits between $9,000–$9,999 in 10 days; no deposit ≥$10K, none <$8K — keeps deposits just below $10K reporting threshold)
5. **DC2US** — annual deposit threshold crossing (first time total yearly deposits exceed $250K)
6. **DC26US** — rapid deposit + small withdrawal structuring (10+ deposits in 48 hours > $25K + 2+ withdrawals all under $10K)

```
│   ├─ #firstdateover50   ← first date deposit_sum > $50K per account

│   └─ #firstdateover250  ← first date deposit_sum > $250K per account
│
├─ Alert computation (one temp table per rule):
│   ├─ #DC10US1A  ← 14-day window, 20+ deposits, >$50K, no trades
│   ├─ #DC10US1B  ← 14-day window, 20+ deposits, >$50K, 5+ withdrawals
│   ├─ #DC5US     ← 48-hour window, 10+ deposits, >$25K, no trades

│   ├─ #DC2US     ← calendar year deposits >$250K, first time on @Date
│   └─ #DC26US    ← 48-hour window, 10+ deposits, >$25K, 2+ withdrawals all <$10K
│
├─ #final  ← UNION ALL of all 6 rule outputs + 1 Dummy Line sentinel
│
└─ DELETE WHERE AlertDate=@Date + INSERT → BI_DB_AML_Gatsby_Alerts
```

## 4. Business Logic

### 4.1 Alert Rule Details

| Rule | Time Window | Trigger Condition | AML Concern |
|------|------------|-------------------|-------------|
| DC10US-1A | 14 days | deposit_count ≥ 20 AND deposit_sum > $50K (first time); NO trades in same window | Deposits without trading activity — suggests layering |
| DC10US-1B | 14 days | deposit_count ≥ 20 AND deposit_sum > $50K (first time); withdrawal_count ≥ 5 | Round-trip movement of funds |
| DC5US | 48 hours | deposit_count ≥ 10 AND deposit_sum > $25K (first time); NO trades in 48 hours | Rapid deposit structuring without trading |
| DC25US | 10 days | 3+ INDIVIDUAL deposits each $9,000–$9,999; 0 deposits ≥ $10K; 0 deposits < $8K | CTR threshold avoidance (smurfing) |
| DC2US | Calendar year | SUM(deposits) > $250K for the first time in the year, first date = @Date | Annual high-value threshold crossing |
| DC26US | 48 hours | deposit_count ≥ 10 AND deposit_sum > $25K; 2+ withdrawals each < $10K | Rapid deposit + sub-threshold withdrawal structuring |

### 4.2 Dummy Line Sentinel

Every SP run unconditionally inserts one row with:
- `Rule_Checked = 'Dummy Line'`, `AlertType = 'Dummy Line'`  
- `AccountNumber = 'Dummy Line'`, `GCID = 1111`, `CID = 1111`
- `DateOfBirth = GETDATE()` (not a real date of birth)

This sentinel ensures the `AlertDate` always appears in the table, confirming the SP ran for that date. Consumers **must filter `WHERE Rule_Checked <> 'Dummy Line'`** to retrieve real alerts only.

### 4.3 Region Determination

```sql
CASE 
    WHEN RegisteredRepCode = 'GAT' THEN 'USA'
    WHEN RegisteredRepCode = 'UK1' THEN 'UK'
END AS Region
```
House accounts (neither GAT nor UK1) are excluded by the `IS NOT NULL` check on the CASE expression.

### 4.4 APEX Data Availability

> *Note in SP*: "APEX SOD files only go back to November 1st, 2022 when the eToro Options App launched in the USA"

The SP has been running since 2024-01-17. Historical backfill is available from Nov 2022 using the commented-out WHILE loop in the SP header.

### 4.5 Customer ID Resolution

APEX accounts use an `AccountNumber` (format: 4GS#####). The SP resolves this to eToro's global IDs via:
```
AccountNumber → EXT1034 (for name/DOB)
AccountNumber → USABroker_Apex_Options.OptionsApexID → GCID → Dim_Customer.RealCID (→ CID)
```
The JOIN is LEFT-joined, so accounts not found in USABroker_Apex_Options will have NULL GCID and CID.

## 7. Usage Notes

- **Always filter `WHERE Rule_Checked <> 'Dummy Line'`** to exclude sentinel rows from analyses.
- **`DateOfBirth` is unreliable for dummy rows** — it stores `GETDATE()` for the sentinel. Only use `DateOfBirth` when `Rule_Checked <> 'Dummy Line'`.
- **`CID` and `GCID` are 1111 for dummy rows** — these are not real customer IDs.
- The table is **USA and UK APEX Options accounts only** — it does not cover the broader eToro platform or other jurisdictions.
- DC25US operates at the **individual transaction level** (not daily aggregate), making it distinct from the other rules which use daily aggregates from `#approved_deposit_since_unity`.
- **DC2US fires only once per account per year** — the `first date = @Date` condition (`fr.ProcessDate = @Date`) means it generates an alert only on the day a customer first crosses $250K, not on subsequent days above that threshold.
