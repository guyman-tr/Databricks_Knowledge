# Review Needed: BI_DB_dbo.BI_DB_PositionHoldingTime

## Tier 4 / Low-Confidence Items

No Tier 4 columns. All columns traced to SP code (Tier 2) or DWH dimension wikis (Tier 1).

## Reviewer Questions

1. **HoldingTime rounding intent**: The SP computes `DATEDIFF(mi, CAST(OpenOccurred AS DATE), CAST(CloseOccurred AS DATE))` — rounding to calendar-day granularity. Was this intentional (to avoid intra-day noise) or a bug? If precise holding time is needed, the formula should be `DATEDIFF(mi, OpenOccurred, CloseOccurred)` without the CAST. Confirm with the original author (Tom Boksenbojm).

2. **InstrumentID = ParentCID for copy rows**: The SP inserts `dm.ParentCID` into the `InstrumentID` column for Copy Trade and Copy Portfolio rows. This means the column carries two different semantic types (InstrumentID vs. customer CID) depending on InstrumentType. Is this the intended schema, or should a separate column (e.g., `ParentCID`) be added to avoid confusion?

3. **MirrorID=0 filter**: The SP only loads Dim_Position rows where `MirrorID=0` (non-copy positions). This means copy-mirrored positions from Dim_Position are excluded and tracked only from Dim_Mirror. Confirm that the Dim_Mirror branch correctly captures all positions that were generated by copy relationships (including multi-leg copy fund positions).

4. **Pre-2022 data gap**: The table starts at 2022-01-01 but eToro's position history in Dim_Position dates back to 2007. Is the 2022 start date intentional (new monitoring scope) or a data loading artifact that could be backfilled?

5. **NOLOCK hints in SP**: The SP uses `WITH (NOLOCK)` on both Dim_Position and Dim_Instrument reads. This can cause dirty reads if ETL is running concurrently. Is there a known risk of reading in-flight ETL data from these tables?

## Known Anomalies

- `HoldingTime` is always a multiple of 1,440 or 0 — this is not a bug but an architectural constraint of the CAST(datetime AS DATE) approach. Any HoldingTime analysis assuming clock-time precision will be incorrect.
- `InstrumentID` contains ParentCID values for Copy Trade/Portfolio rows — JOINs to Dim_Instrument without an InstrumentType filter will silently return wrong or missing data.
- `Leverage=0` for all copy rows — this constant value is not an informative dimension for copy rows.
- No Synapse index or partition on 368M rows — full table scans without CloseDate pre-filtering are very expensive.
