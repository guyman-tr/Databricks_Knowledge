## 1. Business Meaning

This table is a **cumulative compliance alert log** produced daily by `SP_Compliance_Forbidden_Trades`. Each row represents one alert event: a customer who triggered one of 30+ compliance rules on a given date. Rules are grouped into three families:

- **BU (Blocked Users)**: Actions by customers whose account is blocked.
- **BC (Blocked Countries)**: Customers from high-risk or sanctioned countries performing restricted actions (registrations, deposits, trades).
- **PC (Position/Country/Regulation Compliance)**: Customers who opened positions that violate regulation-level, country-level, or platform-level restrictions (leverage limits, restricted instruments, verification requirements, etc.).

The SP is designed to "catch all sorts of trades and actions which are supposed to be programmed into the app as not allowed but still might be occurring" (author comment). Alerts are reviewed by the Compliance team.

Not all columns are populated for every alert. Columns like PositionID, InstrumentType, InvestedAmount are NULL for non-position alerts (e.g., deposit-related BC rules). Similarly, TranID, AmountUSD, CryptoName are NULL for position-only alerts.

As of 2026-04-13: **259,077 rows** across **798 daily dates** from 2023-01-01 to 2026-04-11. 36 distinct AlertType codes active. Top alerts: PC51 'German clients without Video Identification cannot copy anyone' 22% (57,643), PC12 'Unverified Clients deposit after 15 days from FTD' 16% (40,461), PC17 'Retail Client Traded More Than The Max Leverage' 14% (36,438 combined), PC36 'Trades in high risk country' 10% (27,112), BC7 'Client Traded Sanctioned stock' 6% (16,754).

### 2.2 Customer Population (#pop)

**What**: All valid customers as of @Date form the base population for rule evaluation.
**Columns Involved**: `RealCID`, `Country`, `Regulation`, `MifidCategorization`, `Language`, `PlayerStatus`, `VerificationLevelID`, `BlockDate`
**Rules**:
- Source: `Fact_SnapshotCustomer WHERE IsValidCustomer=1 AND DateRangeID effective for @DateID`
- Joined with: Dim_Customer, Dim_Regulation, Dim_Country, Dim_Manager, Dim_MifidCategorization, Dim_Language, Dim_PlayerStatus, Dim_PlayerStatusReasons, Dim_PlayerStatusSubReasons, Dim_PlayerLevel, BI_DB_CIDFirstDates, External_etoro_BackOffice_Customer (FSA Seychelles categorization)
- `BlockDate`: last date customer's PlayerStatus entered a blocked state (PlayerStatusID IN 2,4,9,13,15), computed via Fact_SnapshotCustomer history LAG window

### 2.3 Load Pattern (Daily Incremental)

**What**: Each day's alerts overwrite the same date's slice; historical dates are immutable.
**Columns Involved**: `Date`, `RecordID`
**Rules**:
- `DELETE FROM BI_DB_Compliance_Illegal_Trades_Alerts WHERE Date = @Date` before INSERT
- `RecordID` is a running sequence: max(existing RecordID) + ROW_NUMBER() per batch. Ensures globally unique IDs but only for rows generated since 2025-03-09 (column addition date).

## 3. Query Advisory

### 3.1 Synapse Distribution & Index

HASH(RealCID) with CLUSTERED INDEX on Date. Customer-level lookups (WHERE RealCID = X) benefit from the hash distribution. Date-range scans use the clustered index. Cross-table JOINs with other HASH(RealCID) tables (Fact_SnapshotCustomer, Dim_Position) are colocation-efficient.

**Note**: RealCID is VARCHAR(100) in this table. JOINs to tables with int RealCID require explicit CAST.

### 3.2 Common Query Patterns

| Analyst Question | Recommended Approach |
|-----------------|----------------------|
| All alerts for a specific customer | `WHERE CAST(RealCID AS INT) = @cid` |
| Alert volume by rule over time | `SELECT Date, AlertType, COUNT(*) GROUP BY Date, AlertType ORDER BY Date` |
| Today's new alerts | `WHERE Date = (SELECT MAX(Date) FROM ...)` |
| Leverage violation alerts with position detail | `WHERE AlertType = 'PC17' AND PositionID IS NOT NULL` |
| Deposit-related alerts | `WHERE AlertType IN ('BC4','BC5','BC3','PC12','PC13','PC16')` |

### 3.3 Common JOINs

| Join To | Join Condition | Purpose |
|---------|----------------|---------|
| DWH_dbo.Dim_Position | `CAST(PositionID AS BIGINT) = dp.PositionID` | Enrich with full position detail |
| DWH_dbo.Dim_Customer | `CAST(RealCID AS INT) = dc.RealCID` | Get current customer state |
| DWH_dbo.Dim_Regulation | `Regulation = dr.Name` | Decode regulation metadata |

### 3.4 Gotchas

- **RealCID is VARCHAR(100)**: Despite the column name, RealCID is stored as text. Use `CAST(RealCID AS INT)` for JOINs or numeric comparisons.
- **Sparse columns per alert type**: Many columns (PositionID, InvestedAmount, TranID, CryptoName, etc.) are NULL for alert types that don't involve positions or deposits. Always check IS NOT NULL before using these columns.
- **RecordID NULL before 2025-03-09**: The RecordID column was added 2025-03-09. Rows from Jan 2023 to early March 2025 have NULL RecordID (not a data quality issue).
- **AccountMgr truncation**: In some rules, AccountMgr and other text fields are truncated to 10 characters (casting artifact). Do not use AccountMgr for exact name lookups.
- **DLT rules removed**: DLT1-DLT4 appear in some historical rows but were deactivated and removed (per 2025-11-04 change log). Historical rows with these AlertTypes still exist in the table.
- **UpdateDate is run timestamp**: All rows inserted in a single SP run share the same UpdateDate.
- **HASH on varchar**: HASH(RealCID) where RealCID is varchar(100) may produce unexpected distribution behavior if NULL or non-numeric values are present.

## 7. Sample Queries

### Latest day's alerts by rule type

```sql
SELECT AlertType,
       Synopsis,
       COUNT(*) AS alert_count
FROM [BI_DB_dbo].[BI_DB_Compliance_Illegal_Trades_Alerts]
WHERE Date = (SELECT MAX(Date) FROM [BI_DB_dbo].[BI_DB_Compliance_Illegal_Trades_Alerts])
GROUP BY AlertType, Synopsis
ORDER BY alert_count DESC;
```

### Leverage violation details for a specific date

```sql
SELECT Date,
       CAST(RealCID AS INT) AS RealCID,
       Country,
       Regulation,
       InstrumentType,
       IsSettled,
       Leverage,
       PositionID
FROM [BI_DB_dbo].[BI_DB_Compliance_Illegal_Trades_Alerts]
WHERE AlertType = 'PC17'
  AND Date = '2026-04-11'
ORDER BY CAST(Leverage AS INT) DESC;
```

### Alert trend by rule over the last 30 days

```sql
SELECT AlertType,
       Synopsis,
       COUNT(*) AS total_alerts,
       COUNT(DISTINCT CAST(RealCID AS INT)) AS distinct_customers
FROM [BI_DB_dbo].[BI_DB_Compliance_Illegal_Trades_Alerts]
WHERE Date >= DATEADD(DAY, -30, CAST(GETDATE() AS DATE))
GROUP BY AlertType, Synopsis
ORDER BY total_alerts DESC;
```
