## Findings

### 1. The one AML-named workbook

```
workbook_id: c992b919-6a3f-54be-525b-907efb8b0c63
workbook_name: "AML Clients – Trading & eMoney Net Deposits"
project: FCMU
owner: Georgios Kyriakou
updated_at: 2026-04-27T08:23:57Z
underlying_table: BI_DB_dbo.BI_DB_DDR_Fact_MIMO_AllPlatforms
```

This is an FCMU (Financial Crime / AML) dashboard that queries net deposits **filtered to CIDs flagged in the AML alert tables**. Custom SQL excerpt:

```sql
FROM BI_DB_dbo.BI_DB_DDR_Fact_MIMO_AllPlatforms mimo
WHERE mimo.RealCID IN (
    SELECT DISTINCT CID FROM BI_DB_dbo.BI_DB_AML_BI_Alerts_New
    UNION
    SELECT DISTINCT CID FROM BI_DB_dbo.BI_DB_RiskAlertManagementTool
    WHERE CategoryName = 'AML'
)
```

## Recommendation

For v1 (AML risk classification + scoring), add **`BI_DB_dbo.BI_DB_RiskAlertManagementTool`** to the D scope as a Tableau-surfaced AML node. It is the only in-scope discovery from the fly-over. Add it to `tools/skills/_seeds/compliance.yaml` `hub_tables` so the next `summarize_subgraph.py` run picks it up.

For v2 (KYC screening, post-v1 spec), pull in `BI_DB_OPS_VerificationLevel2Stuck` and `BI_DB_OPS_VerificationPipeline_OverLevel2` — these are the KYC operations queue tables. Defer to spec 012 or successor.

The Tableau under-weighting (0.5x in `merge_graph.py`) is **vindicated** for the D domain — Tableau has minimal AML/compliance coverage in the index, and what little it has (the FCMU AML workbook) is one analyst's custom-SQL aggregation, not a load-bearing semantic signal. The decision to keep Tableau at 0.5x stands.
