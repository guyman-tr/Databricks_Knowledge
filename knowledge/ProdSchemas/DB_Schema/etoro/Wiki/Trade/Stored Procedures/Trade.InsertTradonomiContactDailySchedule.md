## 1. Business Meaning

Trade.InsertTradonomiContactDailySchedule is the write endpoint for adding trading schedule windows to the Tradonomi instrument configuration. Each schedule window defines a time range (FromTime to ToTime) during which an instrument is available for trading. For instruments with restricted market hours (e.g., stock exchanges that close at 17:00) or instruments with split sessions (e.g., a maintenance window from 21:15-21:30), multiple schedule windows are created per instrument by calling this procedure once per window.

Without this procedure there would be no consistent way to populate Trade.TradonomiContractsDailySchedule. The ID generation is delegated to Internal.GetTradonomiContractsDailyScheduleID, which centralizes DailyScheduleID allocation (likely incrementing a sequence or identity-based generator). The caller receives the newly generated @DailyScheduleID as an OUTPUT parameter for immediate use or logging.

Data flow: Configuration tools or stored procedures that set up instrument trading parameters call this procedure for each session window of an instrument. For a 24-hour instrument, one call is made with FromTime=ToTime at midnight. For split-session instruments, two or more calls are made with non-overlapping windows. Trade.TradonomiContractsDailySchedule stores the results and is queried by the view Trade.GetTradonomiContractsDailySchedule.

## 2. Business Logic

### 2.1 ID Generation Before Insert

**What**: The DailyScheduleID is obtained from an internal ID generator before the INSERT, not from a table IDENTITY column.

**Columns/Parameters Involved**: `@DailyScheduleID` (OUTPUT)

**Rules**:
- Internal.GetTradonomiContractsDailyScheduleID is called first, setting @DailyScheduleID OUTPUT.
- The @DailyScheduleID is then used explicitly in the INSERT - it is not auto-generated by the table.
- If the ID generator fails, the CATCH block returns -1 and no INSERT occurs.
- The caller receives @DailyScheduleID even if the call succeeded - this allows the caller to track which schedule ID was assigned.

### 2.2 Return Code Convention

**What**: The procedure uses SQL RETURN codes rather than raising errors.

**Columns/Parameters Involved**: RETURN 0, RETURN -1

**Rules**:
- RETURN 0: Success - ID was generated and INSERT completed.
- RETURN -1: Any error (ID generation failure or INSERT failure) was caught. No explicit ROLLBACK - if the INSERT fails after the ID was obtained, the ID may be consumed without a corresponding row in the table.
- Callers must check the RETURN value to determine success/failure.

## 8. Sample Queries

### 8.1 Add a 24-hour trading schedule for an instrument

```sql
DECLARE @NewScheduleID INT;
EXEC Trade.InsertTradonomiContactDailySchedule
    @InstrumentID    = 1,
    @FromDate        = '2010-01-01 00:00:00',
    @ToDate          = '2010-01-01 00:00:00',
    @DailyScheduleID = @NewScheduleID OUTPUT;

SELECT @NewScheduleID AS AssignedScheduleID;
-- Returns the new DailyScheduleID
```

### 8.2 Add a split-session instrument (main + alternate session)

```sql
DECLARE @ID1 INT, @ID2 INT;

EXEC Trade.InsertTradonomiContactDailySchedule
    @InstrumentID    = 17,
    @FromDate        = '2010-04-08 00:00:00',
    @ToDate          = '2010-04-08 21:15:00',
    @DailyScheduleID = @ID1 OUTPUT;

EXEC Trade.InsertTradonomiContactDailySchedule
    @InstrumentID    = 17,
    @FromDate        = '2010-04-08 21:30:00',
    @ToDate          = '2010-04-08 00:00:00',
    @DailyScheduleID = @ID2 OUTPUT;

SELECT @ID1 AS MainSessionID, @ID2 AS AltSessionID;
```

### 8.3 Verify the schedule after insertion

```sql
SELECT
    DailyScheduleID,
    InstrumentID,
    FromTime,
    ToTime
FROM Trade.TradonomiContractsDailySchedule WITH (NOLOCK)
WHERE InstrumentID = 17
ORDER BY DailyScheduleID;
```
