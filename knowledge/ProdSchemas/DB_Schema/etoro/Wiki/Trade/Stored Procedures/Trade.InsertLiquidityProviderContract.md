# Trade.InsertLiquidityProviderContract

> Creates a single liquidity provider contract mapping (instrument + provider + exchange + ticker + validity window), returns the auto-generated ContractID as an OUTPUT parameter, and returns 0 on success or -1 with error message on failure.

## 1. Business Meaning

Trade.InsertLiquidityProviderContract is the atomic row-insertion endpoint for Trade.LiquidityProviderContracts. It creates one contract record that maps an instrument to a specific liquidity provider, exchange, and ticker symbol for a defined validity window. The caller receives the new ContractID via an OUTPUT parameter, which is needed for downstream steps such as Trade.TradonomiToLiquidityProviderContracts.

Each contract row tells the price and hedge subsystems: "for InstrumentID X, liquidity provider type Y uses ticker Z on exchange E from date A to date B." This enables external ticker resolution for price feeds and hedge order routing. Different providers may use different ticker conventions (e.g., EUR/USD vs EURUSD vs EURUSD.fx), so one instrument can have multiple contract rows - one per active provider.

Data flow in instrument onboarding: After InsertInstrumentMetadataSecurityOpsAPI creates the core instrument record, liquidity provider contracts are added via either this SP (one-by-one) or InsertInstrumentRealTable (bulk, reads from ## temp tables populated by CheckValidInstruments).

## 2. Business Logic

### 2.1 INSERT + @@identity - ContractID Return

**What**: Inserts the contract row then captures the IDENTITY value to return to the caller.

**Columns/Parameters Involved**: `@ContractID OUTPUT`, `@@identity`

**Rules**:
- INSERT uses explicit column list: LiquidityProviderID, InstrumentID, FromDate, ToDate, Ticker, ExchangeID, RateConversionFactor.
- ContractID column is IDENTITY - auto-generated by the table.
- `SET @ContractID = @@identity` immediately after INSERT captures the generated ContractID.
- Uses `@@identity` (not `SCOPE_IDENTITY()`): safe here as the procedure has no triggers on LiquidityProviderContracts, but this is a legacy pattern.
- RETURN(0) on success; RETURN(-1) on CATCH (with `SELECT ERROR_MESSAGE()` to surface the error as a result set).

### 2.2 Try/Catch Error Handling

**What**: Errors during INSERT are caught and returned as -1 with error text.

**Rules**:
- BEGIN TRY / END TRY + BEGIN CATCH / END CATCH wraps the INSERT.
- On CATCH: `SELECT ERROR_MESSAGE()` returns the error string as a single-row result set (not RAISERROR).
- RETURN(-1) signals failure to the caller programmatically.
- No explicit transaction - single-statement INSERT is auto-commit; no ROLLBACK needed.

### 2.3 Contract Validity Window

**What**: FromDate/ToDate define when this provider-instrument-exchange mapping is active.

**Columns/Parameters Involved**: `@FromDate`, `@ToDate`

**Rules**:
- Open-ended (active) contracts use ToDate = '9999-12-31' or '2100-01-01'.
- Price and hedge subsystems filter by date range to find active contracts.
- The same InstrumentID + LiquidityProviderID + ExchangeID combination must not exist already (PK constraint). If a duplicate key scenario arises, the CATCH block surfaces the error.

## 8. Sample Queries

### 8.1 Insert a liquidity provider contract and capture ContractID

```sql
DECLARE @NewContractID INT;

EXEC Trade.InsertLiquidityProviderContract
    @InstrumentID          = 9999,
    @LiquidityProviderID   = 2,          -- e.g. FXCM
    @FromDate              = '2026-01-01',
    @ToDate                = '9999-12-31',
    @Ticker                = 'EXMP.US',
    @ExchangeID            = 4,          -- US exchange
    @RateConversionFactor  = 1.0,
    @ContractID            = @NewContractID OUTPUT;

SELECT @NewContractID AS NewContractID;
```

### 8.2 Verify the created contract

```sql
SELECT ContractID, InstrumentID, LiquidityProviderID, ExchangeID, Ticker, FromDate, ToDate, RateConversionFactor
FROM   Trade.LiquidityProviderContracts WITH (NOLOCK)
WHERE  InstrumentID = 9999;
```
