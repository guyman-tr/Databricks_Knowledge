## 1. Business Meaning

Trade.InsertCopyTradeSettlementRestrictions is the **write gateway for copy-trading settlement restriction configuration**. Operations teams and admin tools submit a batch of restriction rows via the typed TVP `Trade.CopyTradeSettlementRestrictionsTbl`, the SP validates dictionary integrity, then bulk-inserts them into the system-versioned table `Trade.CopyTradeSettlementRestrictions`.

This SP exists to give a safe, validated entrypoint for adding jurisdiction-, regulation-, or asset-scoped restrictions that control which instruments customers can copy-trade. The sibling `Trade.InsertCopyTradeSettlementRestrictions_TRDOPS` offers the same function but through the Trading Operations (TRDOPS) access path, which typically implies a different permission context. By centralizing the insert logic here, all callers benefit from the same validation: no orphaned RestrictionTypeIDs can enter the table.

Data flow: caller constructs a TVP of type `Trade.CopyTradeSettlementRestrictionsTbl` and calls this SP. If `@AppLoginName` is provided, its binary-cast value is written to `CONTEXT_INFO` so the system-versioning trigger can record the operator who made the change. After validation, all rows from the TVP are inserted in one batch.

## 2. Business Logic

### 2.1 RestrictionTypeID Validation (Dictionary Integrity Guard)

**What**: Rejects the entire batch if any row contains a RestrictionTypeID that does not exist in Dictionary.RestrictionType.

**Columns/Parameters Involved**: `RestrictionTypeID` (in TVP), `Dictionary.RestrictionType.RestrictionTypeID`

**Rules**:
- `IF EXISTS (SELECT 1 FROM @RequestedRestrictionsTable r WHERE NOT EXISTS (SELECT 1 FROM Dictionary.RestrictionType d WHERE d.RestrictionTypeID = r.RestrictionTypeID))`
- If any row fails: `RAISERROR('One or more RestrictionTypeID values do not exist in Dictionary.RestrictionType.', 16, 1); RETURN`
- All-or-nothing: even one invalid RestrictionTypeID aborts the entire insert

### 2.2 Operator Audit via CONTEXT_INFO

**What**: Propagates the caller's login name into SQL Server session context so system-versioning triggers and auditing mechanisms can record who made the change.

**Columns/Parameters Involved**: `@AppLoginName VARCHAR(50)`

**Rules**:
- If `@AppLoginName != ''`: `DECLARE @OpsUserInfo VARBINARY(128) = CAST(@AppLoginName AS VARBINARY(128)); SET CONTEXT_INFO @OpsUserInfo`
- Default is empty string (no context info set - anonymous/application call)
- System-versioning on Trade.CopyTradeSettlementRestrictions captures changes to History.CopyTradeSettlementRestrictions; CONTEXT_INFO lets triggers identify the human operator

### 2.3 Bulk Insert from TVP

**What**: Inserts all rows from the TVP into the target table in a single statement.

**Columns Inserted**: `CountryID`, `RegulationID`, `InstrumentTypeID`, `ExchangeID`, `InstrumentID`, `RestrictionTypeID`, `UnblockReasonId`, `GroupID`, `RegistrationDate`

**Rules**:
- Direct SELECT from `@RequestedRestrictionsTable` into `Trade.CopyTradeSettlementRestrictions`
- No row-by-row processing; single INSERT-SELECT
- The target table's `CK_CopyTradeSettlementRestriction_Asset` CHECK constraint still applies (at least one of InstrumentTypeID/ExchangeID/InstrumentID/GroupID must be non-NULL)
- `ID` (PK IDENTITY) and `UpdatedDate` are auto-generated by the table
