# Billing.PaymentActionAdd

> Records a payment action (authorization, purchase, refund, etc.) against a legacy Billing.Payment record in History.PaymentAction, assigning a per-customer-unique TransactionID and returning the new action identifier.

## 1. Business Meaning

`Billing.PaymentActionAdd` is the write entry point for `History.PaymentAction`, the action audit trail for the legacy `Billing.Payment` table. Every payment processing step (pre-authorization, purchase, refund, cashout, etc.) against a legacy payment record is recorded here. The procedure is part of the pre-2011 payment infrastructure - `Billing.Payment` was frozen after migration to `Billing.Deposit`, so `PaymentActionAdd` applies to historical payment records only.

The procedure ensures each payment action has a TransactionID that is unique within the customer's payment history. A 6-character alphanumeric TransactionID is generated by `Billing.GetTransactionID()` and collision-checked against existing `History.PaymentAction` rows joined to `Billing.Payment` for the same CID. If a collision is detected, it retries with `dbo.GetTransactionID()` (the earlier schema's version) until a unique value is found.

The procedure returns the `TransactionID` as a result set (SELECT) in addition to the `@PaymentActionID` OUTPUT and always returns 0 (success) as the RETURN code.

## 2. Business Logic

### 2.1 TransactionID Uniqueness - Per-Customer Collision Check

**What**: Guarantees no two payment actions for the same customer share a TransactionID.

**Parameters Involved**: `@CID`, `@PaymentID`

**Rules**:
- TransactionID is 6 characters (CHAR(6)) generated by `Billing.GetTransactionID()`
- Collision check: EXISTS on History.PaymentAction JOIN Billing.Payment WHERE TransactionID = @TransactionID AND Billing.Payment.CID = @CID
- On collision, retries with `dbo.GetTransactionID()` (legacy function, different schema)
- Loop continues until a unique TransactionID is found (in practice, collisions are extremely rare with a 6-char space)
- @CID is NOT stored in History.PaymentAction - it is only used in the collision check via Billing.Payment.CID

### 2.2 Payment Action Types and Statuses

**What**: Records the type and status of each payment processing step.

**Parameters Involved**: `@PaymentActionStatusID`, `@PaymentActionTypeID`

**Rules**:
- `PaymentActionTypeID` (from Dictionary.PaymentActionType):
  - 1 = PreAuthorization - card/account authorization hold
  - 2 = Purchase - actual charge/debit
  - 3 = Cashout - funds withdrawal to customer
  - 4 = Refund - reversal/chargeback
  - 5 = Settle - settlement confirmation
  - 6 = PostBack - payment gateway callback record
  - 7 = Cancel - authorization/transaction cancellation
- `PaymentActionStatusID` (from Dictionary.PaymentActionStatus):
  - 1 = New - action initiated
  - 2 = InProcess - processing with gateway
  - 3 = Closed - action finalized

### 2.3 Amount and Date Handling

**Parameters Involved**: `@Amount`, `@PaymentDate`

**Rules**:
- `@Amount` stored as INTEGER in smallest currency unit (cents). Consistent with Billing.Payment.Amount convention.
- `@PaymentDate` defaults to GETDATE() if NULL - records the UTC server time of action creation
- RETURN is always 0 (no error propagation - unlike Billing.NetellerAdd which returns @@ERROR)

## 8. Sample Queries

### 8.1 Record a new purchase action for a legacy payment

```sql
DECLARE @NewActionID INTEGER;
EXEC Billing.PaymentActionAdd
    @PaymentActionID     = @NewActionID OUTPUT,
    @CID                 = 123456,
    @PaymentID           = 9876,
    @PaymentActionStatusID = 1,  -- New
    @PaymentActionTypeID = 2,    -- Purchase
    @Amount              = 10000, -- $100.00
    @PaymentDate         = NULL;  -- defaults to now
SELECT @NewActionID AS NewPaymentActionID;
```

### 8.2 Review all actions for a legacy payment record

```sql
SELECT
    hpma.PaymentActionID,
    hpma.PaymentID,
    hpma.TransactionID,
    hpma.Amount,
    hpma.PaymentDate,
    pas.Name AS Status,
    pat.Name AS ActionType
FROM History.PaymentAction hpma WITH (NOLOCK)
INNER JOIN Dictionary.PaymentActionStatus pas WITH (NOLOCK) ON pas.PaymentActionStatusID = hpma.PaymentActionStatusID
INNER JOIN Dictionary.PaymentActionType pat WITH (NOLOCK) ON pat.PaymentActionTypeID = hpma.PaymentActionTypeID
WHERE hpma.PaymentID = 9876
ORDER BY hpma.PaymentDate;
```

### 8.3 Count legacy payment actions by type

```sql
SELECT
    pat.Name AS ActionType,
    COUNT(*) AS ActionCount
FROM History.PaymentAction hpma WITH (NOLOCK)
INNER JOIN Dictionary.PaymentActionType pat WITH (NOLOCK) ON pat.PaymentActionTypeID = hpma.PaymentActionTypeID
GROUP BY pat.Name
ORDER BY ActionCount DESC;
```
