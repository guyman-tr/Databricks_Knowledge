## 1. Business Meaning

Billing.MemberLimitAdd creates a new deposit velocity limit configuration row in Billing.MemberLimit. Each row defines, for a given eToro Club loyalty tier, the maximum number of deposits (transaction count) and maximum total deposit value (in cents) allowed per day, week, and month. Billing.CheckMemberLimit enforces these limits at deposit time.

This procedure is typically called by a Billing backoffice or admin tool when a new loyalty tier needs limits configured, or when a tier's limits need to be reset by adding a new row (rather than updating the existing one). Currently only one row exists in Billing.MemberLimit (Bronze / PlayerLevelID=1), suggesting this procedure has been called very infrequently.

The procedure returns the newly created MemberLimitID via @LimitID OUTPUT, allowing callers to reference or log the new row.

## 2. Business Logic

### 2.1 Limit Row Creation

**What**: Inserts a new tier limit row with all six velocity thresholds.

**Columns/Parameters Involved**: `@PlayerLevelID`, `@DailyTransaction`, `@DailyAmount`, `@WeeklyTransaction`, `@WeeklyAmount`, `@MonthlyTransaction`, `@MonthlyAmount`

**Rules**:
- Inserts into Billing.MemberLimit: (PlayerLevelID, DailyTransaction, DailyAmount, WeeklyTransaction, WeeklyAmount, MonthlyTransaction, MonthlyAmount).
- MemberLimitID is auto-generated by IDENTITY - returned via SCOPE_IDENTITY() into @LimitID OUTPUT.
- Amount columns are in CENTS (multiply USD by 100). E.g., $30,000 limit = 3,000,000.
- Transaction count columns are raw counts (no unit conversion).
- Returns @LocalError = @@ERROR (0 on success). Uses SET NOCOUNT ON.
- No duplicate check - if a row already exists for @PlayerLevelID, a second row will be inserted. Billing.CheckMemberLimit finds the row by PlayerLevelID using a WHERE clause - two rows for the same tier could cause ambiguous behavior.
- No TRY/CATCH - errors propagate unhandled to the caller.

**Enforcement semantics** (from Billing.CheckMemberLimit):
- DailyTransaction: max deposits per calendar day (@CheckResult=5 when exceeded)
- DailyAmount: max deposit value per calendar day in cents (@CheckResult=6 when exceeded)
- WeeklyTransaction: max deposits per calendar week (Mon-Sun, @CheckResult=3)
- WeeklyAmount: max deposit value per calendar week in cents (@CheckResult=4)
- MonthlyTransaction: max deposits per calendar month (@CheckResult=1)
- MonthlyAmount: max deposit value per calendar month in cents (@CheckResult=2)
- Monthly limits are checked first, then weekly, then daily. First violation stops processing.

## 8. Sample Queries

### 8.1 Add a new limit row for Silver tier
```sql
DECLARE @NewLimitID INTEGER;
EXEC Billing.MemberLimitAdd
    @LimitID           = @NewLimitID OUTPUT,
    @PlayerLevelID     = 5,          -- Silver
    @DailyTransaction  = 200,        -- 200 deposits/day
    @DailyAmount       = 10000000,   -- $100,000/day
    @WeeklyTransaction = 2000,
    @WeeklyAmount      = 50000000,   -- $500,000/week
    @MonthlyTransaction= 10000,
    @MonthlyAmount     = 200000000;  -- $2,000,000/month
SELECT @NewLimitID AS NewMemberLimitID;
```

### 8.2 View configured limits after adding
```sql
SELECT
    ml.MemberLimitID,
    pl.Name AS PlayerLevel,
    ml.DailyTransaction, ml.DailyAmount / 100.0 AS DailyAmountUSD,
    ml.WeeklyTransaction, ml.WeeklyAmount / 100.0 AS WeeklyAmountUSD,
    ml.MonthlyTransaction, ml.MonthlyAmount / 100.0 AS MonthlyAmountUSD
FROM Billing.MemberLimit ml WITH (NOLOCK)
JOIN Dictionary.PlayerLevel pl WITH (NOLOCK) ON ml.PlayerLevelID = pl.PlayerLevelID
ORDER BY pl.Sort;
```

### 8.3 Verify current Bronze limits (only row in production)
```sql
SELECT * FROM Billing.MemberLimit WITH (NOLOCK) WHERE PlayerLevelID = 1;
-- MemberLimitID=1, DailyTransaction=100, DailyAmount=3000000, etc.
```
