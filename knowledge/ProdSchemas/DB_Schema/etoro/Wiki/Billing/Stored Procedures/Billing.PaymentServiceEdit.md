## 1. Business Meaning

`Billing.PaymentServiceEdit` is the administration procedure for `Billing.PaymentService` - the table that stores back-office payment reporting portal configurations. Back-office operators use this procedure to register new payment service providers or update the reporting credentials and status of existing ones.

A "payment service" in this context is NOT a payment processing gateway but a **reporting integration**: the URL and credentials used to retrieve transaction status reports from providers like PayPal, googess (payment aggregator), or wire transfer portals. When a provider rotates credentials or changes their reporting API URL, this procedure is called to update the configuration.

The procedure conditionally skips the DML if any required field is empty or the status is -1 (used as a sentinel "no change" value). The caller detects success or failure via the result set (SELECT @@ROWCOUNT) rather than the @RowsAffected OUTPUT parameter (which is declared but never populated - a legacy bug).

## 2. Business Logic

### 2.1 Conditional Upsert - Validation Guard

**What**: Only executes DML if all required fields are populated and status is not -1.

**Parameters Involved**: All parameters

**Rules**:
- Validation: ALL of @Name, @ReportUrl, @ReportUserName, @ReportPassword must be non-null/non-empty AND @PaymentServiceStatusID != -1
- If validation fails: procedure exits silently (no DML, no error, RETURN 0) - the SELECT @@ROWCOUNT is not executed
- @PaymentServiceStatusID = -1 is a sentinel value meaning "skip update" (no enumeration in data - only used as guard)
- @PaymentServiceID = 0: INSERT new service row (identity auto-generated by Billing.PaymentService)
- @PaymentServiceID != 0: UPDATE existing service row WHERE PaymentServiceID = @PaymentServiceID

### 2.2 RowsAffected OUTPUT Bug

**What**: @RowsAffected OUTPUT parameter is declared but never assigned inside the procedure.

**Rules**:
- The result is exposed only via `SELECT @@ROWCOUNT` (result set)
- @RowsAffected remains NULL in all cases
- Callers relying on the OUTPUT parameter receive NULL - must use the result set instead
- This is a legacy interface issue - the OUTPUT parameter is vestigial

## 8. Sample Queries

### 8.1 View current payment service configurations

```sql
SELECT
    ps.PaymentServiceID,
    ps.Name,
    ps.ReportUrl,
    ps.ReportUserName,
    pss.Name AS Status
FROM Billing.PaymentService ps WITH (NOLOCK)
INNER JOIN Dictionary.PaymentServiceStatus pss WITH (NOLOCK) ON pss.PaymentServiceStatusID = ps.PaymentServiceStatusID
ORDER BY ps.PaymentServiceID;
```

### 8.2 Add a new payment reporting service

```sql
DECLARE @Affected INTEGER;
EXEC Billing.PaymentServiceEdit
    @PaymentServiceID      = 0,          -- INSERT
    @Name                  = 'NewPSP',
    @ReportUrl             = 'https://reports.newpsp.com/api',
    @ReportUserName        = 'etoro_api',
    @ReportPassword        = 'securepass',
    @PaymentServiceStatusID = 1,          -- Active
    @RowsAffected          = @Affected OUTPUT;
-- Note: @Affected will be NULL (bug) - check result set instead
```

### 8.3 Update an existing service's reporting URL

```sql
EXEC Billing.PaymentServiceEdit
    @PaymentServiceID      = 2,          -- UPDATE service ID 2
    @Name                  = 'PayPal',
    @ReportUrl             = 'https://reports.paypal.com/new-endpoint',
    @ReportUserName        = 'etoro_paypal',
    @ReportPassword        = 'newpassword',
    @PaymentServiceStatusID = 1,
    @RowsAffected          = NULL;
```
