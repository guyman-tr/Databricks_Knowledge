## 1. Business Meaning

`Billing.Deposit_GetDepositIdByTransactionId` resolves a deposit's internal `DepositID` from the pair of (`TransactionID`, `CID`). It is the lookup bridge between the 6-character `TransactionID` - the external reference generated by `Billing.DepositAdd` and shared with payment providers - and `DepositID`, eToro's internal surrogate key for the deposit ledger.

The procedure was created as a "PayPal blocking SP" (ticket 45017). PayPal callbacks reference deposits by the TransactionID that eToro sent during the initial deposit creation, but internal billing processing needs the DepositID to call downstream procedures (`Billing.DepositActionAdd`, `Billing.DepositUpdate`, etc.). This procedure is the translation layer.

The `WITH(NOLOCK)` hint and `SELECT TOP 1` pattern indicate this is a read-only, non-blocking lookup. `TOP 1` is a safety guard: TransactionID is generated uniquely per CID in `Billing.DepositAdd` (GUID-substring loop), but the combination of `TransactionID + CID` is not enforced as a unique constraint at the table level.

If no matching deposit is found, `@DepositID` remains NULL (not explicitly initialized to 0 or -1), so callers must check for NULL output.

## 2. Business Logic

### 2.1 TransactionID-to-DepositID Lookup

**What**: Resolves the internal DepositID for a given payment provider transaction reference and customer combination.

**Columns/Parameters Involved**: `@TransactionID`, `@CID`, `Billing.Deposit.TransactionID`, `Billing.Deposit.CID`, `Billing.Deposit.DepositID`

**Rules**:
- Queries `Billing.Deposit WITH(NOLOCK)` - non-blocking, eventually consistent
- Filters: `TransactionID = @TransactionID AND CID = @CID`
- `SELECT TOP 1` - returns at most one row; if multiple rows exist (data anomaly), returns the one with lowest physical order (no ORDER BY defined)
- If no row matches: `@DepositID` OUTPUT parameter is not set -> callers receive NULL
- `@TransactionID` is CHAR(6), matching the format produced by `Billing.DepositAdd`: `SUBSTRING(CONVERT(VARCHAR(36), NEWID()), 30, 6)`
- CID scoping prevents cross-customer TransactionID collision (GUID substrings are short enough that collisions across customers are possible over long timeframes)

**Diagram**:
```
@TransactionID (CHAR(6)) + @CID
         |
  SELECT TOP 1 DepositID
  FROM Billing.Deposit WITH(NOLOCK)
  WHERE TransactionID=@TransactionID AND CID=@CID
         |
  @DepositID OUTPUT
  (NULL if not found)
```

## 8. Sample Queries

### 8.1 Resolve DepositID from a PayPal TransactionID

```sql
DECLARE @DepositID INT;
EXEC Billing.Deposit_GetDepositIdByTransactionId
    @TransactionID = 'AB1C2D',
    @CID = 12345678,
    @DepositID = @DepositID OUTPUT;
SELECT @DepositID AS ResolvedDepositID;
-- NULL if not found, integer if found
```

### 8.2 Check how many TransactionIDs exist for a customer (detect duplicates)

```sql
SELECT TransactionID, COUNT(*) AS RecordCount
FROM Billing.Deposit WITH (NOLOCK)
WHERE CID = 12345678
GROUP BY TransactionID
HAVING COUNT(*) > 1;
-- Should return no rows if TransactionID uniqueness holds per CID
```

### 8.3 Verify a TransactionID exists before calling

```sql
SELECT DepositID, TransactionID, CID, PaymentStatusID, Amount, PaymentDate
FROM Billing.Deposit WITH (NOLOCK)
WHERE TransactionID = 'AB1C2D'
  AND CID = 12345678;
```
