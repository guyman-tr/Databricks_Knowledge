## 1. Business Meaning

Billing.PWMBAddAccountRequest tracks the process of customers linking their bank account via the PayWithMyBank (PWMB) Open Banking payment method. PayWithMyBank is a UK Open Banking provider that allows customers to make payments directly from their bank account without a card. When a customer initiates a bank account link request, an ExternalTransactionID is generated by the PWMB provider and stored here as the primary key, along with the customer's CID and (eventually) their FundingID once the account is created.

The table records 4,231 requests from Jan-Oct 2023, predominantly in terminal states (6=3,613 and 7=611 account for 99% of rows). The Status progression tracks the PWMB provider's verification and account-linking workflow. Every INSERT or UPDATE is mirrored to History.PWMBAddAccountRequest via the OUTPUT INTO clause in InsertPWMBAddAccountRequest and UpdatePWMBAddAccountRequest, providing a complete audit trail of state transitions without needing a trigger.

The Jira reference RD-7470 (Adi Cohen, Jun 2019) marks the initial implementation.

## 2. Business Logic

### 2.1 Account Link Request Lifecycle

**What**: Each row tracks one bank account link attempt from initiation to completion.

**Columns/Parameters Involved**: `ExternalTransactionID`, `CID`, `FundingID`, `Status`

**Rules**:
- ExternalTransactionID: The PWMB provider's transaction reference (up to 15 chars). This is the PK - set at creation time from the provider, not generated internally (no IDENTITY).
- FundingID: NULL on initial INSERT (bank account not yet created). Updated via UpdatePWMBAddAccountRequest once the account is successfully linked (FundingID assigned).
- Status progression: observed final states 6 and 7 dominate. Status=4 (1 row) and Status=8 (6 rows) are rare edge cases.
- History mirror: Both INSERT and UPDATE output to History.PWMBAddAccountRequest via OUTPUT INTO. This means every state change is preserved in history - unlike trigger-based history, this is SP-enforced.

### 2.2 Insert and Update Pattern

**What**: Two SPs handle the full lifecycle with history capture on each operation.

**Columns/Parameters Involved**: `ExternalTransactionID`, `Status`, `FundingID`, `InsertedTime`, `LastModificationTime`

**Rules**:
- InsertPWMBAddAccountRequest: Inserts with provided Status. Sets both InsertedTime and LastModificationTime to GETUTCDATE(). Outputs inserted row to History.
- UpdatePWMBAddAccountRequest: Updates FundingID (ISNULL-preserved) and Status. Sets LastModificationTime to GETUTCDATE(). Outputs updated row to History. WHERE ExternalTransactionID = @ExternalTransactionID.
- No MERGE - separate INSERT and UPDATE SPs. ExternalTransactionID uniquely identifies requests.

## 8. Sample Queries

### 8.1 Look up a specific PWMB account link request

```sql
EXEC Billing.GetPWMBAddAccountRequest @ExternalTransactionID = 'PWMB123456789'
```

### 8.2 Get all pending requests (non-terminal statuses)

```sql
SELECT CID, FundingID, Status, ExternalTransactionID, InsertedTime, LastModificationTime
FROM Billing.PWMBAddAccountRequest WITH (NOLOCK)
WHERE Status NOT IN (6, 7)  -- exclude dominant terminal states
ORDER BY InsertedTime DESC
```

### 8.3 Account link success rate by month

```sql
SELECT YEAR(InsertedTime) AS Yr, MONTH(InsertedTime) AS Mo,
       SUM(CASE WHEN Status = 6 THEN 1 ELSE 0 END) AS Linked,
       COUNT(1) AS Total
FROM Billing.PWMBAddAccountRequest WITH (NOLOCK)
GROUP BY YEAR(InsertedTime), MONTH(InsertedTime)
ORDER BY Yr, Mo
```
