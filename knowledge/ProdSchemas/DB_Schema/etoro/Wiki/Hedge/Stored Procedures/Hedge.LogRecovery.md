# Hedge.LogRecovery

> Inserts one row into Hedge.RecoveryLog for a single instrument reconciliation event during a hedge server recovery session, capturing the before/after position state (PrevUnits/NewUnits) and the reconciliation action state (Added/Updated/Removed/Detected).

## 1. Business Meaning

`Hedge.LogRecovery` is the write path for `Hedge.RecoveryLog`, the audit log of hedge position reconciliation events. It is called during hedge server recovery sessions - typically when the hedge server reconnects to a liquidity provider after an outage or restart, compares its internal position state with the LP's actual positions, and logs each reconciliation action.

Each call to this procedure records one instrument's reconciliation outcome within a recovery session:
- The `@RecoveryID` GUID groups all entries from the same recovery session together.
- The `@State` parameter (Dictionary.HedgeRecoveryState) classifies the action: 1=Added (new LP position adopted), 2=Updated (position corrected to match LP), 3=Removed (LP position no longer exists), 4=Detected (position verified as matching).
- The `@Prev*` parameters capture the position state BEFORE the reconciliation action; `@New*` capture the state AFTER.

The procedure is a thin persistence layer with TRY/CATCH THROW for error re-propagation. Recovery sessions are infrequent (only during server restart or LP reconnection), which is why `Hedge.RecoveryLog` is typically empty or contains sparse historical data.

## 2. Business Logic

### 2.1 Recovery Session Audit Pattern

**What**: Multiple calls with the same @RecoveryID form a complete audit record of one recovery session.

**Columns/Parameters Involved**: `@RecoveryID`, `@HedgeServerID`, `@LiquidityAccountID`, `@InstrumentID`, `@State`

**Rules**:
- `@RecoveryID` (GUID): generated by the application and shared across all EXEC calls for the same recovery session. Groups all per-instrument entries for one recovery run.
- One session typically produces one row per instrument processed (one entry per State per instrument, or multiple State entries for the same instrument if multiple actions occurred).
- The composite PK (RecoveryID, HedgeServerID, InstrumentID, State) allows multiple state entries per instrument within a session (e.g., both Detected=4 and Updated=2 for the same instrument if initial detection is followed by a correction).

### 2.2 Before/After Position Snapshots

**What**: Null-safe Prev/New parameter pairs record what the position was before the reconciliation and what it became after.

**Columns/Parameters Involved**: `@PrevUnits`, `@PrevIsBuy`, `@PrevRate`, `@NewUnits`, `@NewIsBuy`, `@NewRate`

**Rules**:
- All six parameters default to NULL - only populated when relevant to the reconciliation action:
  - State=1 (Added): `@Prev*` = NULL (no prior internal position), `@New*` = LP position adopted.
  - State=2 (Updated): `@Prev*` = internal position before correction, `@New*` = LP/corrected position.
  - State=3 (Removed): `@Prev*` = the position that was removed, `@New*` = NULL (no remaining position).
  - State=4 (Detected): `@Prev*` and `@New*` may both contain matching values (position verified as correct).
- `@PrevRate` / `@NewRate` use `dbo.dtPrice` (high-precision decimal) for rate tracking.

**Diagram**:
```
Hedge Server recovery session (RecoveryID = '{GUID-X}')
  |
  | For each instrument reconciled:
  |   EXEC Hedge.LogRecovery(@RecoveryID='{GUID-X}', @InstrumentID=5, @State=2,
  |                          @PrevUnits=100, @PrevIsBuy=1, @PrevRate=1.08500,
  |                          @NewUnits=120, @NewIsBuy=1, @NewRate=1.08520)
  |   -- State=2 (Updated): LP had 120 units, internal had 100 - corrected to 120
  |
  | BEGIN TRY
  |   INSERT INTO Hedge.RecoveryLog (RecoveryID, Time, HedgeServerID, ...)
  | END TRY / BEGIN CATCH -> THROW
  v
Hedge.RecoveryLog row (one per instrument reconciliation event)
```

## 8. Sample Queries

### 8.1 Log an "Updated" reconciliation event
```sql
EXEC [Hedge].[LogRecovery]
    @RecoveryID          = 'A4F9B3C1-1234-5678-ABCD-0123456789AB',
    @Time                = GETUTCDATE(),
    @HedgeServerID       = 1,
    @LiquidityAccountID  = 10,
    @State               = 2,              -- Updated
    @InstrumentID        = 5,
    @PrevUnits           = 100.00,
    @PrevIsBuy           = 1,
    @PrevRate            = 1.08500,
    @NewUnits            = 120.00,
    @NewIsBuy            = 1,
    @NewRate             = 1.08520
```

### 8.2 Review all events from a recovery session
```sql
SELECT RecoveryID, HedgeServerID, LiquidityAccountID,
       InstrumentID, State, Time,
       PrevUnits, PrevIsBuy, PrevRate,
       NewUnits, NewIsBuy, NewRate
FROM [Hedge].[RecoveryLog] WITH (NOLOCK)
WHERE RecoveryID = 'A4F9B3C1-1234-5678-ABCD-0123456789AB'
ORDER BY Time
```

### 8.3 Latest recovery sessions
```sql
SELECT RecoveryID, HedgeServerID, MIN(Time) AS SessionStart,
       MAX(Time) AS SessionEnd, COUNT(1) AS InstrumentsProcessed
FROM [Hedge].[RecoveryLog] WITH (NOLOCK)
GROUP BY RecoveryID, HedgeServerID
ORDER BY SessionStart DESC
```
