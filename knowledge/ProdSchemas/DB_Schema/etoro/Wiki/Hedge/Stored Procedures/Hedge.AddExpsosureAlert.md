## 1. Business Meaning

`Hedge.AddExpsosureAlert` is the standard (local) writer for the `Hedge.ExposureAlerts` table. It records a hedge execution alert event and returns the new row's identity value via the `@AlertID` OUTPUT parameter so the caller can reference the alert (e.g., to log a child record or send a notification).

Per the `Hedge.ExposureAlerts` table documentation, this table is a legacy archive from 2014. The three procedure variants (`AddExpsosureAlert`, `AddExpsosureAlert_Child`, `AddExpsosureAlert_old`) represent the evolution of the multi-server architecture:
- **AddExpsosureAlert** (this procedure): Writes to the local database `Hedge.ExposureAlerts`
- **AddExpsosureAlert_Child**: Writes to `etoro.Hedge.ExposureAlerts` (same server, different database context)
- **AddExpsosureAlert_old**: Writes to `[AO-REAL-DB].etoro.Hedge.ExposureAlerts` (remote linked server)

Note: The procedure name contains a typo (`Expsosure` instead of `Exposure`) - this is the original name and has not been corrected to preserve backward compatibility.

Data flow: These procedures were called when the hedge engine detected a failed TP/SL hedge close (AlertTypeID=1) or a fractional lot execution discrepancy (AlertTypeID=2/3). The table has not been written to since 2014 per the data evidence.

## 2. Business Logic

### 2.1 Identity Return via OUTPUT Parameter

**What**: The newly generated AlertID is returned to the caller via scope_identity().

**Columns/Parameters Involved**: `@AlertID OUTPUT`

**Rules**:
- `@AlertID` is an OUTPUT parameter - caller must declare a variable to receive it
- `SET @AlertID = SCOPE_IDENTITY()` captures the IDENTITY value of the just-inserted row
- The caller uses this ID to link related records (e.g., child alerts via AddExpsosureAlert_Child)

### 2.2 Three-Variant Architecture (Local/Cross-DB/Linked-Server)

**What**: The same alert data is written to three different database targets depending on the server context.

**Columns/Parameters Involved**: All parameters identical across three procedures

**Rules**:
- AddExpsosureAlert: `INSERT INTO Hedge.ExposureAlerts` (local, 2-part name)
- AddExpsosureAlert_Child: `INSERT INTO etoro.Hedge.ExposureAlerts` (3-part name, cross-db same server)
- AddExpsosureAlert_old: `INSERT INTO [AO-REAL-DB].etoro.Hedge.ExposureAlerts` (4-part name, linked server)
- All three have identical parameter signatures and business logic

## 8. Sample Queries

### 8.1 Execute: Insert an exposure alert and capture the new ID

```sql
DECLARE @NewAlertID INT
EXEC Hedge.AddExpsosureAlert
    @AlertID         = @NewAlertID OUTPUT,
    @NotificationTime = GETUTCDATE(),
    @AlertTypeID     = 1,
    @HedgeServerID   = 1,
    @InstrumentID    = 5,
    @UnitAmount      = 0.75,
    @IsBuy           = 1,
    @Description     = 'Failed TP hedge close attempt - no LP response'
SELECT @NewAlertID AS NewAlertID
```
