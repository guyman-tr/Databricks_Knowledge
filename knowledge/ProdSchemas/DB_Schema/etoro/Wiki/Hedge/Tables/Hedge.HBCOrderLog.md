## 1. Business Meaning

Hedge.HBCOrderLog is the child table of Hedge.HBCExecutionLog, storing the individual FIX orders placed as part of each HBC execution attempt. While HBCExecutionLog records the overall execution outcome (one row per attempt), HBCOrderLog records each FIX order within that attempt (potentially multiple orders per execution due to partial fills or order splits).

**Key cross-reference** (from HedgeServer Overview Confluence): `HBCOrderLog.HedgeID = ExecutionLog.OrderID`. This is the link between the HBC order flow and the main execution log - every HBC order that generates an execution response is traceable in both tables.

**When orders appear only in HBCExecutionLog (not HBCOrderLog)**: If execution fails before an order is created (e.g., validation failure, rate check failure), only a row in HBCExecutionLog is written (IsSuccess=0). HBCOrderLog only receives rows when an actual FIX order was submitted.

**Multiple orders per execution**: 53,746 HBCOrderLog rows cover 51,745 distinct ExecutionIDs (vs. 51,747 rows in HBCExecutionLog). ~2,001 executions have 2+ orders - this occurs when IsHBCFillOrKill=false and a partial fill generates additional order state events.

**OrderID is a GUID** (uniqueidentifier): Generated by the HedgeServer application, providing globally unique order identity across servers and databases.

**OrderState distribution** (53,746 rows):
- 2=Filled: 49,041 (91%) - successful executions
- 3=Rejected: 2,649 (5%) - LP rejected the order
- 5=UnRecoverable: 1,582 (3%) - critical failures requiring manual intervention
- 4=Cancelled: 460 (1%) - orders cancelled before execution
- 0=New: 14 (<1%) - likely in-flight or unprocessed

Note: No OrderState=1 (Pending) rows - orders move through Pending very quickly and are logged at terminal state.

## 2. Business Logic

### 2.1 Relationship to HBCExecutionLog

**What**: HBCOrderLog is the order-level detail for HBCExecutionLog's execution-level summary.

**Columns/Parameters Involved**: `ExecutionID`, `OrderID`, `HedgeID`

**Rules**:
- `ExecutionID` -> FK to Hedge.HBCExecutionLog.ExecutionID. Groups orders by execution attempt.
- `OrderID` (GUID) -> The unique FIX order identifier generated by the HedgeServer application. Primary key.
- `HedgeID` (int) -> The ID used in Hedge.ExecutionLog. Join: `HBCOrderLog.HedgeID = ExecutionLog.OrderID`.
- If LogHBCExecution's @IsSuccess=false and no orders were placed, only HBCExecutionLog gets a row.

### 2.2 HBC Order State Machine

**What**: OrderState tracks the FIX order lifecycle from creation to terminal state.

**Columns/Parameters Involved**: `OrderState`, `FailReason`

**Values** (FK to Dictionary.HBCOrderState):
| OrderState | Name | Count | Meaning |
|-----------|------|-------|---------|
| 0 | New | 14 | Order just created - transient, may be in-flight |
| 1 | Pending | 0 | Order submitted, awaiting fill - not seen in terminal logs |
| 2 | Filled | 49,041 | Successfully executed - normal outcome |
| 3 | Rejected | 2,649 | LP rejected the order (price, liquidity) |
| 4 | Cancelled | 460 | Order cancelled before execution |
| 5 | UnRecoverable | 1,582 | Critical failure requiring manual intervention |

- `FailReason`: Populated for non-successful states (Rejected, Cancelled, UnRecoverable). Nullable.

### 2.3 Lot Amounts (Same as Parent)

**What**: RequestAmountInLots and ExecutionAmountInLots mirror the parent execution's values at the order level.

**Columns/Parameters Involved**: `RequestAmountInLots`, `ExecutionAmountInLots`, `ExecutionRate`

**Rules**:
- `RequestAmountInLots`: The pre-rounding lot amount for this specific order.
- `ExecutionAmountInLots`: The actual lots executed (whole number after rounding). For a successful fill: matches the provider's fill amount.
- `ExecutionRate`: The dbo.dtPrice execution rate for this order.
- For multi-order executions, the sum of child ExecutionAmountInLots should equal the parent HBCExecutionLog.ExecutionAmountInLots.

### 2.4 IsCancelOrder

**What**: Distinguishes between regular orders and cancellation orders within an execution.

**Columns/Parameters Involved**: `IsCancelOrder`, `IsBuy`

**Rules**:
- `IsCancelOrder=1`: This order is a cancellation of a previous order within the execution.
- Corresponds to `IsCancelExecution=1` in the parent HBCExecutionLog.
- `IsBuy`: Matches the parent execution direction.

## 8. Sample Queries

### 8.1 Orders for a specific execution
```sql
SELECT OrderID, HedgeID, IsBuy, OrderState, RequestAmountInLots,
       ExecutionAmountInLots, ExecutionRate, StartTime, EndTime, FailReason
FROM Hedge.HBCOrderLog WITH (NOLOCK)
WHERE ExecutionID = 13797342
ORDER BY StartTime;
```

### 8.2 Cross-reference to ExecutionLog
```sql
SELECT ol.OrderID, ol.ExecutionID, ol.HedgeID, ol.OrderState,
       ol.ExecutionAmountInLots, ol.ExecutionRate,
       el.OrderState AS ExecutionLogOrderState, el.LogTime
FROM Hedge.HBCOrderLog ol WITH (NOLOCK)
JOIN Hedge.ExecutionLog el WITH (NOLOCK) ON el.OrderID = ol.HedgeID
WHERE ol.ExecutionID = 13797342;
```

### 8.3 Order state distribution by instrument
```sql
SELECT el.InstrumentID, ol.OrderState,
       COUNT(1) AS OrderCount
FROM Hedge.HBCOrderLog ol WITH (NOLOCK)
JOIN Hedge.HBCExecutionLog el WITH (NOLOCK) ON el.ExecutionID = ol.ExecutionID
WHERE ol.StartTime > DATEADD(day, -7, GETUTCDATE())
GROUP BY el.InstrumentID, ol.OrderState
ORDER BY el.InstrumentID, ol.OrderState;
```
