## 1. Business Meaning

Hedge.RecoveryLog records what happens when a hedge server performs a recovery/reconciliation with a liquidity provider. During a recovery session, the hedge server compares its internal position state with the LP's actual positions and resolves discrepancies. Each action taken for each instrument is logged here.

A "recovery" in the hedge context refers to the process by which a hedge server:
1. Reconnects to an LP after an outage or restart
2. Queries the LP for its current positions
3. Compares LP positions with internal records
4. Identifies discrepancies (extra positions, missing positions, incorrect sizes)
5. Logs each finding/action with before/after state for audit purposes

Each recovery session is identified by a GUID (`RecoveryID`) shared across all log entries from that session. The `State` column (from Dictionary.HedgeRecoveryState) classifies the reconciliation action: Added (new position at LP that eToro didn't expect), Updated (position size or direction changed), Removed (position at LP that no longer exists internally), or Detected (position found/verified as expected).

The `Prev*` and `New*` column pairs capture the before/after position snapshot, enabling post-incident analysis of what changed during recovery and whether the reconciliation was correct.

The table is currently empty (0 rows) - recovery events are infrequent (only occur during server restart or LP reconnection scenarios) and older entries may be purged.

## 2. Business Logic

### 2.1 Recovery Session Structure (RecoveryID Groups All Events)

**What**: All log entries from a single recovery session share the same RecoveryID GUID, forming a recoverable audit group.

**Columns/Parameters Involved**: `RecoveryID`, `Time`, `HedgeServerID`, `LiquidityAccountID`, `State`, `InstrumentID`

**Rules**:
- RecoveryID is a GUID generated by the hedge server application (not IDENTITY/DB-generated)
- All instruments processed in one recovery run share the same RecoveryID
- One recovery run can produce multiple rows for the same instrument if multiple state transitions are logged (e.g., Detected=4 + Updated=2 for the same instrument)
- This is why State is included in the PK - to allow multiple state entries per (RecoveryID, HedgeServerID, InstrumentID)
- Time is per-row (not session-level) - captures the exact moment each reconciliation action occurred

**Diagram**:
```
Recovery session starts on HedgeServerID=3 -> RecoveryID = '{GUID-X}'
  InstrumentID=1: Position at LP matches, log State=4 (Detected)
  InstrumentID=5: Position at LP is larger, log State=2 (Updated), Prev=X, New=Y
  InstrumentID=9: Position at LP is gone, log State=3 (Removed), Prev=Z, New=NULL
  InstrumentID=17: New position at LP not in internal state, log State=1 (Added), Prev=NULL, New=W

All 4 rows share RecoveryID='{GUID-X}' but different InstrumentID/State combinations.
```

### 2.2 Before/After Position Snapshot

**What**: The `Prev*` and `New*` column pairs record the position state on both sides of the reconciliation action.

**Columns/Parameters Involved**: `PrevUnits`, `PrevIsBuy`, `PrevRate`, `NewUnits`, `NewIsBuy`, `NewRate`

**Rules**:
- All six are nullable - not all actions have both a before and after state
- State=1 (Added): Prev* = NULL (no prior internal position), New* = the LP position that was reconciled/added
- State=2 (Updated): Prev* = internal position before update, New* = LP/corrected position after update
- State=3 (Removed): Prev* = the position that was removed, New* = NULL (no remaining position)
- State=4 (Detected): Prev* and New* may both contain the same values (position verified as matching)
- State=0 (None): Likely a default/initialization entry with NULL values

### 2.3 State Values (Dictionary.HedgeRecoveryState)

| ID | Name | Meaning in Context |
|---|---|---|
| 0 | None | Default/unset state. Not typically used in active log entries. |
| 1 | Added | A position exists at the LP that was NOT in the hedge server's internal state. The LP position was adopted/added. |
| 2 | Updated | The hedge server's position was corrected to match the LP's actual position (different units, rate, or direction). |
| 3 | Removed | A position was in the hedge server's state but NOT at the LP. The internal record was marked for removal. |
| 4 | Detected | A position was found at the LP and verified as matching the internal state. No correction needed. |

## 8. Sample Queries

### 8.1 All events in a recovery session (when populated)
```sql
SELECT  rl.Time,
        rl.InstrumentID,
        hrs.Name AS StateName,
        rl.PrevUnits, rl.PrevIsBuy, rl.PrevRate,
        rl.NewUnits,  rl.NewIsBuy,  rl.NewRate
FROM    [Hedge].[RecoveryLog] rl WITH (NOLOCK)
INNER JOIN [Dictionary].[HedgeRecoveryState] hrs WITH (NOLOCK)
        ON rl.State = hrs.ID
WHERE   rl.RecoveryID = '{recovery-guid-here}'
ORDER BY rl.Time, rl.InstrumentID;
```

### 8.2 Recent recovery sessions per hedge server
```sql
SELECT  rl.RecoveryID,
        rl.HedgeServerID,
        rl.LiquidityAccountID,
        MIN(rl.Time)  AS SessionStart,
        MAX(rl.Time)  AS SessionEnd,
        COUNT(*)      AS EventCount,
        SUM(CASE WHEN rl.State = 1 THEN 1 ELSE 0 END) AS Added,
        SUM(CASE WHEN rl.State = 2 THEN 1 ELSE 0 END) AS Updated,
        SUM(CASE WHEN rl.State = 3 THEN 1 ELSE 0 END) AS Removed,
        SUM(CASE WHEN rl.State = 4 THEN 1 ELSE 0 END) AS Detected
FROM    [Hedge].[RecoveryLog] rl WITH (NOLOCK)
GROUP BY rl.RecoveryID, rl.HedgeServerID, rl.LiquidityAccountID
ORDER BY MAX(rl.Time) DESC;
```

### 8.3 Find corrections (Updated/Added/Removed) from most recent session
```sql
SELECT  rl.InstrumentID,
        hrs.Name AS Action,
        rl.PrevUnits, rl.PrevIsBuy, rl.PrevRate,
        rl.NewUnits,  rl.NewIsBuy,  rl.NewRate
FROM    [Hedge].[RecoveryLog] rl WITH (NOLOCK)
INNER JOIN [Dictionary].[HedgeRecoveryState] hrs WITH (NOLOCK)
        ON rl.State = hrs.ID
WHERE   rl.RecoveryID = (
            SELECT TOP 1 RecoveryID FROM [Hedge].[RecoveryLog] WITH (NOLOCK)
            ORDER BY Time DESC
        )
AND     rl.State IN (1, 2, 3)
ORDER BY rl.InstrumentID, rl.State;
```
