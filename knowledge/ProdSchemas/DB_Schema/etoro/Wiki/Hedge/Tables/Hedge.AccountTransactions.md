## 1. Business Meaning

Hedge.AccountTransactions records every monetary transaction that occurs within a hedge server's liquidity provider accounts - deposits, withdrawals, commissions, overnight fees, rebates, and other cash movements. Each row represents a single transaction event as reported by the liquidity provider: which hedge server and account it happened on, what type of transaction it was, the amount, whether it was entered manually or generated automatically, the provider's timestamp, and an optional free-text comment. For instrument-specific transactions (such as overnight fees or conversion costs), the InstrumentID column links the transaction to a specific traded instrument.

This table exists so the hedge monitoring system can track the full cash flow lifecycle of each liquidity account - identifying when deposits are made, fees are charged, or rebates are received. Without it, the hedge cost reporting system could not calculate net account profitability or track commission income vs. fee outflows per server and account.

Data flows as follows: `Hedge.AddAccountTransaction` inserts rows (note: the procedure inserts into the alias `HedgeAccountTransactions`, which resolves to this table). `Hedge.GetAccountTransactions` returns all transactions for a specific account. `Hedge.GetAccountTransactionsData` aggregates transactions since a reference date per (HedgeServerID, LiquidityAccountID, TransactionTypeID) - same pattern used for real-time monitoring. `Hedge.GetAggregatedAccountTransactions` sums deposits (TypeID=1) and withdrawals (TypeID=2) separately for a given account. `Hedge.GetAggregatedAccountTransactionsByType` provides a flexible multi-filter aggregation using TVP parameters (Hedge.IDs type) for server IDs, account IDs, transaction types, and start date.

## 2. Business Logic

### 2.1 Transaction Type Classification

**What**: Every transaction is classified by type, enabling separate aggregation of deposits vs. withdrawals vs. fee categories.

**Columns/Parameters Involved**: `TransactionTypeID`, `Amount`

**Rules**:
- TransactionTypeID = 1 (Deposit): Cash added to the liquidity account. Summed as TotalDepositsAmount in GetAggregatedAccountTransactions.
- TransactionTypeID = 2 (Withdrawal): Cash removed from the account. Summed as TotalWithdrawsAmount separately.
- TransactionTypeID = 3-13: Fee/cost/revenue categories recorded as separate transaction rows.
- Amount represents the cash flow amount in USD (decimal(14,4)). Sign convention not enforced by DDL - callers determine positive/negative based on type.
- GetAccountTransactionsData groups by (HedgeServerID, LiquidityAccountID, TransactionTypeID) with SUM(Amount) - enables pivot-style reporting by type.

**Full Transaction Type Map** (Dictionary.AccountTransactionType):

| TransactionTypeID | TransactionTypeName |
|---|---|
| 1 | Deposit |
| 2 | Withdrawal |
| 3 | Refund |
| 4 | Compensation |
| 5 | Commission |
| 6 | Adjustment |
| 7 | Interest |
| 8 | Transaction Fees |
| 9 | Overnight Fees |
| 10 | Conversion |
| 11 | Rebate |
| 12 | Manual Cost |
| 13 | System Cost |

### 2.2 Manual vs Automated Transactions

**What**: Transactions can be system-generated (automated) or manually entered by operations staff.

**Columns/Parameters Involved**: `IsManual`, `Comment`

**Rules**:
- IsManual = 0 (DEFAULT): Automated transaction generated by the hedge system (e.g., fee charged by provider, commission processed).
- IsManual = 1: Manually entered transaction, typically entered by operations staff for corrections, compensations, or adjustments.
- Comment (DEFAULT = '', max 500 chars) captures the reason or description, especially useful for manual transactions where the human-entered note explains the business reason.

### 2.3 PK Design - Per Timestamp/Server/Type

**What**: The primary key constrains uniqueness to one record per provider timestamp, server, and transaction type - not including LiquidityAccountID.

**Columns/Parameters Involved**: `OccurredAtAccount`, `HedgeServerID`, `TransactionTypeID`

**Rules**:
- PK: (OccurredAtAccount, HedgeServerID, TransactionTypeID) - only ONE row per server/type per provider timestamp, regardless of liquidity account
- This means if multiple accounts on the same server received the same transaction type at the exact same provider timestamp, they would conflict
- LiquidityAccountID is required (NOT NULL) but not in the PK - the table assumes each server/timestamp/type combination maps to one account
- OccurredAtAccount is the PROVIDER's timestamp (not DB server time) - this is the sole timestamp column, unlike AccountStatus which also has a DB-side OccurredAt

## 8. Sample Queries

### 8.1 Get all transactions for a specific liquidity account
```sql
SELECT HedgeServerID, LiquidityAccountID, TransactionTypeID,
       Amount, IsManual, OccurredAtAccount, Comment, InstrumentID
FROM Hedge.AccountTransactions WITH (NOLOCK)
WHERE LiquidityAccountID = 8
ORDER BY OccurredAtAccount DESC;
```

### 8.2 Aggregate deposits and withdrawals by account (mirrors GetAggregatedAccountTransactions)
```sql
SELECT LiquidityAccountID, HedgeServerID,
       SUM(CASE WHEN TransactionTypeID = 1 THEN Amount ELSE 0 END) AS TotalDeposits,
       SUM(CASE WHEN TransactionTypeID = 2 THEN Amount ELSE 0 END) AS TotalWithdrawals,
       SUM(CASE WHEN TransactionTypeID NOT IN (1,2) THEN Amount ELSE 0 END) AS TotalFees
FROM Hedge.AccountTransactions WITH (NOLOCK)
GROUP BY LiquidityAccountID, HedgeServerID
ORDER BY LiquidityAccountID;
```

### 8.3 Recent transactions joined to type name and account name
```sql
SELECT AT.OccurredAtAccount,
       HS.IPAddress + ':' + CAST(HS.Port AS varchar) AS HedgeServer,
       LA.LiquidityAccountName,
       ATT.TransactionTypeName,
       AT.Amount,
       AT.IsManual,
       AT.Comment,
       AT.InstrumentID
FROM Hedge.AccountTransactions AT WITH (NOLOCK)
JOIN Trade.HedgeServer HS WITH (NOLOCK)
  ON AT.HedgeServerID = HS.HedgeServerID
JOIN Trade.LiquidityAccounts LA WITH (NOLOCK)
  ON AT.LiquidityAccountID = LA.LiquidityAccountID
JOIN Dictionary.AccountTransactionType ATT WITH (NOLOCK)
  ON AT.TransactionTypeID = ATT.TransactionTypeID
WHERE AT.OccurredAtAccount > DATEADD(day, -7, GETUTCDATE())
ORDER BY AT.OccurredAtAccount DESC;
```
