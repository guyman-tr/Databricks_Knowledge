## 1. Business Meaning

This table is the permanent historical record of forex game sessions in eToro's early gamified trading platform. Each row represents a single game session: a customer participated in a game of a specific type, took a directional position on a currency pair, and the game ran from StartDateTime to EndDateTime. The table captures the game's identity (ForexGameID), the game type/mode (GameTypeID), the customer (CID), the primary currency direction bet (PrimaryCurrencyDirection), and the game's lifecycle window.

Without this table, it would be impossible to reconstruct the historical game activity of customers after the live `Game.ForexResult` records are deleted at game close. The table serves as the long-term archive that maintains referential integrity for positions and trades linked to ForexResultIDs after those games have completed and been removed from the active `Game.ForexResult` table.

Data enters this table at game creation via `Game.GameOpen`, which inserts a row with EndDateTime=NULL simultaneously with the live `Game.ForexResult` row. When a game closes via `Game.GameClose`, EndDateTime is stamped with the closure timestamp and the live `Game.ForexResult` row is deleted - leaving `History.ForexResult` as the sole permanent record. This pattern ensures referential integrity for any positions or orders linked to a ForexResultID even after the game's live record is removed. ForexResultID is generated by `Internal.GetActionID`, a shared global action ID sequence (not an IDENTITY column), so ForexResultIDs are unique across all action types.

**Note**: Based on live data analysis, this table contains only 1,285 rows spanning 2009-09-16 to 2014-11-23. It is a legacy artifact from eToro's gaming platform era. GameTypeID=34 (eToro Trading) accounts for 94% of rows. Modern trading does not use this game framework.

## 2. Business Logic

### 2.1 Dual-Write Active/History Pattern

**What**: At game open, a row is inserted into both `Game.ForexResult` (the active game table) and `History.ForexResult` (this table) simultaneously, ensuring the history record exists throughout the game's lifetime.

**Columns/Parameters Involved**: `ForexResultID`, `StartDateTime`, `EndDateTime`

**Rules**:
- `Game.GameOpen` inserts into BOTH `Game.ForexResult` AND `History.ForexResult` in the same transaction; the History row is inserted with `EndDateTime=NULL`
- Comment in `Game.GameOpen` explains: "This will help to keep referential integrity when we will move links between open positions (that become closed) and the game"
- `Game.GameClose` then UPDATEs `History.ForexResult.EndDateTime = GETDATE()` and DELETEs from `Game.ForexResult` in the same transaction
- After game close: only `History.ForexResult` retains the game record; `Game.ForexResult` no longer has the row
- A row with `EndDateTime IS NULL` represents either a game still in progress OR a game that was never properly closed
- Live data: only 1 of 1,285 rows has NULL EndDateTime, confirming nearly all games are properly closed

**Diagram**:
```
Game.GameOpen called
         |
         v
INSERT INTO Game.ForexResult (EndDateTime not written)   <- active game row
INSERT INTO History.ForexResult (EndDateTime = NULL)     <- permanent archive, game in progress

Game.GameClose called
         |
         v
UPDATE History.ForexResult SET EndDateTime = GETDATE()   <- stamp closure time
DELETE FROM Game.ForexResult                              <- remove active row
         |
         v
Only History.ForexResult remains (EndDateTime IS NOT NULL)
```

### 2.2 Game Type Classification

**What**: GameTypeID classifies which game mode/product the session used, ranging from early gamified formats to the primary eToro Trading platform.

**Columns/Parameters Involved**: `GameTypeID`, `ForexGameID`

**Rules**:
- GameTypeID 34 (eToro Trading) is dominant with 1,207 of 1,285 rows (94%) - the main eToro trading game
- GameTypeID 52 (Forex Charts) with 20 rows, 31 (Globe Trader) with 19 rows, 33 (Trade Box) with 18 rows represent earlier/alternative game formats
- GameTypeID 3 (Forex Marathon) and 4 (Dollar Trend) with 12 and 9 rows represent the oldest game formats (earliest dates back to 2009)
- ForexGameID links to `Game.ForexGame`, which defines the game's parameters (currency set, leverage, stop/loss ranges, game bet amount)
- ForexResultID values in the 200M-250M range are globally unique action IDs generated by `Internal.GetActionID`, not sequential game counters

## 8. Sample Queries

### 8.1 Get all game sessions for a customer

```sql
SELECT
    h.ForexResultID,
    gt.Name AS GameType,
    h.ForexGameID,
    h.PrimaryCurrencyDirection,
    h.StartDateTime,
    h.EndDateTime,
    DATEDIFF(SECOND, h.StartDateTime, h.EndDateTime) AS DurationSeconds
FROM History.ForexResult h WITH (NOLOCK)
JOIN Dictionary.GameType gt WITH (NOLOCK) ON h.GameTypeID = gt.GameTypeID
WHERE h.CID = @CID
ORDER BY h.StartDateTime DESC;
```

### 8.2 Find all in-progress games (EndDateTime IS NULL)

```sql
SELECT
    h.ForexResultID,
    h.CID,
    gt.Name AS GameType,
    h.StartDateTime,
    DATEDIFF(MINUTE, h.StartDateTime, GETDATE()) AS MinutesElapsed
FROM History.ForexResult h WITH (NOLOCK)
JOIN Dictionary.GameType gt WITH (NOLOCK) ON h.GameTypeID = gt.GameTypeID
WHERE h.EndDateTime IS NULL
ORDER BY h.StartDateTime ASC;
```

### 8.3 Game type breakdown for closed sessions

```sql
SELECT
    gt.Name AS GameType,
    COUNT(*) AS SessionCount,
    AVG(DATEDIFF(SECOND, h.StartDateTime, h.EndDateTime)) AS AvgDurationSeconds,
    MIN(h.StartDateTime) AS FirstSession,
    MAX(h.EndDateTime) AS LastSession
FROM History.ForexResult h WITH (NOLOCK)
JOIN Dictionary.GameType gt WITH (NOLOCK) ON h.GameTypeID = gt.GameTypeID
WHERE h.EndDateTime IS NOT NULL
GROUP BY gt.GameTypeID, gt.Name
ORDER BY SessionCount DESC;
```
