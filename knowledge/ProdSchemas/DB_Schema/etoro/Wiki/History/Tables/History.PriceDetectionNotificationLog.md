## 1. Business Meaning

`History.PriceDetectionNotificationLog` stores the alert message content (subject line and body) for each price feed anomaly event detected by the price monitoring system. It is the exact 1:1 companion to `History.PriceDetectionDifferenceLog` - every row in DifferenceLog has a corresponding row here with the same `NotificationLogID`, providing the human-readable context for what that anomaly event communicated to operations.

The body of each notification contains structured feed comparison data: the instrument ID, timestamp, active feed provider and its price, and secondary feed data. Subject lines follow the pattern `"{InstrumentID} : {High|Low} Price Problem Alert for server {ServerName}"`, where "Low" indicates the active feed's price is below secondary feeds and "High" indicates it is above. This directional description (Low/High price) refers to the active provider's price relative to secondary feeds - not to be confused with the `NotificationSeverityTypeID` which describes how severe the discrepancy is.

With 208,489 rows, this table holds an eight-year archive of price monitoring alerts. The external price detection service (not in SSDT) writes both this table and DifferenceLog together for each detected anomaly.

## 2. Business Logic

### 2.1 1:1 Pairing with PriceDetectionDifferenceLog

**What**: Each row here directly corresponds to one row in PriceDetectionDifferenceLog, sharing the same NotificationLogID.

**Columns/Parameters Involved**: `NotificationLogID`, `NotificationSeverityTypeID`

**Rules**:
- `NotificationLogID` in this table is NOT an IDENTITY - it receives the value generated by PriceDetectionDifferenceLog's IDENTITY(1,1)
- Both tables are inserted in the same operation by the price detection service
- NotificationSeverityTypeID is duplicated here (same FK to Dictionary.DowntimeSeverity) to allow querying this table independently without joining DifferenceLog
- To get the full event record: `JOIN History.PriceDetectionDifferenceLog ON NotificationLogID`

### 2.2 Alert Subject Format

**What**: Subject lines encode instrument, anomaly direction, and server in a standard format.

**Columns/Parameters Involved**: `Subject`

**Rules**:
- Format: `"{InstrumentID} : {Low|High} Price Problem Alert for server {ServerName}"`
- "Low Price Problem Alert" = active provider's price is LOWER than secondary feeds
- "High Price Problem Alert" = active provider's price is HIGHER than secondary feeds
- Server name (e.g., "OMPD") identifies which trading server processed the detection
- From live data: NotificationSeverityTypeID=1 (Critical) correlates with "Low Price Problem Alert"; NotificationSeverityTypeID=3 (Medium) with "High Price Problem Alert"

## 8. Sample Queries

### 8.1 Full notification details for a specific event

```sql
SELECT
    nl.NotificationLogID,
    ds.Name AS Severity,
    nl.Subject,
    nl.Body,
    nl.Occurred
FROM History.PriceDetectionNotificationLog nl WITH (NOLOCK)
JOIN Dictionary.DowntimeSeverity ds WITH (NOLOCK) ON ds.DowntimeSeverityID = nl.NotificationSeverityTypeID
WHERE nl.NotificationLogID = @NotificationLogID
```

### 8.2 Combined technical + notification view for recent anomalies

```sql
SELECT
    ddl.NotificationLogID,
    ds.Name AS Severity,
    ddl.InstrumentID,
    ddl.ActiveProviderID,
    ddl.ActiveProviderPrice,
    nl.Subject,
    ddl.Occurred
FROM History.PriceDetectionDifferenceLog ddl WITH (NOLOCK)
JOIN History.PriceDetectionNotificationLog nl WITH (NOLOCK) ON nl.NotificationLogID = ddl.NotificationLogID
JOIN Dictionary.DowntimeSeverity ds WITH (NOLOCK) ON ds.DowntimeSeverityID = ddl.NotificationSeverityTypeID
WHERE ddl.Occurred >= DATEADD(HOUR, -1, GETDATE())
ORDER BY ddl.Occurred DESC
```

### 8.3 Search notifications by subject keyword

```sql
SELECT
    NotificationLogID,
    NotificationSeverityTypeID,
    Subject,
    Occurred
FROM History.PriceDetectionNotificationLog WITH (NOLOCK)
WHERE Subject LIKE '%' + @InstrumentID + '%'
  AND Occurred >= DATEADD(DAY, -7, GETDATE())
ORDER BY Occurred DESC
```
