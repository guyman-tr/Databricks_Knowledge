## 1. Business Meaning

History.WithdrawToFundingAction_InsertedChangedFunding is a narrow companion table to History.WithdrawToFundingAction that captures a specific type of event: when a payment order's FundingID (the customer's payment method) was changed mid-process.

The normal withdrawal payment flow pairs a withdrawal with a specific FundingID (bank account, card, etc.) via Billing.WithdrawToFunding. Occasionally an operator or automated process redirects a payment to a different funding instrument - this table records the BW2F record affected, the History.WithdrawToFundingAction row(s) generated by that change, and both the old and new FundingID values.

With only 1,462 rows, FundingID changes are rare (occurring in fewer than 0.1% of payment orders). The table has no indexes, no constraints, and is referenced only by its own DDL - suggesting it is written directly by a procedure or trigger that is not yet present in the SSDT repo, or by a now-retired workflow.

## 2. Business Logic

### 2.1 FundingID Change Tracking

**What**: When a Billing.WithdrawToFunding record's FundingID is updated (switching which payment method processes the withdrawal), one or more rows are inserted here capturing the before/after state.

**Columns/Parameters Involved**: `BW2F_ID`, `WithdrawToFundingActionID`, `FundingID_AfterUpdate`, `FundingID_BeforeUpdate`

**Rules**:
- Multiple rows per BW2F_ID are possible - each corresponding to a different History.WithdrawToFundingAction row generated during the funding switch
- `FundingID_BeforeUpdate` = the original customer payment method that was replaced
- `FundingID_AfterUpdate` = the new payment method that was substituted
- `WithdrawToFundingActionID` links back to the specific action row in History.WithdrawToFundingAction that captured this change
- NULL values in `WithdrawToFundingActionID`, `FundingID_AfterUpdate`, `FundingID_BeforeUpdate` are allowed (nullable columns)

**Pattern observed in data**:
```
BW2F_ID=19475 has 3 rows:
  Row 1: WTFAID=58384, FundingID_Before=50651, FundingID_After=79830
  Row 2: WTFAID=58385, FundingID_Before=50651, FundingID_After=79830
  Row 3: WTFAID=58386, FundingID_Before=50651, FundingID_After=79830

Same funding switch (50651 -> 79830) associated with 3 consecutive action rows
- likely 3 status transitions captured during the redirect process
```

## 8. Sample Queries

### 8.1 Find all FundingID changes for a specific payment order
```sql
SELECT
    cf.BW2F_ID,
    cf.WithdrawToFundingActionID,
    cf.FundingID_BeforeUpdate,
    cf.FundingID_AfterUpdate
FROM History.WithdrawToFundingAction_InsertedChangedFunding cf WITH (NOLOCK)
WHERE cf.BW2F_ID = 19475
ORDER BY cf.WithdrawToFundingActionID;
```

### 8.2 Join to action table to get funding change context

    cf.FundingID_AfterUpdate,
