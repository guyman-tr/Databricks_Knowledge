## 1. Business Meaning

History.OrdersChangeLog records every operation applied to an order throughout its lifecycle. Each row captures a single operation event: what happened to the order (OperationTypeID), when it happened, which service triggered it, and the order's settlement/risk properties at that moment.

The table is written directly by application services - primarily ROS (the real-time Rate/Order Service) and PREEXE (pre-execution processing) - rather than stored procedures. This makes it an application-layer audit trail rather than a database-layer trigger log.

With 14,538 rows split nearly equally between OperationTypeID=1 (50.4%) and OperationTypeID=2 (49.6%), the table captures a two-event lifecycle per order: a creation/registration event (type 1) that records the order's settlement and risk properties, followed by a completion/cancellation event (type 2) where those properties are NULL (already captured on type 1).

The `ClientRequestGuid` links each operation back to the originating client request, supporting idempotency checks and duplicate request detection. `RequestingService` identifies which application component triggered the change.

## 2. Business Logic

### 2.1 Two-Event Order Lifecycle

**What**: Each order typically generates exactly two rows: one for creation/registration (OperationTypeID=1) and one for completion/cancellation (OperationTypeID=2).

**Columns/Parameters Involved**: `OrderID`, `OperationTypeID`, `IsSettled`, `SettlementTypeID`, `IsNoStopLoss`, `IsNoTakeProfit`

**Rules**:
- OperationTypeID=1 (creation): IsSettled, SettlementTypeID, IsNoStopLoss, IsNoTakeProfit are populated - capturing the order's properties at time of creation
- OperationTypeID=2 (completion/cancellation): these property columns are NULL - the state was already captured on type 1; type 2 records only that the event occurred
- Time gap between type 1 and type 2 rows for the same OrderID is typically seconds to minutes (observed: 19 seconds between events for the same order)
- Some orders may have only one row (type 1 without a subsequent type 2) if the completion event has not yet occurred

**Example from data**:
```
OrderID=24336164, OperationTypeID=1: IsSettled=1, SettlementTypeID=1, IsNoStopLoss=1, IsNoTakeProfit=0  <- order created
OrderID=24336164, OperationTypeID=2: IsSettled=NULL, SettlementTypeID=NULL, IsNoStopLoss=NULL, IsNoTakeProfit=NULL <- order completed (19s later)
```

### 2.2 Settlement and Risk Properties (Type 1 Rows)

**What**: OperationTypeID=1 rows capture the order's settlement and risk-protection configuration at creation time.

**Rules**:
- IsSettled: whether the order is treated as settled (1) or unsettled (0)
- SettlementTypeID: the settlement mechanism type (tinyint). Links implicitly to settlement type dictionary
- IsNoStopLoss: TRUE (1) = this order has no stop loss protection; FALSE (0) = stop loss is active
- IsNoTakeProfit: TRUE (1) = this order has no take profit target; FALSE (0) = take profit is active
- These properties are intentionally NULL on type 2 rows to avoid redundant storage

### 2.3 RequestingService - Application Source Identification

**What**: Identifies which application service triggered the order operation. Critical for root cause analysis when unexpected order events occur.

**Rules**:
- "ROS": Rate/Order Service - the primary real-time order processing service (82.1% of rows)
- "PREEXE": Pre-execution processing service (9.8% of rows)
- NULL: rows written before the RequestingService column was added, or from systems that do not set this field (8.1% of rows)
- varchar(10) constrains service name length - new services must use codes of 10 chars or fewer

### 2.4 ClientRequestGuid - Client Request Tracing

**What**: Links the order operation back to the originating client API request. Enables idempotency and duplicate detection.

**Rules**:
- Nullable: older rows and internal system operations may not have a client request context
- uniqueidentifier: GUIDs generated by the calling client/service for request deduplication
- Multiple rows can share the same ClientRequestGuid only if the same request triggered multiple order operations (unusual)
- Useful for tracing a specific client action through all the order events it triggered

## 8. Sample Queries

### 8.1 Full lifecycle for a specific order

```sql
SELECT ID, OrderID, OperationTypeID, ClientRequestGuid,
       Occurred, IsSettled, SettlementTypeID, IsNoStopLoss, IsNoTakeProfit,
       RequestingService
FROM [History].[OrdersChangeLog] WITH (NOLOCK)
WHERE OrderID = @OrderID
ORDER BY Occurred ASC
```

### 8.2 Orders with no completion event (type 1 without type 2)

```sql
SELECT ocl1.OrderID, ocl1.Occurred AS CreatedAt, ocl1.RequestingService
FROM [History].[OrdersChangeLog] ocl1 WITH (NOLOCK)
WHERE ocl1.OperationTypeID = 1
  AND NOT EXISTS (
    SELECT 1 FROM [History].[OrdersChangeLog] ocl2 WITH (NOLOCK)
    WHERE ocl2.OrderID = ocl1.OrderID AND ocl2.OperationTypeID = 2
  )
ORDER BY ocl1.Occurred DESC
```

### 8.3 Activity by requesting service (last 30 days)

```sql
SELECT RequestingService,
       OperationTypeID,
       COUNT(*) AS EventCount,
       MIN(Occurred) AS FirstEvent,
       MAX(Occurred) AS LastEvent
FROM [History].[OrdersChangeLog] WITH (NOLOCK)
WHERE Occurred >= DATEADD(DAY, -30, GETDATE())
GROUP BY RequestingService, OperationTypeID
ORDER BY EventCount DESC
```
