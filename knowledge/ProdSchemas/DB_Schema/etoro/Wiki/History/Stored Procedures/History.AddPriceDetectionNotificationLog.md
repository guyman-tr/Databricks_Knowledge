# History.AddPriceDetectionNotificationLog

> Writer procedure that logs the human-readable alert content (subject and body) for a price feed anomaly event into History.PriceDetectionNotificationLog, using the IDENTITY value generated by the sibling AddPriceDetectionDifferenceLog call as the shared log ID.

## 1. Business Meaning

`History.AddPriceDetectionNotificationLog` is the alert-content writer for the price anomaly detection system. It is always called as the second step after `History.AddPriceDetectionDifferenceLog`: that procedure generates an IDENTITY log ID and returns it; this procedure receives that ID as `@ID` and inserts the human-readable notification content (subject line and body) linked to the same event.

This procedure exists because price anomaly events have two distinct audiences: operations engineers need structured technical data (instrument, provider, price - stored in DifferenceLog), while alert recipients need human-readable messages (email subject/body - stored in NotificationLog). Separating these into two tables (and two writer procedures) allows each to evolve independently and be queried independently without needing a JOIN.

The external price detection service always calls these two procedures in sequence: AddPriceDetectionDifferenceLog first (to get the new log ID), then this procedure. The @ID parameter is the link that keeps the two sibling log tables joinable. If this procedure fails (returns -1), the NotificationLog row is missing but the DifferenceLog row still exists - the technical event is recorded even if the notification content is lost.

## 2. Business Logic

### 2.1 Linked Write Pattern (Second Step)

**What**: This procedure always follows AddPriceDetectionDifferenceLog in sequence; the @ID it receives IS the NotificationLogID generated by that first call.

**Columns/Parameters Involved**: `@ID`, `@NotificationSeverityTypeID`

**Rules**:
- @ID must be the value returned by the preceding AddPriceDetectionDifferenceLog call
- The INSERT uses @ID as the explicit NotificationLogID (History.PriceDetectionNotificationLog has no IDENTITY - it receives this value)
- If @ID = -1 (the error sentinel from the first call), the external service must NOT call this procedure
- RETURN(0) on success; RETURN(-1) on INSERT failure

**Diagram**:
```
Step 1: AddPriceDetectionDifferenceLog(...) -> returns logID (e.g., 1639035)
Step 2: AddPriceDetectionNotificationLog(@ID=1639035, @Subject="100024 : Low Price...", ...)
         -> INSERT History.PriceDetectionNotificationLog(NotificationLogID=1639035, ...)
         -> RETURN(0) on success
```

### 2.2 Dead Code Warning

**What**: The stored procedure body contains UPDATE statements on Trade.Instrument that can never execute.

**Columns/Parameters Involved**: N/A (dead code - not reachable)

**Rules**:
- After the END CATCH block, the file contains 7 UPDATE statements setting Trade.Instrument.PipDifferenceThreshold for various InstrumentID ranges
- These UPDATEs are syntactically part of the procedure body (they appear before the GO batch separator)
- They are UNREACHABLE: the TRY path executes RETURN(0) and the CATCH path executes RETURN(-1) - both exit the procedure before reaching the UPDATEs
- These appear to be leftover one-time migration/setup SQL accidentally embedded in the procedure definition during a historical deployment
- These UPDATE statements NEVER execute when the procedure is called

**Diagram**:

    |   END TRY

**Note**: The procedure file contains dead code after END CATCH: 7 UPDATE statements on Trade.Instrument (setting PipDifferenceThreshold for InstrumentID ranges 1-7, 8-16, 17-18, 19, 27-29, 30-32, and 0). ## 8. Sample Queries

### 8.1 Find recent notification alerts written by this procedure

```sql
SELECT TOP 10
    NotificationLogID,
    NotificationSeverityTypeID,
    Subject,
    Occurred
FROM History.PriceDetectionNotificationLog WITH (NOLOCK)
ORDER BY NotificationLogID DESC
```

### 8.2 Join with DifferenceLog to see full anomaly event

```sql
SELECT TOP 20
    n.NotificationLogID,
    n.Subject,
    d.InstrumentID,
    d.ActiveProviderID,
    d.ActiveProviderPrice,
    d.NotificationSeverityTypeID,
    n.Occurred
FROM History.PriceDetectionNotificationLog n WITH (NOLOCK)
JOIN History.PriceDetectionDifferenceLog d WITH (NOLOCK)
    ON d.NotificationLogID = n.NotificationLogID
ORDER BY n.NotificationLogID DESC
```

### 8.3 Search alert subjects for a specific instrument

```sql
SELECT TOP 10
    NotificationLogID,
    Subject,
    Occurred
FROM History.PriceDetectionNotificationLog WITH (NOLOCK)
WHERE Subject LIKE '100024%'
ORDER BY NotificationLogID DESC
```
