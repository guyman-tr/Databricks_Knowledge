# History.GetMirrorOperationDetails

> Retrieves the full details of a CopyTrader mirror operation by ReferenceID and CID, returning the success record if the operation succeeded or the failure record if it failed - used by support tooling to diagnose and display the outcome of a specific mirror operation.

## 1. Business Meaning

`History.GetMirrorOperationDetails` is a read-only lookup procedure that returns the full context of a specific CopyTrader mirror operation, identified by its client-side `ReferenceID` (GUID) and the copier's `CID`. It is designed to answer the question: "What happened when this customer performed this copy-trading operation?"

The procedure exists because mirror operation outcomes are recorded in two separate tables: `History.Mirror` for successes and `History.MirrorFail` for failures. Without this procedure, support tools would need to query both tables separately and merge the results. This SP handles that complexity: it queries both tables, produces a unified result set, and returns exactly one row - the success record if the operation succeeded, or the failure record if it failed.

It is called by support and operations tooling when a customer or agent reports an issue with a specific mirror operation. The `ReferenceID` (GUID) is the correlation key that the calling application (MOE - Mirror Operation Engine) generates per operation, logs, and passes to this procedure.

## 2. Business Logic

### 2.1 Success-Preferred UNION Pattern

**What**: The procedure uses a UNION of success and failure tables, ordered so that success records take precedence.

**Columns/Parameters Involved**: `@ReferenceID`, `@CID`, implicit `Ordinal` sort column

**Rules**:
- Success records (from History.Mirror) get Ordinal=1
- Failure records (from History.MirrorFail) get Ordinal=2
- SELECT TOP 1 ORDER BY Ordinal returns the success record if one exists
- If only a failure record exists, that is returned instead
- If a ReferenceID appears in both tables (partial success scenario), the success record wins

**Diagram**:
```
History.Mirror WHERE ReferenceID=X AND CID=Y   -> Ordinal=1 (success)
    UNION ALL
History.MirrorFail WHERE ReferenceID=X AND CID=Y -> Ordinal=2 (failure)
    |
    SELECT TOP 1 ... ORDER BY Ordinal
    -> Returns: success record (Ordinal=1) if exists, otherwise failure record (Ordinal=2)
```

### 2.2 DepositSummary Computation

**What**: DepositSummary on success is computed from two History.Mirror columns.

**Columns/Parameters Involved**: `DepositSummary`, `InitialInvestment` (source: History.Mirror)

**Rules**:
- Returned `DepositSummary` = `m.DepositSummary` + `m.InitialInvestment`
- This merges the cumulative deposit total with the initial registration investment into a single "total deposits" figure
- On failure records (History.MirrorFail), DepositSummary returns NULL

### 2.3 AvailableCashAmount vs DeltaAmount

**What**: Two output amount columns have conditional meaning based on MirrorOperationID.

**Columns/Parameters Involved**: `AvailableCashAmount`, `DeltaAmount`, `MirrorOperationID`

**Rules**:
```
MirrorOperationID = 3 (AddFunds):
    AvailableCashAmount = NULL
    DeltaAmount = m.Amount (the funds added)

MirrorOperationID = 1 (RegisterMirror):
    AvailableCashAmount = m.Amount
    DeltaAmount = m.Amount (the initial investment)

Other MirrorOperationIDs:
    AvailableCashAmount = m.Amount
    DeltaAmount = NULL
```

For failure records, these are sourced differently:
- `RequestedDeltaAmount`: Amount from failure record (MirrorOperationID=1 or 3 only)
- Other failure-specific columns: `ErrorMessageCode`, `FailReason`, `ExceptionType`, `RequestedEditSLPercentageDelta`, `RequestedRegisterSLPercentage`, `RequestedMirrorCalculationType`, `RequestedMirrorCloseReason`

### 2.4 Customer Context Enrichment

**What**: The result is enriched with customer-level data from Customer.Customer.

**Columns/Parameters Involved**: `CopierCountryID`, `GCID`

**Rules**:
- JOIN Customer.Customer ON c.CID = u.CID
- Returns `GCID` (global customer ID) and `CountryID` (as CopierCountryID)
- Read Uncommitted isolation level used (SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED)

## 8. Sample Queries

### 8.1 Look up a specific mirror operation by ReferenceID

```sql
EXEC History.GetMirrorOperationDetails
    @ReferenceID = 'a1b2c3d4-e5f6-7890-abcd-ef1234567890',
    @CID = 5000123
```

### 8.2 Find recent operations for a CID to get their ReferenceIDs

```sql
SELECT TOP 10
    m.ReferenceID,
    m.MirrorID,
    m.MirrorOperationID,
    m.Occurred,
    m.Amount
FROM History.Mirror m WITH (NOLOCK)
WHERE m.CID = 5000123
ORDER BY m.Occurred DESC
```

### 8.3 Check if a ReferenceID has both success and failure records (debugging)

```sql
SELECT 'Success' AS Source, ReferenceID, CID, MirrorOperationID, Occurred AS EventTime
FROM History.Mirror WITH (NOLOCK)
WHERE ReferenceID = 'a1b2c3d4-e5f6-7890-abcd-ef1234567890'
  AND CID = 5000123
UNION ALL
SELECT 'Failure', ReferenceID, CID, MirrorOperationID, FailOccurred
FROM History.MirrorFail WITH (NOLOCK)
WHERE ReferenceID = 'a1b2c3d4-e5f6-7890-abcd-ef1234567890'
  AND CID = 5000123
```
