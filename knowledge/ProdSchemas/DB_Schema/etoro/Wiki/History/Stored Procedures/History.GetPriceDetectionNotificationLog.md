## 1. Business Meaning

`History.GetPriceDetectionNotificationLog` is the primary read procedure for the price feed anomaly monitoring system. It joins `History.PriceDetectionDifferenceLog` (technical event data: instrument, provider, price, severity) with `History.PriceDetectionNotificationLog` (alert content: subject, body) and `Trade.Instrument` (configured pip threshold), producing a consolidated view of recent price detection alerts.

The procedure exists to provide operations dashboards and support tools with a rich, query-ready snapshot of recent price anomaly events. Without this join, callers would need to manually assemble data from three separate tables. The inclusion of `PipDifferenceThreshold` from `Trade.Instrument` lets consumers compare the magnitude of the detected discrepancy against the alert threshold that triggered it.

The price detection service (an external application, not in SSDT) continuously writes to both PriceDetectionDifferenceLog and PriceDetectionNotificationLog. This procedure is the read side, consumed by operations tooling or monitoring dashboards to review recent alerts.

## 2. Business Logic

### 2.1 Paginated Alert Feed (Top N Most Recent)

**What**: Returns up to @Top most recent price detection events, ordered by event time and notification ID.

**Columns/Parameters Involved**: `@Top`, `Occurred`, `NotificationLogID`

**Rules**:
- Default @Top = 100; when NULL is passed, the procedure explicitly resets it to 100 (double safety)
- Results are ordered `ORDER BY Occurred DESC, NotificationLogID DESC` - most recent events appear first
- The secondary sort by NotificationLogID (IDENTITY) breaks ties for events at the same millisecond
- No date filter is applied - the Top N approach is used to return the most recent without a fixed window

### 2.2 Consolidated Notification View

**What**: Merges technical event data (DifferenceLog) with human-readable alert content (NotificationLog) and instrument configuration (Trade.Instrument).

**Columns/Parameters Involved**: `NotificationLogID`, `Subject`, `Body`, `PipDifferenceThreshold`

**Rules**:
- Every DifferenceLog row has exactly one matching NotificationLog row (1:1 via NotificationLogID INNER JOIN)
- Subject format: "{InstrumentID} : {High|Low} Price Problem Alert for server {ServerName}" - "Low" means active feed price is below secondary feeds, "High" means above
- PipDifferenceThreshold is pulled from Trade.Instrument (per-instrument configuration) to show what threshold was configured at alert time - note this is the CURRENT threshold, not the historical value at event time
- NotificationSeverityTypeID classification: 1=Critical, 2=High, 3=Medium, 4=Low (FK to Dictionary.DowntimeSeverity)

**Diagram**:
```
History.PriceDetectionDifferenceLog (HPDD)
    |-- NotificationLogID (IDENTITY - drives 1:1 join)
    |-- InstrumentID --> JOIN Trade.Instrument (TI)
    |-- NotificationLogID --> JOIN History.PriceDetectionNotificationLog (HPDN)
    |
    v
Result: NotificationLogID, NotificationSeverityTypeID, InstrumentID,
        ActiveProviderPrice, Subject, Body, Occurred, PipDifferenceThreshold
```

## 8. Sample Queries

### 8.1 Get the 100 most recent price detection alerts (default)

```sql
EXEC History.GetPriceDetectionNotificationLog
```

### 8.2 Get the 10 most recent alerts

```sql
EXEC History.GetPriceDetectionNotificationLog @Top = 10
```

### 8.3 Direct query with severity filter for critical alerts only

```sql
SELECT TOP 50
    HPDD.NotificationLogID,
    HPDD.NotificationSeverityTypeID,
    HPDD.InstrumentID,
    HPDD.ActiveProviderPrice,
    HPDN.Subject,
    HPDN.Body,
    HPDD.Occurred,
    TI.PipDifferenceThreshold
FROM History.PriceDetectionDifferenceLog HPDD WITH (NOLOCK)
INNER JOIN History.PriceDetectionNotificationLog HPDN WITH (NOLOCK)
    ON HPDD.NotificationLogID = HPDN.NotificationLogID
INNER JOIN Trade.Instrument TI WITH (NOLOCK)
    ON TI.InstrumentID = HPDD.InstrumentID
WHERE HPDD.NotificationSeverityTypeID = 1  -- Critical only
ORDER BY HPDD.Occurred DESC
```
