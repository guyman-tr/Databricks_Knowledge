## 1. Business Meaning

`History.CurrencyPrice_Active` is a SQL synonym (alias) that resolves to the `CurrencyPrice_Active` table on the dedicated Price server (`AO-PRICE-LSN-ROR`), in its `Price` database under the `History` schema. This synonym makes the price feed available within the local `etoro` database context without requiring callers to reference the four-part linked-server name directly.

This object exists to abstract the physical location of price tick data. The Price server stores all incoming Bid/Ask ticks from liquidity providers as they arrive; by aliasing it here, stored procedures and views in `etoro` can query price data using two-part names (`History.CurrencyPrice_Active`) rather than the full cross-server path. If the Price server name or schema ever changes, only this synonym (and its `dbo` counterpart) needs updating, not every consumer.

Data flows in one direction only: the Price server writes new price ticks to `[AO-PRICE-LSN-ROR].[Price].[History].[CurrencyPrice_Active]` continuously. This synonym provides read-only access from the etoro DB. The companion view `History.CurrencyPrice` is a thin SELECT wrapper over this synonym, and is the recommended access path for most consumers. Two synonyms point to the same target: `History.CurrencyPrice_Active` (schema-qualified, preferred) and `dbo.HistoryCurrencyPrice_Active` (legacy alias in the dbo schema).

## 2. Business Logic

### 2.1 Cross-Server Linked-Server Access Pattern

**What**: This synonym uses a SQL Server linked-server connection to transparently access data on a remote server as if it were a local table.

**Columns/Parameters Involved**: All columns of the target table (28 columns - see Section 4)

**Rules**:
- Queries against this synonym are forwarded to `AO-PRICE-LSN-ROR` via the configured linked server. Network latency and the remote server's load affect query performance.
- The synonym is READ-ONLY from the etoro context - no INSERT/UPDATE/DELETE is performed against this synonym by any local procedure.
- Because the target is on a linked server, distributed transactions and certain query hints behave differently than with local tables.
- Access control is enforced at the linked-server level; if the linked-server login lacks permissions on the remote `Price` database, queries will fail with a cross-database security error (as observed with read-only MCP users).

**Diagram**:
```
etoro DB (local)
    History.CurrencyPrice_Active (synonym)
            |
            | via linked server: AO-PRICE-LSN-ROR
            v
    [AO-PRICE-LSN-ROR].[Price].[History].[CurrencyPrice_Active]
    (physical table - live price ticks, written by Price service)
```

### 2.2 Dual-Synonym Architecture

**What**: Two synonyms in different schemas both alias the same target, supporting both legacy and current access patterns.

**Columns/Parameters Involved**: N/A (object-level, not column-level)

**Rules**:

## 3. Data Overview

Data is sourced from the Price server via linked server. Direct MCP query is blocked (cross-database security context). The following column schema is known from `History.CurrencyPrice`, which is a direct `SELECT *`-equivalent wrapper over this synonym.

The target table stores one row per price tick received from a liquidity provider for a given instrument. Each row represents a single Bid/Ask price event with timestamps for when it was generated by the provider, when it arrived at the price server, and the validity window during which it was the active price.

| Column (example) | Typical Value | Meaning |
|-----------------|--------------|---------|
| CurrencyPriceID | 8,000,000,000+ | Auto-incrementing surrogate key on the Price server; very high values indicate the volume of ticks processed |
| InstrumentID | 1 (Gold), 7 (EUR/USD) | The traded instrument this price tick applies to |
| Bid / Ask | e.g., 1.08520 / 1.08530 | The market price at the moment of this tick; spread = Ask - Bid |
| ValidFrom / ValidTo | UTC timestamps | The window during which this was the prevailing price for the instrument |
| MarkupPips | 0.5 | Spread markup added on top of raw provider price |

*Note: Live data sample not available via MCP due to linked-server access restrictions on the read-only user.*

## 8. Sample Queries

### 8.1 Query recent price ticks for an instrument via the recommended wrapper view

```sql
-- Use History.CurrencyPrice (view) rather than the synonym directly for most use cases
SELECT TOP 100
    CurrencyPriceID,
    ProviderID,
    InstrumentID,
    Bid,
    Ask,
    BidSpreaded,
    AskSpreaded,
    MarkupPips,
    Occurred,
    ReceivedOnPriceServer
FROM History.CurrencyPrice WITH (NOLOCK)
WHERE InstrumentID = 7  -- EUR/USD
ORDER BY Occurred DESC
```

### 8.2 Find a specific price tick by PriceRateID (used by close-at-price logic)

```sql
-- Retrieve a specific historical price by its unique rate ID
-- Used by Trade.ClosePositionAtPriceRateID
SELECT
    CurrencyPriceID,
    PriceRateID,
    InstrumentID,
    Bid,
    Ask,
    Occurred,
    USDConversionRate
FROM History.CurrencyPrice_Active WITH (NOLOCK)
WHERE PriceRateID = @TargetPriceRateID
```

### 8.3 Price feed gap detection (pattern used by MissingPricesReportDaily)

```sql
-- Detect instruments with no price received in the last N minutes
SELECT
    InstrumentID,
    MAX(ReceivedOnPriceServer) AS LastPriceReceived,
    DATEDIFF(MINUTE, MAX(ReceivedOnPriceServer), GETUTCDATE()) AS MinutesSinceLastPrice
FROM History.CurrencyPrice_Active WITH (NOLOCK)
WHERE ReceivedOnPriceServer >= DATEADD(HOUR, -24, GETUTCDATE())
GROUP BY InstrumentID
HAVING DATEDIFF(MINUTE, MAX(ReceivedOnPriceServer), GETUTCDATE()) > 5
ORDER BY MinutesSinceLastPrice DESC
```
