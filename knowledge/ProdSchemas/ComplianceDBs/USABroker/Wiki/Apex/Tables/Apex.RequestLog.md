## 1. Business Meaning

Apex.RequestLog tracks every API request sent to Apex Clearing for account operations (create, update, close). Each row represents a single request-response lifecycle with the clearing house. The table serves as both an operational tracking tool (what is the current status of this request?) and an audit trail (what requests were made for this customer?).

This table is critical for the request polling workflow. When the system sends a request to Apex (e.g., create account), it receives a request ID back. The system then polls for the result using this request ID. RequestLog tracks the request throughout its lifecycle: from initial submission (StatusID=1/NEW), through polling (tracking ApexLastEventID progression), to completion (StatusID=12/COMPLETE or 11/REJECTED). The UpdateEventMask tracks which data fields triggered the request.

Data is created and updated by Apex.SaveRequestLog using a MERGE pattern that matches on GCID+ApexRequestID. It only updates when status, event ID, mask, or log ID actually change. The system retrieves the latest request via Apex.GetLastRequestLog (using MAX(RequestLogID) per GCID). System versioning (History.RequestLog) provides full change history.

## 2. Business Logic

### 2.1 Request Lifecycle Tracking

**What**: Each request to Apex Clearing goes through a status progression tracked by StatusID, with ApexLastEventID marking how far through the Apex event stream the response has been processed.

**Columns/Parameters Involved**: `ApexRequestID`, `StatusID`, `ApexLastEventID`, `ModifyTypeID`

**Rules**:
- StatusID uses Dictionary.ApexStatus values to track request progress (1=NEW -> processing -> 12=COMPLETE or 11=REJECTED)
- ApexLastEventID starts at 0 and advances as Apex emits events for this request
- ModifyTypeID classifies the request: 1=Create, 2=Update, 3=Close (Dictionary.ModifyType)
- The UNIQUE constraint on (GCID, ApexRequestID) ensures one log entry per request per customer
- SaveRequestLog uses MERGE with change detection - only updates if status, event ID, mask, or log ID changed

### 2.2 Update Event Mask

**What**: For account update requests (ModifyTypeID=2), the UpdateEventMask records which fields were changed, using the same bitmask as Dictionary.UserDataUpdatesMask.

**Columns/Parameters Involved**: `UpdateEventMask`, `ModifyTypeID`

**Rules**:
- UpdateEventMask=0 for create and close requests (no specific field changes)
- For update requests, the mask indicates which user data fields triggered the update (e.g., 128=HomeAddress, 64=PhoneNumber)
- See [User Data Updates Mask](_glossary.md#user-data-updates-mask) for bitmask values

## 8. Sample Queries

### 8.1 Get request history for a customer with status names

```sql
SELECT rl.RequestLogID, rl.GCID, rl.ApexRequestID,
       s.Name AS StatusName, mt.Name AS ModifyType,
       rl.UpdateEventMask, rl.BeginTime
FROM Apex.RequestLog rl WITH (NOLOCK)
INNER JOIN Dictionary.ApexStatus s WITH (NOLOCK) ON s.StatusID = rl.StatusID
LEFT JOIN Dictionary.ModifyType mt WITH (NOLOCK) ON mt.ModifyTypeID = rl.ModifyTypeID
WHERE rl.GCID = 22055177
ORDER BY rl.RequestLogID DESC;
```

### 8.2 Find requests stuck in NEW status for more than 1 hour

```sql
SELECT rl.RequestLogID, rl.GCID, rl.ApexRequestID, rl.ModifyTypeID, rl.BeginTime,
       DATEDIFF(MINUTE, rl.BeginTime, GETUTCDATE()) AS MinutesSinceCreation
FROM Apex.RequestLog rl WITH (NOLOCK)
WHERE rl.StatusID = 1
  AND DATEDIFF(MINUTE, rl.BeginTime, GETUTCDATE()) > 60
ORDER BY rl.BeginTime ASC;
```

### 8.3 Get the latest request for each modify type for a customer

```sql
SELECT mt.Name AS ModifyType, rl.RequestLogID, rl.ApexRequestID,
       s.Name AS StatusName, rl.BeginTime
FROM Apex.RequestLog rl WITH (NOLOCK)
INNER JOIN Dictionary.ApexStatus s WITH (NOLOCK) ON s.StatusID = rl.StatusID
INNER JOIN Dictionary.ModifyType mt WITH (NOLOCK) ON mt.ModifyTypeID = rl.ModifyTypeID
WHERE rl.GCID = 22055177
  AND rl.RequestLogID IN (
      SELECT MAX(RequestLogID) FROM Apex.RequestLog WITH (NOLOCK)
      WHERE GCID = 22055177 GROUP BY ModifyTypeID
  );
```
