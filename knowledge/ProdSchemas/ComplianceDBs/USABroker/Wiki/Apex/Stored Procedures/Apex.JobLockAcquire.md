## 1. Business Meaning

`Apex.JobLockAcquire` implements a **distributed mutual-exclusion lock** for named background jobs. It ensures that only one instance of a named job (e.g., a scheduled data-sync service running on multiple nodes) holds the active lock at any time. Locks are time-bounded: they carry an expiration timestamp, so a crashed or hung job will not permanently block other instances from acquiring the lock.

The procedure is called by background job services at startup and before beginning work to claim exclusive ownership of a named resource. On a multi-node deployment, only the node that successfully inserts or updates the lock row (indicated by `RowCount = 1`) should proceed with the job.

## 2. Parameters

| Parameter | Type | Nullable | Description |
|-----------|------|----------|-------------|
| `@Key` | `varchar(50)` | No | The name of the logical job/resource to lock (e.g., `"ApexSyncJob"`). |
| `@LockID` | `uniqueidentifier` | No | A GUID generated by the caller to identify this specific lock instance (proves ownership). |
| `@ExpiresAt` | `datetime` | No | UTC datetime when this lock should be considered expired if not renewed or released. |
| `@CurrentTimeUtc` | `datetime` | No | The caller's current UTC time; used to check whether an existing lock has expired. |

## 5. Logic Flow

The core is a `MERGE` statement with `WITH (XLOCK)` on the target:

1. **Source** is a single-row virtual table `(SELECT @Key AS [Key])`.
2. **Match condition:** `target.[Key] = source.[Key]`.
3. **WHEN MATCHED AND `target.ExpiresAt <= @CurrentTimeUtc`** (existing lock has expired): UPDATE the row with the new `LockID` and `ExpiresAt` — the caller takes over the expired lock.
4. **WHEN NOT MATCHED BY TARGET** (no lock exists for this key): INSERT a new row with `Key`, `LockID`, `ExpiresAt`.
5. If a row exists **and** `ExpiresAt > @CurrentTimeUtc` (lock is still live), neither branch fires — `@@ROWCOUNT = 0`.
6. Returns `@@ROWCOUNT AS [RowCount]` — 1 means acquired, 0 means not acquired.

---

## 6. Error Handling

No explicit TRY/CATCH. The `XLOCK` hint serialises concurrent acquisition attempts at the database level. If two callers race, SQL Server's lock manager ensures only one proceeds through the MERGE at a time; the other waits and then finds the lock already acquired (ROWCOUNT = 0).

## 8. Usage Notes

- **Ownership proof:** Always store the `@LockID` GUID you passed here; you will need it to call `JobLockRelease` or `JobLockRenew`. A node that does not know the `LockID` cannot release or renew the lock.
- **Polling retry:** If `RowCount = 0`, the lock is held. The caller should retry after a reasonable delay — typically several seconds — rather than spinning in a tight loop.
- **Clock skew:** `@CurrentTimeUtc` should be sourced from `GETUTCDATE()` or an NTP-synchronised clock on the calling host to avoid false positives in expiry evaluation.
- **Lock duration:** Set `@ExpiresAt` to a value slightly longer than the expected job duration, and call `JobLockRenew` periodically for long-running jobs to avoid premature expiration.
- `XLOCK` on the MERGE target table prevents phantom rows during the check-and-set, making this safe for concurrent callers without application-level coordination.
