## 1. Business Meaning

C2F.FiatTransactions stores the actual fiat-side transaction details for completed conversions. While EstimatedFiatTransactions captures the "quote" at conversion creation, this table records the real amounts at execution time - when rates are locked and fiat is actually credited. Only successfully completed conversions have a fiat transaction (13,713 rows vs 17,039 conversions - 80%).

This is the "destination side" of the conversion - confirming that fiat was credited to the customer's account. The Details column contains a unique client-load reference ID (format "C2F" + 8 digits) generated by `C2F.GenerateUniqueClientLoadReferenceId`, which serves as the external transaction reference for the fiat payment system.

Created by `C2F.InsertFiatTransaction`, which includes deduplication - it only inserts if no fiat transaction exists yet for that conversion.

## 2. Business Logic

### 2.1 Actual vs Estimated Rate Comparison

**What**: Actual rates may differ from estimates due to market movement during the conversion pipeline execution (~4 minutes).

**Columns/Parameters Involved**: `CryptoToFiatRate`, `FiatToUsdRate`, `CryptoToUsdRate`, `FiatAmount`, `UsdAmount`

**Rules**:
- Same rate columns as EstimatedFiatTransactions but with actual execution-time values
- RateTimestamp records exactly when the rate was locked
- FiatAmount and UsdAmount reflect the actual credited amounts
- Query SPs prefer actual over estimated: `CASE WHEN ft.UsdAmount IS NULL THEN eft.UsdAmount ELSE ft.UsdAmount`

### 2.2 Client Load Reference ID Pattern

**What**: Details column contains a unique reference ID used as the external transaction identifier for the fiat payment system.

**Columns/Parameters Involved**: `Details`

**Rules**:
- Format: "C2F" + 8 random digits (e.g., "C2F42756316", "C2F52271336")
- Generated by C2F.GenerateUniqueClientLoadReferenceId before fiat transaction insertion
- Uniqueness checked against existing FiatTransactions.Details values
- Indexed for lookups (IX_FiatTransactions_Details)

### 2.3 Conversion Fee Calculation

**What**: ConversionFeeAmount represents the actual fee deducted from the conversion.

**Columns/Parameters Involved**: `ConversionFeeAmount`

**Rules**:
- Nullable (NULL for zero-fee conversions, e.g., some EtoroPosition targets)
- Derived from Conversions.ConversionFeePercentage applied to the conversion amount
- Live examples: 0.134 fee on 134 fiat (0.1%), 0.468 on 467.32 fiat (0.1%)

## 8. Sample Queries

### 8.1 Get fiat transaction for a conversion
```sql
SELECT ft.FiatAmount, ft.UsdAmount, ft.CryptoToFiatRate, ft.AccountId, ft.Details, ft.ConversionFeeAmount
FROM C2F.FiatTransactions ft WITH (NOLOCK)
WHERE ft.ConversionId = @ConversionId
```

### 8.2 Find fiat transaction by reference ID
```sql
SELECT ft.*, c.Gcid, c.CorrelationId
FROM C2F.FiatTransactions ft WITH (NOLOCK)
INNER JOIN C2F.Conversions c WITH (NOLOCK) ON c.Id = ft.ConversionId
WHERE ft.Details = @ReferenceId
```

### 8.3 Rate slippage analysis (estimated vs actual)
```sql
SELECT c.Id, eft.CryptoToFiatRate AS EstimatedRate, ft.CryptoToFiatRate AS ActualRate,
       ft.CryptoToFiatRate - eft.CryptoToFiatRate AS RateSlippage
FROM C2F.Conversions c WITH (NOLOCK)
INNER JOIN C2F.EstimatedFiatTransactions eft WITH (NOLOCK) ON eft.ConversionId = c.Id
INNER JOIN C2F.FiatTransactions ft WITH (NOLOCK) ON ft.ConversionId = c.Id
ORDER BY ABS(ft.CryptoToFiatRate - eft.CryptoToFiatRate) DESC
```
