## 1. Business Meaning

This table is the central registry for every saga execution within the Saga schema's infrastructure. A saga is a multi-step distributed transaction pattern used to coordinate complex wallet operations that span multiple services and require compensating rollback actions if any step fails. Each row represents one saga instance from creation through its terminal state. The table holds approximately 102K rows spanning from June 2021 to the present.

The Saga schema's SagaRuns table serves the same architectural role as `Wallet.SagaRuns` in the Wallet schema, but handles a distinct set of saga types. The dominant saga type is `ExternalReceiveTransactionSaga` (91% of all runs), which orchestrates multi-step crypto receive flows including travel rule checks, AML verification, and optional auto crypto-to-position conversion. Other saga types include travel rule message handling (`TravelRuleMessageReceiveSentSaga`, `TravelRuleMessageReceiveAckSaga`) and staking operations (`saga_staking`).

New saga runs are created by `Saga.InsertSagaRunWithLeaseTime`, which atomically inserts the saga run, its initial status record, and a lease time entry. Status transitions are performed by `Saga.InsertSagaRunStatus`, which updates the denormalized `SagaStatusTypeId` on this table while inserting a history record into `Saga.SagaRunStatuses`. Recovery procedures like `Saga.GetSagaRunsForRecovery` and `Saga.GetAllAbandonedSagaRuns` query this table by status to find sagas that need intervention. The `Saga.TakeSagaRun` procedure implements distributed locking for saga processing via the `SagaLeaseTime` table.

## 2. Business Logic

### 2.1 Saga Lifecycle States

**What**: Each saga progresses through defined status transitions from creation to a terminal state.

**Columns/Parameters Involved**: `SagaStatusTypeId`, `SagaName`, `Created`, `LastUpdated`

**Rules**:
- SagaStatusTypeId=1 (Start): Saga initiated, forward steps executing. 278 currently in this state, indicating sagas in progress or potentially stuck
- SagaStatusTypeId=2 (Rollback): A step failed, compensating transactions executing. Currently 0 rows - rollback is a transient state
- SagaStatusTypeId=3 (Completed): All steps completed successfully. 96% of all sagas reach this terminal state
- SagaStatusTypeId=4 (Failed): Saga failed permanently. 4% of all sagas - operations teams investigate these for stuck transactions
- SagaStatusTypeId=5 (ForceStop): Manually terminated. Currently 0 rows
- See [Saga Status Type](../../_glossary.md#saga-status-type) for full value definitions

**Diagram**:
```
[1: Start] --all steps pass--> [3: Completed]  (96% of sagas)
    |
    +--step fails--> [2: Rollback] --rollback done--> [4: Failed]  (4% of sagas)
    |
    +--operator--> [5: ForceStop]
```

### 2.2 Saga Identity and Correlation

**What**: Each saga is identified by two GUIDs: SagaKey (technical instance ID) and CorrelationId (business request ID).

**Columns/Parameters Involved**: `SagaKey`, `CorrelationId`, `SagaName`

**Rules**:
- SagaKey is generated by the saga framework; uniquely identifies this saga instance for infrastructure purposes (lease management, step tracking, event logging)
- CorrelationId links the saga to the originating business request across systems
- Both have unique indexes - one business request maps to at most one saga run
- SagaName identifies the saga type (factory class name) and determines which steps the saga will execute
- The same SagaName + SagaStatusTypeId combination is indexed for efficient status-based queries per saga type

### 2.3 Saga Types and Their Purposes

**What**: Four distinct saga types operate in this schema, each coordinating a different multi-step wallet workflow.

**Columns/Parameters Involved**: `SagaName`

**Rules**:
- `ExternalReceiveTransactionSaga` (91%): Orchestrates crypto receive flows - travel rule checks, AML verification, optional auto C2P conversion (11-17 steps per Confluence architecture)
- `TravelRuleMessageReceiveSentSaga` (4%): Handles outbound travel rule messages for received transactions
- `TravelRuleMessageReceiveAckSaga` (3%): Handles acknowledgment of travel rule messages
- `saga_staking` (2%): Coordinates staking operations where crypto assets are locked to earn rewards

## 8. Sample Queries

### 8.1 Get saga run with human-readable status
```sql
SELECT sr.Id, sr.SagaName, sr.SagaKey, sr.Created, sst.Name AS Status,
       sr.CorrelationId, sr.LastUpdated
FROM Saga.SagaRuns sr WITH (NOLOCK)
JOIN Saga.SagaStatusTypes sst WITH (NOLOCK) ON sr.SagaStatusTypeId = sst.Id
WHERE sr.SagaKey = @SagaKey
```

### 8.2 Find potentially stuck sagas (in Start state for over 1 hour)
```sql
SELECT sr.Id, sr.SagaName, sr.SagaKey, sr.Created,
       DATEDIFF(MINUTE, sr.Created, GETUTCDATE()) AS MinutesSinceCreation,
       sr.CorrelationId
FROM Saga.SagaRuns sr WITH (NOLOCK)
WHERE sr.SagaStatusTypeId = 1
  AND sr.Created < DATEADD(HOUR, -1, GETUTCDATE())
ORDER BY sr.Created ASC
```

### 8.3 Saga completion rate by type
```sql
SELECT sr.SagaName, sst.Name AS Status, COUNT(*) AS Cnt
FROM Saga.SagaRuns sr WITH (NOLOCK)
JOIN Saga.SagaStatusTypes sst WITH (NOLOCK) ON sr.SagaStatusTypeId = sst.Id
GROUP BY sr.SagaName, sst.Name
ORDER BY sr.SagaName, sst.Name
```
