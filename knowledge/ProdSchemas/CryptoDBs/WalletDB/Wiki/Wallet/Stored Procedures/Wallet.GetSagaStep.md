## 1. Business Meaning

This procedure retrieves the full details of a single step within a saga (distributed transaction) execution. Given a saga's unique key and a step index, it returns the step's ID, request payload, response payload, and current status. This is the primary programmatic interface for the saga orchestration framework to inspect step state during execution.

The procedure serves two main consumers: the conversion service (`ConversionUser`) and the staking service (`StakingUser`), both of which run sagas with multiple sequential steps. During saga execution, the framework calls this procedure to check whether a specific step has completed, failed, or is still in progress before deciding whether to advance to the next step, retry, or initiate rollback.

Operationally, this procedure is also used during stuck saga recovery. When a saga is stuck at a specific step (StepStatusTypeId=5, InProgress), operations staff use this query pattern to inspect the step's request and response payloads to diagnose the root cause before deciding whether to reverse the saga to an earlier step.

## 2. Business Logic

### 2.1 Saga Step Lookup Pattern

**What**: Combines the saga identity (SagaKey) with step position (StepIndex) to locate a specific step record.

**Columns/Parameters Involved**: `@SagaKey`, `@StepIndex`, `SagaRuns.SagaKey`, `SagaSteps.StepIndex`, `SagaSteps.SagaRunId`

**Rules**:
- SagaKey is the unique GUID identifying a saga instance in Wallet.SagaRuns
- StepIndex is the zero-based position of the step within the saga sequence (0, 1, 2, ...)
- The JOIN bridges SagaRuns (identity layer) to SagaSteps (execution layer) via SagaRunId
- Returns at most one row since (SagaRunId, StepIndex) is unique in SagaSteps
- If no matching saga or step exists, returns an empty result set

### 2.2 Step Status Interpretation

**What**: The returned StepStatusTypeId indicates the step's execution state, which drives the saga framework's next action.

**Columns/Parameters Involved**: `StepStatusTypeId`

**Rules**:
- 1=Start: Step has been initiated but not yet completed
- 2=Failed: Step execution failed; saga may enter rollback
- 3=Retry: Step is being retried after a transient failure
- 4=Done: Step completed successfully; saga can advance to next step
- 5=InProgress: Step is actively executing (a step stuck in this state may indicate a processing problem)

**Diagram**:
```
Step Status Flow:
  1 (Start) --> 5 (InProgress) --> 4 (Done)
                    |
                    +--> 2 (Failed) --> rollback
                    |
                    +--> 3 (Retry) --> 5 (InProgress)
```

## 8. Sample Queries

### 8.1 Get a specific saga step by key and index
```sql
EXEC Wallet.GetSagaStep
    @SagaKey = 'A1B2C3D4-E5F6-7890-ABCD-EF1234567890',
    @StepIndex = 2;
```

### 8.2 Find all steps for a saga and then inspect a specific one
```sql
-- First find the saga
SELECT Id, SagaKey, SagaName, SagaStatusTypeId
FROM Wallet.SagaRuns WITH (NOLOCK)
WHERE CorrelationId = 'YOUR-CORRELATION-ID';

-- Then get a specific step
EXEC Wallet.GetSagaStep
    @SagaKey = 'SAGA-KEY-FROM-ABOVE',
    @StepIndex = 0;  -- First step
```

### 8.3 Diagnose a stuck saga by checking step status and payloads
```sql
-- Get the stuck step's details
DECLARE @SagaKey UNIQUEIDENTIFIER = 'STUCK-SAGA-KEY';
EXEC Wallet.GetSagaStep @SagaKey = @SagaKey, @StepIndex = 3;
-- Check Request/Response JSON to understand where the step got stuck
```
