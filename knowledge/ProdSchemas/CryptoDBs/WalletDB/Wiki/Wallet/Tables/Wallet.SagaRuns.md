## 1. Business Meaning

This table is the central registry for every saga execution within the wallet platform. A saga is a multi-step distributed transaction pattern used to coordinate operations that span multiple services or require rollback capability if any step fails (e.g., a crypto send that must atomically debit a wallet, submit to the blockchain, and update statuses). Each row in this table represents one saga instance from creation through its terminal state.

With ~164K rows, the table reflects the volume of complex, multi-step wallet operations processed over the platform's lifetime. The `SagaStatusTypeId` column tracks lifecycle progression: sagas start at status 1 (Start), advance to 3 (Completed) on success, or transition to 2 (Rollback) and then 4 (Failed) if a step cannot be recovered. This status trail, combined with `Wallet.SagaRunStatuses` (the status history table) and `Wallet.SagaSteps` (the individual step records), provides a complete audit log of every distributed transaction.

The `CorrelationId` column bridges the saga layer back to the wallet request layer - the same CorrelationId used throughout `Wallet.Requests`, `Wallet.SentTransactions`, and other tables identifies which business operation this saga was orchestrating. Engineers and operations teams use this table as the entry point for investigating stuck, failed, or incomplete distributed transactions.

## 2. Business Logic

### 2.1 Saga Lifecycle States

**What**: Each saga progresses through defined status transitions from creation to terminal state.

**Columns/Parameters Involved**: `SagaStatusTypeId`, `SagaName`, `Created`

**Rules**:
- SagaStatusTypeId=1 (Start): Saga has been initiated; steps are executing
- SagaStatusTypeId=2 (Rollback): A step failed; the saga is executing compensating transactions to undo completed steps
- SagaStatusTypeId=3 (Completed): All steps completed successfully; the distributed transaction is committed
- SagaStatusTypeId=4 (Failed): Rollback completed (or rollback itself failed); the saga is permanently failed
- Status transitions are recorded in Wallet.SagaRunStatuses for full history
- A saga stuck in status 1 (Start) for an unusual duration may indicate a processing problem

### 2.2 Saga Identity and Correlation

**What**: Each saga is identified by two GUIDs: SagaKey (technical instance ID) and CorrelationId (business request ID).

**Columns/Parameters Involved**: `SagaKey`, `CorrelationId`, `SagaName`

**Rules**:
- SagaKey is generated by the saga framework; uniquely identifies this saga instance for infrastructure purposes (lease management, step tracking)
- CorrelationId matches the business request CorrelationId from Wallet.Requests - links the saga to the business context
- Both are unique constraints - one business request maps to at most one saga run
- `SagaName` identifies the saga type/class (e.g., "SendCryptoSaga", "RedemptionSaga") - determines which steps are expected

## 8. Sample Queries

### 8.1 All currently active sagas (status = Start)
```sql
SELECT sr.Id, sr.SagaName, sr.SagaKey,
       sr.CorrelationId, sr.Created,
       DATEDIFF(MINUTE, sr.Created, GETUTCDATE()) AS AgeMinutes
FROM Wallet.SagaRuns sr WITH (NOLOCK)
WHERE sr.SagaStatusTypeId = 1  -- Start
ORDER BY sr.Created ASC
```

### 8.2 Failed sagas in the last 24 hours
```sql
SELECT sr.Id, sr.SagaName, sr.CorrelationId,
       sr.Created, sr.SagaStatusTypeId
FROM Wallet.SagaRuns sr WITH (NOLOCK)
WHERE sr.SagaStatusTypeId = 4   -- Failed
  AND sr.Created >= DATEADD(HOUR, -24, GETUTCDATE())
ORDER BY sr.Created DESC
```

### 8.3 Look up saga run by business correlation ID
```sql
SELECT sr.Id, sr.SagaName, sr.SagaKey,
       sr.SagaStatusTypeId, sr.Created
FROM Wallet.SagaRuns sr WITH (NOLOCK)
WHERE sr.CorrelationId = 'A1B2C3D4-0000-0000-0000-000000000000'
```
