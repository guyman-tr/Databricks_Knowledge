## 1. Business Meaning

This table records every client-facing notification dispatched by the wallet platform - covering push notifications and in-app messages that inform customers of wallet activity. Each row captures the notification type, which customer received it, which crypto was involved, and a JSON payload with supplementary parameters. With ~387K rows, the table reflects the volume of transaction-driven communication generated by the platform.

The table exists as a communication audit trail. Compliance and operations teams can verify that customers were notified of specific events (sent transactions, received transactions, pending payments) and when those notifications were delivered. When a customer disputes a transaction or claims they were not informed, this table provides the authoritative record of what notifications were dispatched.

Rows are created by the wallet notification service whenever a qualifying wallet event occurs. The `CorrelationId` ties each notification to the originating request, enabling end-to-end tracing from the transaction event through to the customer-facing alert. The `ParametersJson` field carries dynamic content (amounts, addresses, currencies) that was rendered into the notification.

## 2. Business Logic

### 2.1 Notification Type Classification

**What**: Each notification is categorised by the wallet event that triggered it.

**Columns/Parameters Involved**: `NotificationType`, `Gcid`, `CryptoId`

**Rules**:
- NotificationType=1 (SentTransaction): User's outbound crypto transfer was broadcast. Notification confirms the send.
- NotificationType=2 (ReceivedTransaction): Inbound crypto detected on the user's wallet address. Notification alerts the user of incoming funds.
- NotificationType=3 (PendingPayment): A payment request is awaiting action. Notification prompts the user to review.
- Implicit reference to Dict.ClientNotificationTypes; no FK constraint enforced.
- All three types carry a `CryptoId` identifying which asset the notification relates to.

### 2.2 JSON Parameters Payload

**What**: Dynamic notification content (amounts, txids, counterparty info) is stored as JSON.

**Columns/Parameters Involved**: `ParametersJson`

**Rules**:
- Content varies by `NotificationType`; parsing code in the notification service interprets the schema.
- Null or empty `ParametersJson` indicates a notification dispatched without supplementary context.
- Parameters are consumed at send time; this column serves as the post-dispatch audit record.

## 8. Sample Queries

### 8.1 All notifications sent to a specific customer
```sql
SELECT Id, NotificationType, CryptoId, CorrelationId, Occurred, ParametersJson
FROM Wallet.ClientNotifications WITH (NOLOCK)
WHERE Gcid = 18445512
ORDER BY Occurred DESC
```

### 8.2 Notification volume by type over the last 7 days
```sql
SELECT NotificationType,
       COUNT(*) AS NotificationCount,
       CAST(Occurred AS DATE) AS Day
FROM Wallet.ClientNotifications WITH (NOLOCK)
WHERE Occurred >= DATEADD(DAY, -7, GETUTCDATE())
GROUP BY NotificationType, CAST(Occurred AS DATE)
ORDER BY Day DESC, NotificationType
```

### 8.3 Trace notification for a specific transaction correlation
```sql
SELECT Id, NotificationType, Gcid, CryptoId, Occurred, ParametersJson
FROM Wallet.ClientNotifications WITH (NOLOCK)
WHERE CorrelationId = 'A1B2C3D4-0000-0000-0000-000000000000'
```
