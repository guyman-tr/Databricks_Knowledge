## 1. Business Meaning

Staking.Staking is the central fact table for crypto staking operations in the WalletDB system. Each row represents a single staking transfer - the delegation of a specific amount of a user's cryptocurrency to a staking pool where it earns Proof-of-Stake (PoS) rewards. eToro supports staking for ADA, TRX, SOL, ETH, POL, NEAR, DOT, ATOM, SUI, and AVAX, executing the process automatically on behalf of users who hold these real (non-CFD) crypto positions.

Without this table, the system would have no record of which users staked which amounts and when. It is the anchor entity for the entire Staking schema - the StakingStatuses event table tracks each operation's lifecycle (Pending/Failed/Completed) by FK to this table, and StakingTransactions records the blockchain transfer details including fees.

Rows are created by `Staking.InsertStaking`, which uses CorrelationId as an idempotency key to prevent duplicate staking operations. The procedure inserts the staking record and simultaneously creates the initial status of Pending (1) in Staking.StakingStatuses within a single transaction. Downstream, `Staking.InsertStakingStatus` transitions the status, `Staking.InsertStakingTransaction` records blockchain fees, and `Staking.GetStakingTotals` calculates total staked amounts (filtering only Completed operations). The Staking.StakingData view and Wallet.TransactionsView provide reporting access.

## 2. Business Logic

### 2.1 Idempotent Staking Creation

**What**: Each staking operation is uniquely identified by a CorrelationId to prevent duplicate processing.

**Columns/Parameters Involved**: `CorrelationId`, `Id`

**Rules**:
- `Staking.InsertStaking` checks `WHERE NOT EXISTS (SELECT 1 FROM Staking.Staking WHERE CorrelationId = @CorrelationId)` before inserting
- If a duplicate CorrelationId is detected, the insert is skipped and an error is raised: "Staking with CorrelationId already exists"
- The entire creation (staking record + initial Pending status) runs in a single transaction with rollback on failure
- CorrelationId is the business key used by downstream procedures (`InsertStakingStatus`, `InsertStakingTransaction`) to locate the staking record

### 2.2 Staking Amount Aggregation

**What**: Total staked amounts per wallet are calculated only from Completed operations.

**Columns/Parameters Involved**: `Amount`, `WalletId`, `CryptoId`

**Rules**:
- `Staking.GetStakingTotals` computes `SUM(Amount)` where the latest status is Completed (StakingStatusId=3)
- Failed staking operations are excluded from totals - their amounts were never locked
- Rewards are tracked separately in Staking.StakingRewards and summed independently as TotalRewards

## 8. Sample Queries

### 8.1 Get all staking operations for a specific wallet
```sql
SELECT s.Id, s.Amount, s.CorrelationId, s.Occurred
FROM Staking.Staking s WITH (NOLOCK)
WHERE s.WalletId = @WalletId AND s.CryptoId = @CryptoId
ORDER BY s.Occurred DESC
```

### 8.2 Staking operations with their latest status
```sql
SELECT s.Id, s.WalletId, s.Amount, s.CryptoId, ds.Name AS CurrentStatus
FROM Staking.Staking s WITH (NOLOCK)
INNER JOIN (
    SELECT StakingId, StakingStatusId,
           ROW_NUMBER() OVER (PARTITION BY StakingId ORDER BY Occurred DESC) AS rn
    FROM Staking.StakingStatuses WITH (NOLOCK)
) ss ON ss.StakingId = s.Id AND ss.rn = 1
INNER JOIN Dictionary.StakingStatuses ds WITH (NOLOCK) ON ds.Id = ss.StakingStatusId
ORDER BY s.Id DESC
```

### 8.3 Total staked amounts per wallet (completed only)
```sql
SELECT s.WalletId, ct.Name AS CryptoName, SUM(s.Amount) AS TotalStaked
FROM Staking.Staking s WITH (NOLOCK)
INNER JOIN Staking.StakingStatuses ss WITH (NOLOCK) ON ss.StakingId = s.Id
INNER JOIN Wallet.CryptoTypes ct WITH (NOLOCK) ON ct.CryptoID = s.CryptoId
WHERE ss.StakingStatusId = 3
GROUP BY s.WalletId, ct.Name
```
