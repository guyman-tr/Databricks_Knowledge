## 1. Business Meaning

Recurring.RecurringProgramMessages is a message queue table that records event notifications generated by the recurring payments system. Each row represents a message about a significant state change - either a payment execution result (deposit success/failure) or a recurring program status change (activation, cancellation, blocking). The table uses a polymorphic entity reference pattern where EntityTypeId determines whether EntityId refers to a Payment or PaymentExecution.

This table exists to decouple the recurring payment processing pipeline from downstream notification consumers. When a payment status changes or an execution completes, a message row is inserted here rather than synchronously notifying external systems. This allows the system to retry failed message delivery and provides an audit trail of all outbound events.

Messages are created by Recurring.GetOrInsertMessage (idempotent insert using composite key EntityId + EntityTypeId + AdditionalData) and their delivery status is tracked by Recurring.UpdateMessageStatus. The Body column contains a JSON payload with the event details. System-versioned with History.RecurringProgramMessages for full message lifecycle audit.

## 2. Business Logic

### 2.1 Polymorphic Entity Reference

**What**: Messages can refer to either a Payment or a PaymentExecution, with the EntityTypeId discriminator determining the target entity type.

**Columns/Parameters Involved**: `EntityId`, `EntityTypeId`, `AdditionalData`

**Rules**:
- EntityTypeId=1 (Payment): EntityId is a PaymentId - message describes a payment-level event (status change, cancellation)
- EntityTypeId=2 (PaymentExecution): EntityId is a PaymentExecutionId - message describes an execution-level event (deposit result)
- PaymentExecution messages (80%) are more frequent than Payment messages (20%) because each payment has multiple execution cycles
- AdditionalData provides context-dependent extra info (e.g., status value) and participates in the uniqueness check
- The composite key (EntityId, EntityTypeId, AdditionalData) is the idempotency key - GetOrInsertMessage checks this before inserting

### 2.2 Message Delivery Lifecycle

**What**: Each message progresses through a delivery status lifecycle from creation to successful send or failure.

**Columns/Parameters Involved**: `MessageStatusId`, `Retries`, `ModificationDate`

**Rules**:
- Messages are created with a specified MessageStatusId (typically 1=New)
- UpdateMessageStatus transitions the status: 1=New -> 2=Sent (on successful delivery) or 1=New -> 3=Failed (on delivery failure)
- Retries tracks the number of delivery attempts. GetOrInsertMessage initializes it from the caller's value (defaults to 1 if NULL)
- ModificationDate is updated to GETUTCDATE() on each status change via UpdateMessageStatus
- From sample data: most recent messages show MessageStatusId=2 (Sent) with Retries=0, indicating successful first-attempt delivery

**Diagram**:
```
[1 New] --delivery succeeds--> [2 Sent]
[1 New] --delivery fails-----> [3 Failed] --retry--> [1 New] (re-queued)
```

### 2.3 Message Body Content (JSON)

**What**: The Body column contains a JSON payload describing the event, with different structures based on the message type.

**Columns/Parameters Involved**: `Body`, `MessagesTypeId`

**Rules**:
- MessagesTypeId=1 (PaymentExecutionResult): Body contains execution outcome details
- MessagesTypeId=2 (RecurringProgramStatus): Body contains payment status change details
- Sample JSON structure (for payment status change): `{"PaymentId":N, "IsActive":bool, "RecurringProgramType":N, "PlanStatus":N, "StatusReason":N, "Cid":N, "StatusUpdateDate":"ISO8601"}`
- The JSON payload embeds denormalized data (Cid, ProgramType, StatusReason) so consumers don't need to query back

## 8. Sample Queries

### 8.1 Find unsent (new) messages for retry processing
```sql
SELECT rpm.RecurringProgramMessageId, rpm.EntityId, et.Name AS EntityType,
       rmt.Name AS MessageType, rpm.Retries, rpm.CreateDate
FROM Recurring.RecurringProgramMessages rpm WITH (NOLOCK)
INNER JOIN Dictionary.EntityType et WITH (NOLOCK) ON rpm.EntityTypeId = et.EntityTypeID
INNER JOIN Dictionary.RecurringProgramMessageType rmt WITH (NOLOCK) ON rpm.MessagesTypeId = rmt.RecurringProgramMessageTypeID
WHERE rpm.MessageStatusId = 1
ORDER BY rpm.CreateDate ASC
```

### 8.2 Get all messages for a specific payment
```sql
SELECT rpm.RecurringProgramMessageId, rpm.EntityId, et.Name AS EntityType,
       ms.Name AS MessageStatus, rpm.Body, rpm.CreateDate, rpm.ModificationDate
FROM Recurring.RecurringProgramMessages rpm WITH (NOLOCK)
INNER JOIN Dictionary.EntityType et WITH (NOLOCK) ON rpm.EntityTypeId = et.EntityTypeID
INNER JOIN Dictionary.MessageStatus ms WITH (NOLOCK) ON rpm.MessageStatusId = ms.MessageStatusID
WHERE rpm.EntityId = @PaymentId AND rpm.EntityTypeId = 1
ORDER BY rpm.RecurringProgramMessageId DESC
```

### 8.3 Message delivery summary by status and type
```sql
SELECT et.Name AS EntityType, rmt.Name AS MessageType, ms.Name AS MessageStatus,
       COUNT(*) AS MessageCount
FROM Recurring.RecurringProgramMessages rpm WITH (NOLOCK)
INNER JOIN Dictionary.EntityType et WITH (NOLOCK) ON rpm.EntityTypeId = et.EntityTypeID
INNER JOIN Dictionary.RecurringProgramMessageType rmt WITH (NOLOCK) ON rpm.MessagesTypeId = rmt.RecurringProgramMessageTypeID
INNER JOIN Dictionary.MessageStatus ms WITH (NOLOCK) ON rpm.MessageStatusId = ms.MessageStatusID
GROUP BY et.Name, rmt.Name, ms.Name
ORDER BY MessageCount DESC
```
