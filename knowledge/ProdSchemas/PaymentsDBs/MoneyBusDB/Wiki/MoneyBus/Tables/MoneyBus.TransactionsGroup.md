# MoneyBus.TransactionsGroup

> Groups related money transfer transactions together under a single logical unit, enabling atomic multi-leg transfers where debits and credits across account types are tied to one business operation.

## 1. Business Meaning

MoneyBus.TransactionsGroup represents a logical grouping of related money transfer transactions that belong to a single business operation. When a user initiates a fund transfer (e.g., deposit from IBAN to Trading account), the system creates a group that ties together all the individual transaction legs (debit, credit, hold, etc.) required to complete that operation. Each group is uniquely identified by a combination of customer (GCID) and external reference (ReferenceID), ensuring idempotent processing.

This table exists to provide atomicity and traceability for multi-leg transfers. Without it, individual transactions in MoneyBus.Transactions would be disconnected records with no way to identify which debit corresponds to which credit, or to determine that a set of transactions represents a single user action. The group also records which account type initiated the operation, enabling analytics on fund flow origins.

Data flows into this table from two paths: TransactionsAndGroupAdd (which atomically creates the group and all its transactions in one call) and TransactionsGroupAdd (which creates the group alone, with transactions added separately). TransactionsGroupGet reads individual groups by ID. The Transactions table references this via its GroupID column.

## 2. Business Logic

### 2.1 Idempotency via GCID + ReferenceID

**What**: The unique constraint on (GCID, ReferenceID) ensures that the same business operation for the same customer cannot create duplicate groups.

**Columns/Parameters Involved**: `GCID`, `ReferenceID`

**Rules**:
- Each customer (GCID) can have at most one group per ReferenceID
- ReferenceID is a UUID generated by the calling application to identify the business operation
- If the application retries a failed call with the same GCID + ReferenceID, the unique constraint prevents duplicate groups
- This is the foundation of idempotent processing in the MoneyBus payment pipeline

### 2.2 Initiator Account Type Tracking

**What**: Records which account type started the fund transfer operation, enabling directional analytics on money flow patterns.

**Columns/Parameters Involved**: `InitiatorAccountTypeId`

**Rules**:
- Value maps to Dictionary.AccountTypes: 1=Trading, 2=Options, 3=IBAN, 4=MoneyFarm
- IBAN-initiated groups (3) dominate (~75%) - representing external deposits and withdrawals
- Trading-initiated groups (1) are ~20% - representing internal transfers from trading accounts
- NULL values exist for legacy groups created before this column was added
- Value 0 appears in a small number of records - likely a default/unknown state

## 8. Sample Queries

### 8.1 Get a transaction group with all its transactions
```sql
SELECT g.ID AS GroupID, g.Created, g.ReferenceID, g.GCID, g.InitiatorAccountTypeId,
       t.ID AS TransactionID, t.Amount, t.CurrencyID, t.StatusID, t.CreditorTypeID, t.DebitorTypeID
FROM MoneyBus.TransactionsGroup g WITH (NOLOCK)
JOIN MoneyBus.Transactions t WITH (NOLOCK) ON t.GroupID = g.ID
WHERE g.ID = @GroupID
ORDER BY t.ID;
```

### 8.2 Find groups for a specific customer with initiator type resolved
```sql
SELECT g.ID, g.Created, g.ReferenceID, at.Name AS InitiatorAccountType
FROM MoneyBus.TransactionsGroup g WITH (NOLOCK)
LEFT JOIN Dictionary.AccountTypes at WITH (NOLOCK) ON at.ID = g.InitiatorAccountTypeId
WHERE g.GCID = @GCID
ORDER BY g.Created DESC;
```

### 8.3 Check for duplicate reference attempts (idempotency verification)
```sql
SELECT GCID, ReferenceID, COUNT(*) AS GroupCount
FROM MoneyBus.TransactionsGroup WITH (NOLOCK)
WHERE Created > DATEADD(DAY, -1, GETUTCDATE())
GROUP BY GCID, ReferenceID
HAVING COUNT(*) > 1;
```
