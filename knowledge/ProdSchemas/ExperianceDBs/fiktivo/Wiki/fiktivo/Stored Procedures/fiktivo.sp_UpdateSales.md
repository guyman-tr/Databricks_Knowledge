# fiktivo.sp_UpdateSales

> Records or updates a sale event in the affiliate sales table, either inserting a new sale or updating the total for an existing sale when the same affiliate-customer-date combination already exists.

## 1. Business Meaning

This procedure manages sale event records in `dbo.tblaff_Sales` and also interacts with `dbo.tblaff_Sales_Commissions`. It is the most complex event writer in the affiliate system because it implements upsert logic: if a sale already exists for the same AffiliateID + CUSTOMER_ID + ORDER_DATE combination, it updates the GRAND_TOTAL rather than creating a duplicate. This handles the common scenario where a customer's daily trading activity accumulates into a single sale record.

Sales are the core revenue-generating events in the affiliate system. They represent the trading activity (revenue, volume, or profit) generated by referred customers. The affiliate's revenue share commission is calculated based on this data. Without this procedure, the system could not track or aggregate daily trading activity for affiliate compensation.

The procedure is called by the commission engine as trading activity is processed. It checks for an existing sale record. If found, it updates the GRAND_TOTAL and sets @DidCreateNew=0. If not found, it inserts a new row with all tracking dimensions and sets @DidCreateNew=1. The @SalesID output and @DidCreateNew flag allow the downstream commission logic to decide whether to create a new commission or update an existing one.

## 2. Business Logic

### 2.1 Upsert Logic (Insert or Update)

**What**: Determines whether to create a new sale record or update an existing one based on the AffiliateID + CUSTOMER_ID + ORDER_DATE composite key.

**Columns/Parameters Involved**: `@AffiliateID`, `@CUSTOMER_ID`, `@ORDER_DATE`, `@GRAND_TOTAL`, `@SalesID`, `@DidCreateNew`

**Rules**:
- Checks for an existing sale WHERE AffiliateID = @AffiliateID AND CUSTOMER_ID = @CUSTOMER_ID AND ORDER_DATE = @ORDER_DATE
- If EXISTS: UPDATE the existing record's GRAND_TOTAL, set @DidCreateNew = 0, return the existing SalesID
- If NOT EXISTS: INSERT a new row with all parameters, set @DidCreateNew = 1, return SCOPE_IDENTITY() as @SalesID
- This prevents duplicate sale records for the same affiliate-customer-day combination

**Diagram**:
```
Trading Activity Detected
    |
    v
sp_UpdateSales
    |
    +--> CHECK: EXISTS sale for (AffiliateID, CUSTOMER_ID, ORDER_DATE)?
    |
    +-- YES --> UPDATE tblaff_Sales SET GRAND_TOTAL = @GRAND_TOTAL
    |           @DidCreateNew = 0, @SalesID = existing ID
    |
    +-- NO  --> INSERT INTO tblaff_Sales (all columns)
    |           @DidCreateNew = 1, @SalesID = SCOPE_IDENTITY()
    |
    v
Downstream: sp_UpdateSalesCommissions
    uses @SalesID and @DidCreateNew to INSERT or UPDATE commission
```

### 2.2 Hedge Commission Tracking

**What**: Tracks the hedge commission component separately from the main sale total.

**Columns/Parameters Involved**: `@GRAND_TOTAL`, `@HedgeCommission`, `@USED_BONUS_GRAND_TOTAL`

**Rules**:
- HedgeCommission captures the hedging-related portion of the affiliate's compensation
- USED_BONUS_GRAND_TOTAL tracks the portion of the sale's grand total that was bonus-funded
- These values allow net revenue calculation: effective revenue = GRAND_TOTAL - USED_BONUS_GRAND_TOTAL

### 2.3 Multi-Dimensional Event Classification

**What**: Classifies each sale event across multiple business dimensions for reporting and commission rule application.

**Columns/Parameters Involved**: `@ProviderID`, `@CountryID`, `@FunnelID`, `@LabelID`, `@PlayerLevelID`, `@LotCount`, `@NetProfit`

**Rules**:
- LotCount tracks the trading volume in lots for volume-based commission models
- NetProfit captures the P&L for the sale period for profit-sharing commission models
- Country, funnel, label, and player level enable dimension-based commission rate overrides

## 8. Sample Queries

### 8.1 Record a new sale event
```sql
DECLARE @SalesID INT, @DidCreateNew BIT
EXEC fiktivo.sp_UpdateSales
    @AffiliateID = 100,
    @CUSTOMER_ID = '12345',
    @ORDER_DATE = '2026-04-12',
    @GRAND_TOTAL = 1500.00,
    @HedgeCommission = 25.00,
    @AffiliateSaleAccepted = 1,
    @Valid = 1,
    @BannerID = 0,
    @DaysToConvert = 5.0,
    @Optional1 = '', @Optional2 = '', @Optional3 = '',
    @DownloadID = 0,
    @ProviderID = 0,
    @OriginalProviderID = 0,
    @CountryID = 1,
    @NetProfit = 200.00,
    @LotCount = 3.500000,
    @RealProviderID = 1,
    @FunnelID = 0,
    @LabelID = 1,
    @PlayerLevelID = 1,
    @USED_BONUS_GRAND_TOTAL = 0,
    @SalesID = @SalesID OUTPUT,
    @DidCreateNew = @DidCreateNew OUTPUT
SELECT @SalesID AS SalesID, @DidCreateNew AS WasNewRecord
```

### 8.2 Check if a sale exists for a given affiliate-customer-date
```sql
SELECT SalesID, GRAND_TOTAL, NetProfit, LotCount
FROM dbo.tblaff_Sales WITH (NOLOCK)
WHERE AffiliateID = 100 AND CUSTOMER_ID = '12345' AND ORDER_DATE = '2026-04-12'
```

### 8.3 Review sales with commission details for an affiliate
```sql
SELECT s.SalesID, s.CUSTOMER_ID, s.ORDER_DATE, s.GRAND_TOTAL,
       s.NetProfit, s.LotCount, sc.Commission, a.UserName
FROM dbo.tblaff_Sales s WITH (NOLOCK)
INNER JOIN dbo.tblaff_Affiliates a WITH (NOLOCK) ON a.AffiliateID = s.AffiliateID
LEFT JOIN dbo.tblaff_Sales_Commissions sc WITH (NOLOCK) ON sc.SalesID = s.SalesID
WHERE s.AffiliateID = 100
ORDER BY s.ORDER_DATE DESC
```
